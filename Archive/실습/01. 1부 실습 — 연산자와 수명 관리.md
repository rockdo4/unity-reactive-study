# 실습 01 — 연산자와 수명 관리 (1부 마무리)

1부(01~05차시)를 마친 뒤 수행. 새 씬 또는 기초 데모 씬 사본에서 진행하고, **모든 구독에 수명 관리**(`AddTo` 또는 `CompositeDisposable`)가 있어야 함. 제출물: 스크립트 + 실행 영상(또는 GIF) + 과제 C의 답.

## 과제 A. 홀드(길게 누르기) 감지 — 연산자 합성

**요구사항** — 버튼(또는 스페이스 키)을 **0.5초 이상 누르고 있으면** "차지 공격!"을 1회 출력. 떼었다가 다시 누르면 다시 잴 것.

힌트:

- 누름/뗌을 각각 스트림으로: `Observable.EveryUpdate().Where(눌린 프레임)` / `Where(뗀 프레임)` (Input System: `wasPressedThisFrame` / `wasReleasedThisFrame`)
- 접근 1 — 누름 스트림을 `Debounce`… 가 아니라, **누름 후 0.5초 타이머**(`Observable.Timer`)를 시작하고 뗌 스트림이 먼저 오면 무효화하는 합성을 생각해 볼 것 (`TakeUntil`이 유용함 — 카탈로그에서 찾아 보기)
- 접근 2 — `누름.SelectMany(_ => Observable.Timer(0.5s).TakeUntil(뗌))` 형태도 가능
- 어느 접근이든 **상태 플래그·코루틴 금지** — 연산자로만

채점 포인트: 동작 ○ / 플래그·코루틴 없음 / `AddTo` 존재

## 과제 B. 설정 자동 저장 — Debounce 실전

**요구사항** — 슬라이더(볼륨)와 토글(음소거)이 있는 설정 패널에서, **마지막 조작 후 1초가 지나면** "저장됨"을 1회 출력(연속 조작 중에는 저장하지 않음). 슬라이더·토글 어느 쪽을 조작해도 같은 저장 타이머를 공유할 것.

힌트:

- `m_Slider.onValueChanged.AsObservable()`과 `m_Toggle.onValueChanged.AsObservable()`은 타입이 다름 — `Select(_ => Unit.Default)`로 통일한 뒤 `Merge`
- 합친 스트림에 `Debounce(1초)`
- 보너스: 저장 내용에 **최신 슬라이더·토글 값**을 함께 출력 (`.Value`를 직접 읽거나 `WithLatestFrom` 시도)

채점 포인트: Merge로 통합 / Debounce 적용 / 연속 조작 중 침묵 확인

## 과제 C. 누수 수사 — 코드 리뷰

아래 코드에는 **구독 수명 문제 3곳**이 있음. 모두 찾아 ①무엇이 문제인지 ②어떻게 고치는지 쓰고, 고친 코드를 제출할 것.

```csharp
using System;
using R3;
using TMPro;
using UnityEngine;
using UnityEngine.UI;

public class QuestHud : MonoBehaviour
{
    [SerializeField] private TextMeshProUGUI m_TimerText;
    [SerializeField] private Button m_ClaimButton;

    public static Subject<string> QuestCompleted = new();   // 전역 발행기 (다른 씬에서도 씀)

    private void Start()
    {
        // (1) 남은 시간 표시
        Observable.Interval(TimeSpan.FromSeconds(1))
            .Subscribe(_ => m_TimerText.text = GetRemainTime());

        // (2) 퀘스트 완료 알림 받기
        QuestCompleted.Subscribe(name => ShowToast($"{name} 완료!"));

        // (3) 보상 수령 버튼 (연타 방지)
        m_ClaimButton.OnClickAsObservable()
            .ThrottleFirst(TimeSpan.FromSeconds(1))
            .Subscribe(_ => Claim())
            .AddTo(this);
    }

    private string GetRemainTime() => "...";
    private void ShowToast(string msg) { }
    private void Claim() { }
}
```

검증 방법: 고치기 전/후를 `ObservableTracker`(`Window > Observable Tracker`)로 비교 — 이 오브젝트를 파괴했을 때 관련 구독이 전부 사라지면 합격.

<details>
<summary>정답 확인 (먼저 풀고 열 것)</summary>

1. **(1) Interval 구독에 `AddTo(this)` 없음** — 전역 시간 스트림이라 오브젝트 파괴 후에도 영원히 돌고, 파괴된 `m_TimerText` 접근 시 `MissingReferenceException`. → `.AddTo(this)` 추가.
2. **(2) static 발행기 구독에 `AddTo(this)` 없음** — `QuestCompleted`는 씬보다 오래 사는 static이므로, 이 HUD가 파괴돼도 구독이 남음. → `.AddTo(this)` 추가. (발행자 수명 ≠ 내 수명이면 반드시 묶는다)
3. **static `Subject`를 아무도 `Dispose`하지 않음 + `public` 필드로 외부에서 `OnNext` 가능** — 발행기도 IDisposable이므로 소유 주체를 정해 정리하고, 외부에는 `Observable<string>`로만 노출하는 것이 안전. (이 항목은 "필드 노출 캡슐화"까지 지적하면 만점)

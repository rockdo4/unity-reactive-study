# 07. 타이밍 바 과제 — 강사용

> 🎯 1·2부에서 배운 R3를 **학생이 빈 씬에서 스스로 조립해** 완성하는 capstone 과제입니다. 클리커(06)가 강사 주도 라이브 코딩이었다면, 이 과제는 **자력 완성**(숙제 또는 1차시 자습)입니다. 스켈레톤·학생 패키지를 배포하지 않고, **완성본 윈도우 빌드 + 과제 명세** 만 줍니다.
>
> - **학생 배포물**: ① 완성본 **윈도우 빌드**(목표 동작) ② [과제 명세](../과제/타이밍%20바%20—%20빈%20씬에서%20만드는%20R3%20미니게임.md)(노션 공유)
> - **모범답안(강사 전용)**: `Assets/Scripts/TimingBar/TimingBarModel.cs` · `TimingBarPresenter.cs`
> - **정본 씬 생성**: `Tools > Reactive Study > 03. 타이밍 바(과제 정본) 씬 생성` → `Assets/Scenes/TimingBar.unity`(git 비커밋, 빌드·시연용)

## 수업 개요

- **시간**: 숙제 또는 1차시 자습(난이도 종합).
- **선수 지식**: 1부(02~05) 전부. 특히 02 `EveryUpdate`·`ReactiveProperty`, 03 `Merge`·`WithLatestFrom`·`Take`·`DistinctUntilChanged`·`Scan`, 04 `AddTo`·`CompositeDisposable`·`ObservableTracker`, 05 `SubscribeAwait`·`UniTask.Delay`·바인딩.
- **학습 목표**: `Update()`·코루틴·상태 플래그 0개로 작은 게임을 완성한다. 클리커가 안 쓴 **프레임 스트림·입력 시점 샘플링·시퀀스 종료·비동기 라운드 흐름·상태머신** 을 한 게임으로 종합한다.

> 💡 이 과제가 클리커와 겹치지 않는 이유를 학생에게 한 줄로: "클리커는 **끊임없이 쌓는** 게임(Interval/Scan), 타이밍 바는 **한 순간을 맞히는** 게임(EveryUpdate/WithLatestFrom)."

## 빌드·배포 절차 (강사)

1. **정본 씬 생성**: `Tools > Reactive Study > 03. 타이밍 바(과제 정본) 씬 생성`.
2. **확인**: Play로 한 판 동작 확인(카운트다운→마커→판정→결과). Console 에러 0(`Unity_GetConsoleLogs`).
3. **윈도우 빌드**: `File > Build Settings`에 `TimingBar` 씬만 넣고 Windows로 빌드. 산출 exe는 리포에 커밋하지 않고 학생에게 배포(노션 링크/파일 공유).
4. **명세 배포**: [과제 명세](../과제/타이밍%20바%20—%20빈%20씬에서%20만드는%20R3%20미니게임.md)를 노션에 게시.

## 흔한 함정

- **WithLatestFrom용 위상 스트림이 Hot 이어야 한다** — 시각 마커와 입력 판정이 **같은 `ReactiveProperty<float>`** 를 봐야 두 위치가 어긋나지 않습니다(콜드 `EveryUpdate.Select` 를 두 곳이 각자 구독하면 desync).
- **`EveryUpdate` 에 `AddTo` 누락** — 씬을 나가도 프레임 루프가 영원히 돕니다(6단계 Tracker로 적발). 고의 누락→재현→복구가 좋은 시연.
- **라운드 재시작 시 정리 누락** — 라운드 입력 구독은 **라운드용 `CompositeDisposable`** 에 담아 라운드 끝에 정리하고, 영속 구독만 `AddTo(this)`.
- **레거시 Input 사용** — Space는 `Keyboard.current.spaceKey.wasPressedThisFrame`(Input System). 이 프로젝트는 레거시 Input Manager 금지.
- **존 시각과 판정 불일치** — 정본은 존 폭을 `TimingBarModel.GoodHalfWidth/PerfectHalfWidth` 상수로 계산해 시각과 판정을 일치시킵니다. 학생도 위상 좌표계(0~1) 하나로 통일하도록 안내.

## 예상 Q&A

> ❓ **콤보를 Model에 둘까, Scan으로 풀까?**
> 둘 다 정답. 모범답안은 콤보 누적을 Model이 단일 소유(테스트 용이)합니다. "콤보를 입력의 리듬 해석으로 보고 Presenter에서 `Scan`으로" 푸는 풀이도 좋은 토론거리입니다.

> ❓ **라운드 종료를 Take(n)과 제한시간 둘 다로?**
> 입문 난도를 위해 정본은 `Take(n)` 하나로 종료합니다. 제한시간(`TakeUntil(Timer)`)은 도전 과제로 분리했습니다 — 둘을 "먼저 끝나는 쪽"으로 합치려면 추가 합성이 필요해 난도가 오릅니다.

> ❓ **마커 위상 계산을 Model에 넣어도 되나요?**
> 프레임·`Time`·`RectTransform`은 Unity 의존이라 Presenter에 둡니다. Model은 "위상 → 판정 → 점수"라는 순수 규칙만 알게 해야 단위 테스트가 됩니다(`TimingBarModelTests` 참고).

## 수업 개요

- **시간**: 숙제 또는 1차시 자습(난이도 종합).
- **선수 지식**: 1부(02~05) 전부. 특히 02 `EveryUpdate`·`ReactiveProperty`, 03 `Merge`·`WithLatestFrom`·`Take`·`DistinctUntilChanged`·`Scan`, 04 `AddTo`·`CompositeDisposable`·`ObservableTracker`, 05 `SubscribeAwait`·`UniTask.Delay`·바인딩.
- **학습 목표**: `Update()`·코루틴·상태 플래그 0개로 작은 게임 완성. 클리커에서 안 다룬 **프레임 스트림·입력 시점 샘플링·시퀀스 종료·비동기 라운드 흐름·상태머신**을 한 게임으로 종합.

> 💡 클리커와 차이: 클리커는 끊임없이 쌓는 게임(Interval/Scan), 타이밍 바는 한 순간을 맞히는 게임(EveryUpdate/WithLatestFrom).

## 변형: UI 제공 스캐폴드 버전 (선택)

자력 완성(A)이 부담되는 반·시간엔 UI·와이어링까지 준 패키지(B)를 배포. 게임·동작·완료 기준은 A와 동일, **화면 제작만 뺌** — 학생은 R3에 집중.

- **학생 배포물(B)**: ① 패키지 `R3Lecture-Student-3-TimingBar`(`ReactiveTimingBar` 씬 + `MyTimingBarModel`·`MyTimingBarPresenter` 스켈레톤) ② [과제 명세 B](<../과제/타이밍%20바%20—%20UI%20제공%20스켈레톤(R3%20채우기).md>). 폰트는 1부 기초 패키지에 있으니 **1부 먼저 임포트** 안내.
- **패키지 생성**: `Tools > Reactive Study > Student Package > Create All Student Scenes` → `Export 3. TimingBar`(또는 `Export All`). 절차·동기화 규칙은 [학생 배포 가이드](00.%20학생%20배포%20가이드.md) 그대로.
- **스왑 기계 공유**: 클리커처럼 강사용 정본 씬(03)을 빌드 → `TimingBarPresenter`를 `MyTimingBarPresenter`로 스왑 → 학생 씬 저장. `My*` 직렬화 필드명이 원본과 어긋나면 씬 생성 시 명시적 예외.

**B 스켈레톤이 비워 둔 곳**(빈 메서드만 채우는 형식이 아니라, UI·와이어링만 주고 구조는 학생이 설계):

| 파일                      | 제공                                                          | 학생이 채움                                                                                   |
| ------------------------- | ------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| `MyTimingBarModel.cs`     | `ReactiveProperty` 3개·존/점수 상수·생성자(`AddTo`)·`Dispose` | `ZoneOf`·`Judge`·`ApplyHit`·`ResetForNewGame` 규칙 본문                                       |
| `MyTimingBarPresenter.cs` | `[SerializeField]` 11개(UI 연결)·빈 `Start()`                 | 모델 생성·위상/시도 `ReactiveProperty`·마커/판정/표시/라운드 스트림 전부(2~6단계)·`OnDestroy` |

- A의 스켈레톤(클리커 등)은 `BindInput`/`BindView` 빈 메서드를 미리 줬지만, B 타이밍 바는 **메서드 골격도 안 줌** — `Start()`만 비워 두고 스트림 분리·이름까지 학생이 정함. 자력 완성과 클리커식 빈칸 채우기의 중간 난이도.
- 채점 루브릭·흔한 함정·Q&A는 A와 공유(아래). B는 2·3단계(마커·존)는 거의 거저라 4·5단계 배점 비중을 키워도 됨.

## 흔한 함정

- **WithLatestFrom용 위상 스트림은 Hot이어야 함** — 시각 마커와 입력 판정이 같은 `ReactiveProperty<float>`를 봐야 두 위치가 안 어긋남(콜드 `EveryUpdate.Select`를 두 곳이 각자 구독하면 desync).
- **`EveryUpdate`에 `AddTo` 누락** — 씬 나가도 프레임 루프 영원히 돎(6단계 Tracker로 적발). 고의 누락→재현→복구가 좋은 시연.
- **라운드 재시작 시 정리 누락** — 라운드 입력 구독은 라운드용 `CompositeDisposable`에 담아 라운드 끝에 정리, 영속 구독만 `AddTo(this)`.
- **레거시 Input 사용** — Space는 `Keyboard.current.spaceKey.wasPressedThisFrame`(Input System). 레거시 Input Manager 금지.
- **존 시각과 판정 불일치** — 정본은 존 폭을 `TimingBarModel.GoodHalfWidth/PerfectHalfWidth` 상수로 계산해 시각·판정 일치. 학생도 위상 좌표계(0~1) 하나로 통일하도록 안내.

## 예상 Q&A

> ❓ **콤보를 Model에 둘까, Scan으로 풀까?**
> 둘 다 정답. 모범답안은 콤보 누적을 Model이 단일 소유(테스트 용이). 콤보를 입력 리듬 해석으로 보고 Presenter에서 `Scan`으로 푸는 풀이도 좋은 토론거리.

> ❓ **라운드 종료를 Take(n)과 제한시간 둘 다로?**
> 입문 난도 위해 정본은 `Take(n)` 하나로 종료. 제한시간(`TakeUntil(Timer)`)은 도전 과제로 분리 — 둘을 "먼저 끝나는 쪽"으로 합치려면 추가 합성 필요, 난도 오름.

> ❓ **마커 위상 계산을 Model에 넣어도 되나?**
> 프레임·`Time`·`RectTransform`은 Unity 의존이라 Presenter에 둠. Model은 "위상 → 판정 → 점수" 순수 규칙만 알아야 단위 테스트 가능(`TimingBarModelTests` 참고).

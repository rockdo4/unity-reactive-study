## 생성법 1: Range — 즉시 흐르고 완료 (6분)

**실행·관찰**

1. **`Range(1, 4)`** 클릭 → 로그 패널에 `Range OnNext: 1`~`4`가 연달아, 그리고 `Range OnCompleted: ...`.
2. "버튼을 한 번 더 눌러 보세요" → 누를 때마다 처음부터 1~4가 새로 흐름.

> 📌 짚어줄 것: 구독하는 **즉시** 4개가 다 흐르고 **완료(`|`)** 까지 난다. 누를 때마다 자기 전용으로 새로 흐르는 게 **콜드** 스트림. "Subscribe는 등록"이지만 `Range`는 등록과 동시에 값이 오는 특수 경우(이론 2-2·2-6).

**코드 워크스루**

```csharp
// 1) Range — 콜드(cold) Observable: 구독하는 순간 값이 흐르고 즉시 완료된다
m_RangeButton
    .OnClickAsObservable()
    .Subscribe(_ =>
    {
        Log("--- Range(1, 4) 구독 ---");
        Observable
            .Range(1, 4)
            .Subscribe(x => Log($"Range OnNext: {x}"), result => Log($"Range OnCompleted: {result}"))
            .AddTo(this);
    })
    .AddTo(this);
```

- 버튼 클릭 자체가 `OnClickAsObservable()`로 스트림이 됨 — 클릭 스트림을 구독해, 그 안에서 `Range` 스트림을 만들어 구독한다(스트림 안의 스트림).
- `Subscribe`의 두 번째 인자가 `OnCompleted` 콜백 — 그래서 완료 로그가 찍힘. **완료가 있다**는 게 이번 생성법의 핵심.
- `.AddTo(this)`는 4차시 주제 — 지금은 "오브젝트 파괴 시 구독이 자동 해제되도록 묶는 것" 한 줄만.

**변형 실습**: `Range(1, 4)`를 `Range(1, 10)`으로 고쳐 핫 리로드 → 1~10이 흐름. 작은 수정을 직접 해 보는 경험.

**이론 연결**: 2-2, 2-3, 2-6.

## 생성법 2: Timer — 2초 뒤 한 번 (4분)

**실행·관찰**

1. **`Timer 2초`** 클릭 → 즉시 `Timer(2초) 시작...`, **2초 뒤** `Timer 발행! (2초 경과)`.

> 📌 짚어줄 것: 2초 동안 화면은 멈추지 않음(논블로킹). 코루틴 `WaitForSeconds`로 하던 "N초 뒤 1회 실행"이 한 줄이 됨. 일회성이라 한 번 발행하고 끝.

**코드 워크스루**

```csharp
// 2) Timer — 지정 시간 뒤 한 번 발행하고 완료되는 일회성 스트림
m_TimerButton
    .OnClickAsObservable()
    .Subscribe(_ =>
    {
        Log("Timer(2초) 시작...");
        Observable.Timer(TimeSpan.FromSeconds(2)).Subscribe(_ => Log("Timer 발행! (2초 경과)")).AddTo(this);
    })
    .AddTo(this);
```

- `Observable.Timer(TimeSpan.FromSeconds(2))` — 2초 뒤 값 하나를 내보내고 완료. 발행 값 자체는 안 쓰므로 `_`로 버림.

**변형 실습**: `FromSeconds(2)`를 `FromSeconds(0.5)`로 바꿔 체감 — 지연이 짧아짐.

**이론 연결**: 2-3.

## 생성법 3: Interval — 무한, 멈춤은 Dispose (6분)

**실행·관찰**

1. **`Interval 시작`** 클릭 → 1초 간격으로 `Interval tick 1`, `tick 2`, … 가 계속 쌓임. 버튼 라벨이 `Interval 중지`로 바뀜.
2. 다시 클릭(이제 `Interval 중지`) → `Interval 중지 (Dispose 호출)`. tick이 멈춤.

> 📌 짚어줄 것: `Interval`은 스스로 끝나지 않는 **무한** 스트림(완료가 없음). 멈추는 유일한 방법이 구독을 **`Dispose()`** 하는 것 — 이게 4차시 수명 관리의 복선.

**코드 워크스루** — 토글 구조가 핵심:

```csharp
// 3) Interval — 일정 주기로 무한 발행. 멈추려면 구독을 직접 Dispose해야 한다
m_IntervalToggleButton
    .OnClickAsObservable()
    .Subscribe(_ =>
    {
        if (m_IntervalSubscription == null)
        {
            int tick = 0;
            m_IntervalSubscription = Observable
                .Interval(TimeSpan.FromSeconds(1))
                .Subscribe(_ => Log($"Interval tick {++tick}"))
                .AddTo(this); // 토글을 깜빡해도 오브젝트 파괴 시 함께 해제되도록 이중 안전망
            Log("Interval 시작 (1초 간격)");
            SetToggleLabel("Interval 중지");
        }
        else
        {
            m_IntervalSubscription.Dispose(); // Dispose가 곧 구독 해제
            m_IntervalSubscription = null;
            Log("Interval 중지 (Dispose 호출)");
            SetToggleLabel("Interval 시작");
        }
    })
    .AddTo(this);
```

- `m_IntervalSubscription`이라는 **`IDisposable` 필드**에 구독을 보관 — "구독을 변수로 들고 있어야 나중에 멈출 수 있다". 이게 이론 2-2에서 본 "`Subscribe`의 반환값이 구독 그 자체"의 실물.
- `null`이면 시작(구독을 만들어 보관), 아니면 `Dispose()`로 멈추고 `null`로 — **중지 = `Dispose()`** 를 코드로 확인.
- `.AddTo(this)`는 토글을 깜빡 잊고 Play를 정지해도 오브젝트 파괴 시 함께 해제되는 **이중 안전망**(4차시 예고).

**변형 실습**: `FromSeconds(1)`을 `FromSeconds(0.3)`으로 바꿔 tick이 빨라지는 것 확인.

> ⚠️ **함정**: Interval을 여러 번 시작하려는 학생이 나옴(토글이라 안 생기지만 코드를 고치다 보면). "스트림은 만들 때마다 새것"이라는 콜드 개념을 설명할 기회로 삼을 것.

**이론 연결**: 2-3, 2-6 (그리고 4차시 복선).

## 생성법 4: Subject — 손으로 미는 발행기 (4분)

**실행·관찰**

1. **`Subject.OnNext`** 클릭 → 누를 때마다 `Subject 수신: 메시지 #1`, `#2`, … 카운트가 올라감.

> 📌 짚어줄 것: 팩토리(시간·횟수가 값을 만듦)와 달리, **내가 버튼을 누르는 그 순간** 값이 발행됨. `OnNext`를 호출하는 시점이 곧 발행 시점.

**코드 워크스루** — 구독은 `Start`에서 한 번, 발행은 버튼에서:

```csharp
// 4) Subject — OnNext를 호출해 직접 발행
m_Subject.Subscribe(msg => Log($"Subject 수신: {msg}")).AddTo(this);
m_SubjectButton
    .OnClickAsObservable()
    .Subscribe(_ => m_Subject.OnNext($"메시지 #{++m_SubjectCount}"))
    .AddTo(this);
```

```csharp
// 필드 — 코드 어디서든 OnNext로 값을 직접 밀어 넣는 발행기
private readonly Subject<string> m_Subject = new();
...
private void OnDestroy()
{
    m_Subject.Dispose(); // 발행기(Subject)도 IDisposable
}
```

- `m_Subject.Subscribe(...)`로 미리 구독해 두고(수신 측), 버튼 클릭 핸들러에서 `m_Subject.OnNext(...)`로 발행(송신 측). **하나의 `Subject`가 받기도(`OnNext`) 내주기도(`Subscribe`) 하며 두 역할을 겸함**(이론 2-4).
- `Subject`는 발행기 자체가 `IDisposable`이라 `OnDestroy`에서 `Dispose()` — 데모는 핫이라 구독 전에 누른 값은 못 받음(클릭 스트림과 같은 성질).

**변형 실습**: 메시지 문자열을 `$"점수 +{++m_SubjectCount * 10}"` 등으로 바꿔 발행 데이터를 가공해 보기.

**이론 연결**: 2-4, 2-6.

## 생성법 5: FromEvent — 기존 event를 스트림으로 (4분)

**실행·관찰**

1. **`event 발생`** 클릭 → `FromEvent 수신: 점수 10`, `20`, … 10씩 증가.

> 📌 짚어줄 것: 화면에 보이는 동작은 Subject와 비슷하지만, 안에서는 **기존 C# `event`** 를 스트림으로 감싼 것. 기존 이벤트 코드를 점진적으로 리액티브에 합류시키는 다리.

**코드 워크스루**

```csharp
// 기존 C# event. FromEvent로 Observable로 감싸는 시연용
private event Action<int> LegacyScoreEvent;
...
// 5) FromEvent — 기존 C# event를 Observable로 감싼다 (구독 해제 시 -=도 자동)
Observable
    .FromEvent<int>(h => LegacyScoreEvent += h, h => LegacyScoreEvent -= h)
    .Subscribe(score => Log($"FromEvent 수신: 점수 {score}"))
    .AddTo(this);
m_EventButton
    .OnClickAsObservable()
    .Subscribe(_ => LegacyScoreEvent?.Invoke(m_LegacyScore += 10))
    .AddTo(this);
```

- `FromEvent<int>(h => e += h, h => e -= h)` — 구독 시작 시 `+=`, 구독 해제(`Dispose`) 시 `-=` 가 **자동** 호출됨. 시그니처(`h => e += h`)는 "외워 쓰는 관용구"(이론 2-5).
- 버튼은 평범한 C# `event` 발생(`LegacyScoreEvent?.Invoke(...)`) — 기존 코드 그대로인데, 그걸 스트림으로 받고 있다는 게 핵심.

**변형 실습**: 증가폭 `+= 10`을 `+= 25`로 바꿔 점수 흐름이 바뀌는 것 확인.

**이론 연결**: 2-5.

# 🎓 마무리 (5분)

## 만들기 5종 요약

| 생성법         | 핵심                    | 마블           |
| -------------- | ----------------------- | -------------- |
| `Range(1, 4)`  | 즉시 흐르고 완료(콜드)  | `-1-2-3-4-\|`  |
| `Timer(2s)`    | N초 뒤 한 번(일회성)    | `------o-\|`   |
| `Interval(1s)` | 무한 — 멈춤은 `Dispose` | `--o--o--o-->` |
| `Subject`      | `OnNext` 시점마다 발행  | `--o----o-o->` |
| `FromEvent`    | 기존 event 감싸기       | event 발생 시  |

- 읽어 둘 핵심 둘: ① **완료(`|`)가 있는 스트림과 없는 스트림**, ② 무한 스트림(`Interval`)은 **`Dispose()`로만** 멈춘다.
- "구독은 등록일 뿐, 값을 **언제** 흘리는지가 생성법마다 다르다."

## 마무리 Q&A 예상 질문

- **Subject vs ReactiveProperty?** "현재값"이 의미 있으면 RP(체력·골드), 순간의 사건이면 Subject(공격 명중). 데모는 Subject만 다뤘고, RP의 실물(바인딩)은 5차시.
- **`AddTo(this)`가 계속 붙던데?** 구독을 오브젝트 수명에 묶는 것 — 4차시의 주제. 지금은 "파괴 시 자동 해제" 한 줄만.
- **`Interval`을 안 멈추면?** 핸들 없이 만든 무한 구독은 누수가 됨 — 4차시에서 Tracker로 "눈으로" 본다.

> 자매 문서: [04. 구독 수명 관리 이론 — 강사용](04.%20구독%20수명%20관리%20이론%20-%20강사용.md)

> 📂 도구: [씬 생성·패키지 도구 가이드](../데모/씬%20생성·패키지%20도구%20가이드.md)
> 데모: `ReactiveBasics` 씬의 §3 수명 섹션(강사 확인·시연용) · 모범 코드(강사용 완성본·도달점) `Assets/Scripts/ReactiveBasics/LifetimeDemo.cs` + `LifetimeProbe.cs`
>
> **진행 흐름**: ① 빈 메서드(`MyLifetimeDemo.Start()`·`MyLifetimeProbe.Start()`)에 강사와 함께 한 줄씩 작성 → ② 실행·확인 → ③ 변형/관찰(누수를 눈으로). 1부도 2부처럼 R3 코드를 비운 스켈레톤으로 배포(주석 없음)하고, 학생은 빈 메서드를 강사와 함께 채운다. [SerializeField] 필드, 보조 헬퍼(`DumpTracker`·`UpdateStatus`), 생명주기(`Awake`(ObservableTracker 활성)·`OnDestroy`)는 제공된다.

## 📚 수업 개요

- **시간**: 약 20분 (준비 + 함께 작성 + 시연 대본 + 마무리)
- **학습 목표**: 구독 누수를 **눈으로 보이게** 만들고(누수 만들기 → Observable Tracker로 잡기 → 프로브 파괴로 AddTo된 것만 사라지는 대비), 수명 관리 기본기 3종(수동 `Dispose`·`AddTo(this)`·`CompositeDisposable`)이 각각 언제 무엇을 정리하는지 확인한다
- **왜 라이브 코딩인가**: 1부도 함께 작성한다. 누수는 "AddTo를 안 붙였더니 영원히 산다"는 **부재의 결과**라, 학생이 직접 ①관리 구독 ②누수 구독 ③프로브 구독을 빈 `Start()`에 나란히 써 보면 "어디에 AddTo를 붙였고 어디엔 안 붙였나"가 손에 남는다. 그 차이가 곧 Tracker에서 보일 누수다.

이 차시는 **눈으로 보이는 누수**가 핵심이다. 빈 `Start()`에 세 종류 구독을 강사와 함께 작성한 뒤, "누수를 의도적으로 만든다 → `Window > Observable Tracker`로 잡는다 → 프로브를 파괴해 AddTo된 것만 사라지는 대비를 본다"의 흐름을 만들어야 한다. 학생은 패키지 임포트 후 같은 씬을 열어 빈 메서드를 강사와 함께 채운다.

| `LifetimeProbe`        | `AddTo(this)`로 묶인 구독을 가진 시연용 오브젝트(파괴되면 구독도 사라짐) |

| 버튼             | 확인할 것                                                                |
| ---------------- | ------------------------------------------------------------------------ |

수업 전(강사):

## 함께 작성 — 세 종류 구독을 빈 Start에 나란히

> 🔧 **오늘 쓸 부품**: 02차시 `Observable.Interval`, 04차시 `AddTo(this)`·`CompositeDisposable`.

빈 메서드를 함께 채운다. `MyLifetimeDemo.Start()`와 `MyLifetimeProbe.Start()`가 비어 있고, 우리는 여기에 **세 종류 구독을 나란히** 작성한다 — ①관리 구독 ②누수 구독 ③프로브 구독. 핵심은 **세 줄을 거의 똑같이 쓰되 끝의 `AddTo`만 다르게** 쓰는 것: 그 한 줄 차이가 곧 Tracker에서 누수냐 아니냐를 가른다.

**①관리 구독** — `CompositeDisposable`에 담겨 `모두 해제`/파괴 시 정리된다. `MyLifetimeDemo.Start()`에 작성:

```csharp
Observable
    .Interval(System.TimeSpan.FromSeconds(5))
    .Subscribe(_ => Debug.Log($"[LifetimeDemo] 관리되는 구독 #{id} tick"))
    .AddTo(m_Disposables); // CompositeDisposable에 등록
```

**②누수 구독** — `AddTo`도, 변수 보관도 없는 전역 스트림 구독. 핸들이 없어 코드로 해제할 방법이 없다. **일부러** 끝의 `AddTo`를 빼고 작성한다:

```csharp
// 잘못된 예: AddTo도, 변수 보관도 없는 전역 스트림 구독 — 이 구독은 영원히 산다
Observable
    .Interval(System.TimeSpan.FromSeconds(5))
    .Subscribe(_ => Debug.Log($"[LifetimeDemo] 누수 구독 #{id} tick (해제 불가!)"));
```

> 📌 **여기서 멈추고 짚기**: 위 두 구독은 줄 수도, 모양도 거의 같다. **마지막 한 줄(`.AddTo(...)`)만** 빠졌을 뿐이다. "이 한 줄의 유무가 다음 단계에서 Tracker에 영원히 남느냐를 가른다"를 예고하고 넘어간다.

**③프로브 구독** — `AddTo(this)`로 묶여 오브젝트가 파괴되면 자동 해제된다. 이건 `MyLifetimeProbe.Start()`에 작성:

```csharp
// LifetimeProbe
Observable
    .Interval(System.TimeSpan.FromSeconds(3))
    .Subscribe(_ => Debug.Log("[LifetimeProbe] 살아 있음 (AddTo로 수명 묶임)"))
    .AddTo(this); // 이 GameObject가 파괴되면 자동 Dispose
```

세 줄을 다 쓰고 나면 다음 대비가 손에 남는다. `활성 구독 조회`(제공된 `DumpTracker`)는 `ObservableTracker.ForEachActiveTask`로 살아 있는 구독을 나열하고, `OnDestroy`(제공됨)에서는 `m_Disposables.Dispose()`로 그릇째 폐기한다 — 이 둘은 스켈레톤에 이미 있으니 우리가 쓸 건 위 세 구독뿐이다.

```
관리 구독:  AddTo(CompositeDisposable) → "모두 해제"로 정리 ✅
프로브:     AddTo(this)               → 파괴 시 자동 해제 ✅
누수:       (없음)                    → Tracker에 영구 잔류 ❌ → AddTo가 답
```

> 📌 **던질 질문**: "세 구독 중 두 개는 안전하고 하나는 누수다. 코드만 보고 어느 게 누수인지, 왜 그런지 말해 보라." (끝의 `AddTo` 한 줄 — 다음 실행에서 직접 확인한다.)

## 작성 후 실행·확인 — 시연 대본 (~20분, 클라이맥스)

위에서 세 구독을 다 채웠으면 이제 실행해 **누수를 눈으로 본다.** `Window > Observable Tracker`를 띄운 채 진행한다. 권장 순서:

### 조작 ① — 관리 구독 (CompositeDisposable)

- `구독 추가`를 3번 → `활성 구독 조회`(카운트 증가 확인) → `모두 해제` → 다시 `활성 구독 조회`(줄어듦).
- 멘트: "`CompositeDisposable`에 담으면 한 번에 정리된다."

### 조작 ② — AddTo 자동 해제 (프로브)

- `활성 구독 조회`로 프로브 구독 확인 → `프로브 파괴` → 다시 조회 → **프로브 구독이 사라짐**.
- Console: `[LifetimeProbe] 파괴됨 — 구독도 자동 해제`.
- 멘트: "`AddTo(this)`는 오브젝트 파괴에 수명을 묶는다."

### 🔻 고의 실패 시연 — 누수 만들기

- `누수 만들기`를 누른다. 이 구독은 `AddTo`도 변수 보관도 없는 전역 `Interval(5s)` 구독 — **핸들이 없어 해제할 방법이 없다**. `활성 구독 조회`/`Observable Tracker`에서 카운트가 계속 남고, Console에 `누수 구독 #N tick (해제 불가!)`가 5초마다 영원히 찍힌다.
- **눈으로 보이기**: `모두 해제`를 눌러도 누수 구독은 사라지지 않는다(관리 구독만 정리됨) — Tracker에 남아 대비됨.
- **복구**: 누수 구독은 코드로 회수 불가 → **Play 정지**가 유일한 정리. 교훈: "그래서 처음부터 `AddTo`/`Dispose`로 묶어야 한다." (대조: 프로브는 파괴만으로 깨끗이 사라졌다.)

### Tracker 윈도우로 마무리

- `Window > Observable Tracker` 윈도우로 같은 내용을 GUI로 본다 — StackTrace로 생성 지점 추적까지. "씬을 나갔는데 사라지지 않는 구독이 있으면 그게 누수"라는 점검 루틴을 복기.

# 🎓 마무리

## 마무리 멘트

- 구독 = `IDisposable`. **`Dispose()`가 곧 해제** — 모든 수명 관리의 토대.
- 기본기 3종 — 수동 `Dispose`(해제 시점을 코드가 정함) · **`AddTo(this)`**(GameObject 수명에 묶음) · **`CompositeDisposable`**(묶음 해제, 순수 C# 클래스의 표준).
- `AddTo`를 빠뜨리면 똑같이 누수 — 특히 **전역 스트림(`EveryUpdate` 등)은 반드시** 묶을 것.
- **`ObservableTracker`** 로 활성 구독을 추적 — 코드(`ForEachActiveTask`)와 에디터 윈도우(`Window > Observable Tracker`) 양쪽.
- 발행기(`Subject`/`ReactiveProperty`)도 `IDisposable` — 소유자가 책임지고 정리.

## 체크리스트

- [ ] "구독 누수"가 무엇이고 어떤 증상(누수·MissingReferenceException)을 내는지 설명할 수 있음
- [ ] `AddTo(this)`가 하는 일과, 그래도 누수가 날 수 있는 경우(AddTo 누락)를 말할 수 있음
- [ ] `CompositeDisposable`의 `Clear`와 `Dispose` 차이를 앎
- [ ] 순수 C# 클래스에서 구독을 정리하는 표준(CompositeDisposable + Dispose)을 쓸 수 있음
- [ ] ObservableTracker로 누수를 찾는 루틴(켜기→씬 출입→목록 확인→StackTrace)을 수행할 수 있음
- [ ] 발행기(Subject 등)도 Dispose 대상임을 앎

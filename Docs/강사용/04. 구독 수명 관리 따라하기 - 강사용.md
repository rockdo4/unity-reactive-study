## 📚 수업 개요

- **시간**: 약 20분 (준비 + 함께 작성 + 시연 대본 + 마무리)
- **학습 목표**: 구독 누수를 눈으로 보이게 만들고(누수 만들기 → Observable Tracker로 잡기 → 프로브 파괴로 AddTo된 것만 사라지는 대비), 수명 관리 기본기 3종(수동 `Dispose`·`AddTo(this)`·`CompositeDisposable`)이 각각 언제 무엇을 정리하는지 확인.
- **왜 라이브 코딩**: 누수는 "AddTo를 안 붙여서 영원히 산다"는 부재의 결과 — ①관리 ②누수 ③프로브 구독을 빈 `Start()`에 나란히 직접 써 봐야 "어디에 AddTo 붙였고 어디엔 안 붙였나"가 손에 남음. 그 차이가 곧 Tracker에서 보일 누수.

- 핵심: 빈 `Start()`에 세 종류 구독 작성 → 누수 의도적으로 만들기 → `Window > Observable Tracker`로 잡기 → 프로브 파괴해 AddTo된 것만 사라지는 대비. 학생은 패키지 임포트 후 같은 씬 열어 빈 메서드 채움.

## 함께 작성 — 빈 두 `Start()` 채우기

> 🔧 **오늘 쓸 부품**: 02차시 `Observable.Interval`·`OnClickAsObservable`, 04차시 `AddTo(this)`·`CompositeDisposable`.

`MyLifetimeDemo.Start()`와 `MyLifetimeProbe.Start()`가 둘 다 `{ }`로 비어 있음(스켈레톤). 아래 블록을 위에서 아래로 그대로 침. `[SerializeField] 버튼·텍스트` 필드, 보조 헬퍼(`DumpTracker`·`UpdateStatus`), 생명주기(`Awake`에서 `ObservableTracker.EnableTracking = true`, `OnDestroy`에서 `m_Disposables.Dispose()`)는 스켈레톤에 이미 있음 — 우리가 채울 건 두 `Start()` 본문뿐.

다섯 버튼을 차례로 잇되, 핵심은 **`구독 추가`(관리)와 `누수 만들기`(누수)의 안쪽 `Interval` 구독이 거의 똑같고 끝의 `.AddTo(m_Disposables)` 한 줄만 다르다**는 것 — 그 한 줄이 Tracker에서 누수냐 아니냐를 가름.

**① 관리 두 버튼 — `구독 추가` + `모두 해제`.** `MyLifetimeDemo.Start()`에 작성:

```csharp
// 구독 추가 — 누를 때마다 5초 Interval 구독을 만들어 CompositeDisposable에 담는다
m_AddSubscriptionButton
    .OnClickAsObservable()                                  // 버튼 클릭을 스트림으로
    .Subscribe(_ =>
    {
        int id = ++m_AddedCount;                            // 이번 구독 번호(필드는 스켈레톤 제공)
        Observable
            .Interval(System.TimeSpan.FromSeconds(5))       // 5초마다 tick
            .Subscribe(_ => Debug.Log($"[LifetimeDemo] 관리되는 구독 #{id} tick"))
            .AddTo(m_Disposables);                          // ✅ 그릇에 등록 → "모두 해제"/파괴 시 정리
        UpdateStatus($"구독 #{id} 추가");                   // 상태 텍스트 갱신(헬퍼 제공)
    })
    .AddTo(this);                                           // 이 바깥 구독은 컴포넌트 수명에 묶음

// 모두 해제 — 그릇에 담긴 관리 구독을 Clear() 한 방에 전부 해제
m_DisposeAllButton
    .OnClickAsObservable()
    .Subscribe(_ =>
    {
        m_Disposables.Clear();                              // 담아 둔 구독 전부 해제 (그릇은 재사용)
        UpdateStatus("전체 해제 (Clear)");
    })
    .AddTo(this);
```

**② 누수 버튼 — `누수 만들기`.** 위 `구독 추가`와 거의 같지만 안쪽 `Interval` 구독 끝에 **`.AddTo(...)`를 일부러 뺌**. 같은 `Start()`에 이어서:

```csharp
// 누수 만들기 — AddTo도 변수 보관도 없는 전역 Interval 구독. 핸들이 없어 코드로 해제 불가
m_LeakButton
    .OnClickAsObservable()
    .Subscribe(_ =>
    {
        int id = ++m_LeakedCount;
        Observable
            .Interval(System.TimeSpan.FromSeconds(5))
            .Subscribe(_ => Debug.Log($"[LifetimeDemo] 누수 구독 #{id} tick (해제 불가!)"));
            // ❌ 여기 끝에 .AddTo(...)가 없음 → ①의 관리 구독과 유일한 차이. 핸들이 없어 영원히 산다
        UpdateStatus($"누수 구독 #{id} 생성 — Tracker로 확인해 보세요");
    })
    .AddTo(this);
```

> 📌 **여기서 멈추고 짚기**: `구독 추가`(①)와 `누수 만들기`(②)의 안쪽 `Interval` 구독은 줄 수·모양 거의 동일 — 마지막 `.AddTo(m_Disposables)` 한 줄만 빠졌을 뿐. 이 한 줄 유무가 다음 단계에서 Tracker에 영원히 남느냐를 가름. 예고하고 넘어감.

**③ 조회·파괴 두 버튼 — `활성 구독 조회` + `프로브 파괴`.** `MyLifetimeDemo.Start()` 마지막에 작성(끝에 `UpdateStatus("대기 중")`로 첫 화면 표시):

```csharp
// 활성 구독 조회 — 제공된 DumpTracker()가 ObservableTracker로 살아 있는 구독을 나열
m_TrackerButton.OnClickAsObservable().Subscribe(_ => DumpTracker()).AddTo(this);

// 프로브 파괴 — 프로브 오브젝트를 Destroy. AddTo(this)로 묶인 구독이 함께 사라지는지 확인용
m_DestroyProbeButton
    .OnClickAsObservable()
    .Subscribe(_ =>
    {
        if (m_ProbeObject != null)
        {
            Destroy(m_ProbeObject);                         // 파괴되면 ④의 AddTo(this) 프로브 구독이 자동 해제됨
            UpdateStatus("프로브 파괴 — Tracker에서 프로브 구독이 사라졌는지 확인");
        }
        else
        {
            UpdateStatus("프로브가 이미 파괴되었습니다");
        }
    })
    .AddTo(this);

UpdateStatus("대기 중"); // 첫 화면 상태 텍스트
```

**④ 프로브 구독 — `AddTo(this)`로 묶기.** `MyLifetimeProbe.Start()`(별도 컴포넌트)에 작성. 끝에 `.AddTo(this)`가 있어 ②와 대비됨:

```csharp
Observable
    .Interval(System.TimeSpan.FromSeconds(3))
    .Subscribe(_ => Debug.Log("[LifetimeProbe] 살아 있음 (AddTo로 수명 묶임)"))
    .AddTo(this); // ✅ 이 GameObject가 파괴되면 자동 Dispose (②의 누수와 달리 끝줄이 있음)
```

세 종류 구독이 손에 남음:

```
관리(구독 추가):  AddTo(CompositeDisposable) → "모두 해제"로 정리 ✅
프로브:           AddTo(this)               → 파괴 시 자동 해제 ✅
누수(누수 만들기):(없음)                    → Tracker에 영구 잔류 ❌ → AddTo가 답
```

- 요점: 세 구독 중 둘은 안전, 하나는 누수. 차이는 끝의 `AddTo` 한 줄 — 다음 실행에서 직접 확인.

## 작성 후 실행·확인 — 시연 대본 (~20분, 클라이맥스)

세 구독 다 채웠으면 실행해 누수를 눈으로. `Window > Observable Tracker` 띄운 채 진행. 권장 순서:

### 조작 ① — 관리 구독 (CompositeDisposable)

- `구독 추가` 3번 → `활성 구독 조회`(카운트 증가) → `모두 해제` → 다시 `활성 구독 조회`(줄어듦).
- 요점: `CompositeDisposable`에 담으면 한 번에 정리됨.

### 조작 ② — AddTo 자동 해제 (프로브)

- `활성 구독 조회`로 프로브 구독 확인 → `프로브 파괴` → 다시 조회 → 프로브 구독 사라짐.
- Console: `[LifetimeProbe] 파괴됨 — 구독도 자동 해제`.
- 요점: `AddTo(this)`는 오브젝트 파괴에 수명을 묶음.

### 🔻 고의 실패 시연 — 누수 만들기

- `누수 만들기` 누름. 이 구독은 `AddTo`도 변수 보관도 없는 전역 `Interval(5s)` 구독 — 핸들 없어 해제 불가. `활성 구독 조회`/`Observable Tracker`에서 카운트 계속 남고, Console에 `누수 구독 #N tick (해제 불가!)`가 5초마다 영원히 찍힘.
- **눈으로 보이기**: `모두 해제` 눌러도 누수 구독은 안 사라짐(관리 구독만 정리) — Tracker에 남아 대비됨.
- **복구**: 누수 구독은 코드로 회수 불가 → Play 정지가 유일한 정리. 교훈: 처음부터 `AddTo`/`Dispose`로 묶을 것. (대조: 프로브는 파괴만으로 깨끗이 사라짐.)

### Tracker 윈도우로 마무리

- `Window > Observable Tracker` 윈도우로 같은 내용 GUI로 — StackTrace로 생성 지점 추적까지. "씬을 나갔는데 사라지지 않는 구독 = 누수" 점검 루틴 복기.

# 🎓 마무리

## 마무리 요점

- 구독 = `IDisposable`. `Dispose()`가 곧 해제 — 모든 수명 관리의 토대.
- 기본기 3종 — 수동 `Dispose`(해제 시점을 코드가 정함) · `AddTo(this)`(GameObject 수명에 묶음) · `CompositeDisposable`(묶음 해제, 순수 C# 클래스의 표준).
- `AddTo` 빠뜨리면 똑같이 누수 — 특히 전역 스트림(`EveryUpdate` 등)은 반드시 묶을 것.
- `ObservableTracker`로 활성 구독 추적 — 코드(`ForEachActiveTask`)와 에디터 윈도우(`Window > Observable Tracker`) 양쪽.
- 발행기(`Subject`/`ReactiveProperty`)도 `IDisposable` — 소유자가 책임지고 정리.

## 체크리스트

- [ ] "구독 누수"가 무엇이고 어떤 증상(누수·MissingReferenceException)을 내는지 설명할 수 있음
- [ ] `AddTo(this)`가 하는 일과, 그래도 누수가 날 수 있는 경우(AddTo 누락)를 말할 수 있음
- [ ] `CompositeDisposable`의 `Clear`와 `Dispose` 차이를 앎
- [ ] 순수 C# 클래스에서 구독을 정리하는 표준(CompositeDisposable + Dispose)을 쓸 수 있음
- [ ] ObservableTracker로 누수를 찾는 루틴(켜기→씬 출입→목록 확인→StackTrace)을 수행할 수 있음
- [ ] 발행기(Subject 등)도 Dispose 대상임을 앎

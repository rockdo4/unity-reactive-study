## 📚 수업 개요

- **시간**: 약 20분 (준비 + 시연 대본 + 마무리)
- **학습 목표**: 구독 누수를 **눈으로 보이게** 만들고(누수 만들기 → Observable Tracker로 잡기 → 프로브 파괴로 AddTo된 것만 사라지는 대비), 수명 관리 기본기 3종(수동 `Dispose`·`AddTo(this)`·`CompositeDisposable`)이 각각 언제 무엇을 정리하는지 확인한다
- **왜 실행·관찰인가**: 1부 데모이므로 코드를 작성하지 않고 **완성본을 실행·관찰**한다(학생은 패키지 임포트 후 같은 씬을 연다)

이 차시는 **눈으로 보이는 누수**가 핵심이다. "누수를 의도적으로 만든다 → `Window > Observable Tracker`로 잡는다 → 프로브를 파괴해 AddTo된 것만 사라지는 대비를 본다"의 흐름을 만들어야 한다. 1부 데모이므로 코드를 작성하지 않고 **완성본을 실행·관찰**한다(학생은 패키지 임포트 후 같은 씬을 연다).

| 구성요소               | 역할                                                                     |
| ---------------------- | ------------------------------------------------------------------------ |
| `ReactiveBasics` 씬 §3 | 버튼 5개 + 상태/Tracker 텍스트 + 프로브 오브젝트                         |
| `LifetimeDemo`         | 관리 구독(CompositeDisposable)·누수 구독·Tracker 조회를 다루는 본체      |
| `LifetimeProbe`        | `AddTo(this)`로 묶인 구독을 가진 시연용 오브젝트(파괴되면 구독도 사라짐) |

씬 버튼과 확인 포인트:

| 버튼             | 확인할 것                                                                |
| ---------------- | ------------------------------------------------------------------------ |
| `구독 추가`      | `CompositeDisposable`에 담김 — 카운트 증가                               |
| `모두 해제`      | `Clear()` 한 방에 전부 해제                                              |
| `누수 만들기`    | `AddTo` 없는 구독 — 해제할 방법이 없음을 코드에서 확인                   |
| `활성 구독 조회` | `ObservableTracker.ForEachActiveTask`로 살아 있는 구독 나열. 누수가 잡힘 |
| `프로브 파괴`    | `AddTo(this)`로 묶인 구독이 오브젝트 파괴와 함께 사라지는 것 확인        |

## 준비 — 코드 근거: 무엇이 누수이고 무엇이 안전한가

`LifetimeDemo`(+`LifetimeProbe`)의 세 종류 구독을 미리 구분해 둔다.

**관리 구독** — `CompositeDisposable`에 담겨 `모두 해제`/파괴 시 정리된다:

```csharp
Observable
    .Interval(System.TimeSpan.FromSeconds(5))
    .Subscribe(_ => Debug.Log($"[LifetimeDemo] 관리되는 구독 #{id} tick"))
    .AddTo(m_Disposables); // CompositeDisposable에 등록
```

**누수 구독** — `AddTo`도, 변수 보관도 없는 전역 스트림 구독. 핸들이 없어 코드로 해제할 방법이 없다:

```csharp
// 잘못된 예: AddTo도, 변수 보관도 없는 전역 스트림 구독 — 이 구독은 영원히 산다
Observable
    .Interval(System.TimeSpan.FromSeconds(5))
    .Subscribe(_ => Debug.Log($"[LifetimeDemo] 누수 구독 #{id} tick (해제 불가!)"));
```

**프로브 구독** — `AddTo(this)`로 묶여 오브젝트가 파괴되면 자동 해제된다:

```csharp
// LifetimeProbe
Observable
    .Interval(System.TimeSpan.FromSeconds(3))
    .Subscribe(_ => Debug.Log("[LifetimeProbe] 살아 있음 (AddTo로 수명 묶임)"))
    .AddTo(this); // 이 GameObject가 파괴되면 자동 Dispose
```

`활성 구독 조회`는 `ObservableTracker.ForEachActiveTask`로 살아 있는 구독을 나열한다. `OnDestroy`에서는 `m_Disposables.Dispose()`로 그릇째 폐기한다.

```
관리 구독:  AddTo(CompositeDisposable) → "모두 해제"로 정리 ✅
프로브:     AddTo(this)               → 파괴 시 자동 해제 ✅
누수:       (없음)                    → Tracker에 영구 잔류 ❌ → AddTo가 답
```

## 시연 대본 (~20분, 클라이맥스)

`Window > Observable Tracker`를 띄운 채 진행한다. 권장 순서:

### 조작 ① — 관리 구독 (CompositeDisposable)

- `구독 추가`를 3번 → `활성 구독 조회`(카운트 증가 확인) → `모두 해제` → 다시 `활성 구독 조회`(줄어듦).
- 멘트: "`CompositeDisposable`에 담으면 한 번에 정리된다."

### 조작 ② — AddTo 자동 해제 (프로브)

- `활성 구독 조회`로 프로브 구독 확인 → `프로브 파괴` → 다시 조회 → **프로브 구독이 사라짐**.
- Console: `[LifetimeProbe] 파괴됨 — 구독도 자동 해제`.
- 멘트: "`AddTo(this)`는 오브젝트 파괴에 수명을 묶는다."

### 🔻 고의 실패 시연 — 누수 만들기

- `누수 만들기`를 누른다. 이 구독은 `AddTo`도 변수 보관도 없는 전역 `Interval(5s)` 구독 — **핸들이 없어 해제할 방법이 없다**. `활성 구독 조회`/`Observable Tracker`에서 카운트가 계속 남고, Console에 `누수 구독 #N tick (해제 불가!)`가 5초마다 영원히 찍힌다.
- **눈으로 보이기**: `모두 해제`를 눌러도 누수 구독은 사라지지 않는다(관리 구독만 정리됨) — Tracker에 남아 대비됨.
- **복구**: 누수 구독은 코드로 회수 불가 → **Play 정지**가 유일한 정리. 교훈: "그래서 처음부터 `AddTo`/`Dispose`로 묶어야 한다." (대조: 프로브는 파괴만으로 깨끗이 사라졌다.)

### Tracker 윈도우로 마무리

- `Window > Observable Tracker` 윈도우로 같은 내용을 GUI로 본다 — StackTrace로 생성 지점 추적까지. "씬을 나갔는데 사라지지 않는 구독이 있으면 그게 누수"라는 점검 루틴을 복기.

# 🎓 마무리

## 마무리 멘트

- 구독 = `IDisposable`. **`Dispose()`가 곧 해제** — 모든 수명 관리의 토대.
- 기본기 3종 — 수동 `Dispose`(해제 시점을 코드가 정함) · **`AddTo(this)`**(GameObject 수명에 묶음) · **`CompositeDisposable`**(묶음 해제, 순수 C# 클래스의 표준).
- `AddTo`를 빠뜨리면 똑같이 누수 — 특히 **전역 스트림(`EveryUpdate` 등)은 반드시** 묶을 것.
- **`ObservableTracker`** 로 활성 구독을 추적 — 코드(`ForEachActiveTask`)와 에디터 윈도우(`Window > Observable Tracker`) 양쪽.
- 발행기(`Subject`/`ReactiveProperty`)도 `IDisposable` — 소유자가 책임지고 정리.

## 체크리스트

- [ ] "구독 누수"가 무엇이고 어떤 증상(누수·MissingReferenceException)을 내는지 설명할 수 있음
- [ ] `AddTo(this)`가 하는 일과, 그래도 누수가 날 수 있는 경우(AddTo 누락)를 말할 수 있음
- [ ] `CompositeDisposable`의 `Clear`와 `Dispose` 차이를 앎
- [ ] 순수 C# 클래스에서 구독을 정리하는 표준(CompositeDisposable + Dispose)을 쓸 수 있음
- [ ] ObservableTracker로 누수를 찾는 루틴(켜기→씬 출입→목록 확인→StackTrace)을 수행할 수 있음
- [ ] 발행기(Subject 등)도 Dispose 대상임을 앎

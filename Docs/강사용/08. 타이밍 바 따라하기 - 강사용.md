# 08. 타이밍 바 따라하기 — 강사용

> 진행 메모. 타이밍 바 capstone의 **UI 제공 스캐폴드(B) 버전** 따라하기. 학생 패키지 `R3Lecture-Student-3-TimingBar`를 임포트한 상태에서 시작한다 — 화면(UI)과 Inspector 연결은 끝났고, `MyTimingBarModel`(규칙)·`MyTimingBarPresenter`(R3) 본문만 채운다.

> - **게임**: 좌우로 왕복하는 마커가 트랙 가운데 존을 지날 때 버튼(또는 Space)을 눌러 점수를 얻는 한 줄 타이밍 게임. START → 카운트다운 → 정해진 횟수 판정 → 결과 → 다시 시작.
> - **채울 곳**: `MyTimingBarModel` 규칙 4개(`ZoneOf`/`Judge`/`ApplyHit`/`ResetForNewGame`) + `MyTimingBarPresenter` **전부**(런타임 필드·마커 이동·존 강조·입력 판정·라운드 흐름).
> - **강사 시연용 정본 씬**: `Tools > Reactive Study > 03. 타이밍 바(과제 정본) 씬 생성` → `TimingBar.unity`(완성본 `TimingBarPresenter` 동작 확인용, git 비커밋).
>
> 권장 2차시: ①(브리핑 + 0~3단계) ②(4~6단계). 숙제로 낼 거면 0~3단계까지 같이 짚고 4~6단계는 명세만 주고 자습.

## 📚 수업 개요

- **시간**: 약 2차시(개념 브리핑 + 0~3단계 / 4~6단계). 또는 1차시 라이브 + 자습.
- **난이도**: 종합(capstone) · **선수 지식**: 1부 전부. 특히 `EveryUpdate`·`ReactiveProperty`(프레임·상태), `Select`·`DistinctUntilChanged`·`Merge`·`WithLatestFrom`·`Take`·`Share`(합성), `SubscribeAwait`·`UniTask.Delay`(비동기), `AddTo`·`CompositeDisposable`·`ObservableTracker`(수명).
- **학습 목표**
    - `Update()`·코루틴·상태 플래그 **0개**로 작은 게임을 완성한다.
    - 클리커에서 안 다룬 **프레임 스트림·입력 시점 샘플링·시퀀스 종료·비동기 라운드 흐름**을 한 게임으로 종합한다.
    - 빈 메서드조차 없는 스켈레톤에서 **스트림 구조를 스스로 설계**한다(메서드 분리·수명 그릇 배치).

> 💡 클리커와 결이 다른 점 한 줄: **클리커는 끊임없이 쌓는 게임**(`Interval`로 자동 수입, `Scan`으로 콤보 누적), **타이밍 바는 한 순간을 맞히는 게임**(`EveryUpdate`로 마커, `WithLatestFrom`으로 누른 순간 샘플). 같은 R3로 정반대 리듬의 게임을 만든다.

# Part 1. 개념 브리핑 (~12분)

코드 전, 칠판/슬라이드로 세 가지만.

## 1-1. 오늘 만들 게임 — 완성 스펙

강사 정본 씬(`TimingBar.unity`) Play로 1분 시연.

- START → 3·2·1 **카운트다운** → 라운드 시작. 카운트다운 동안 마커가 움직여 리듬 워밍업.
- 마커가 좌우로 **왕복**, 트랙 가운데에 **Good 존**(넓음)과 그 안에 겹친 **Perfect 존**(좁음).
- 마커가 존에 **들어오는 순간** 트랙 색이 바뀜(타이밍 시각 보조).
- 존에서 누르면 **Perfect +3 / Good +1**, 존 밖이면 **Miss(콤보 0)**. 콤보가 길수록 점수 배수 ↑.
- **정해진 횟수(12회)** 를 치면 라운드 끝 → **최종 점수·최고 콤보** 표시 → START로 다시 시작.

> 💡 전달 요점: `Update()` 0개, 코루틴 0개, 상태 플래그 0개. 마커도 라운드도 전부 스트림.

## 1-2. Model과 Presenter를 나눈다

| 클래스                 | 정체           | 책임                                         | 모르는 것                 |
| ---------------------- | -------------- | -------------------------------------------- | ------------------------- |
| `MyTimingBarModel`     | 순수 C# 클래스 | 위상(0~1) → 존 → 판정 → 점수·콤보 규칙       | UI·프레임·`Time`·입력장치 |
| `MyTimingBarPresenter` | MonoBehaviour  | 프레임으로 마커 이동, 입력 합성·샘플, UI표시 | 게임 규칙(점수·판정 공식) |

- Model은 `위상(float) → 판정/점수`라는 **순수 규칙**만 안다 → UI 없이 단위 테스트 가능, 화면 바꿔도 안 깨짐.
- 프레임·`Time`·`RectTransform`·`Keyboard`는 전부 Unity 의존 → Presenter 소관.

> 📌 명분 한 문장(아키텍처 논쟁 회피): "Model이 화면·프레임을 모르면, 판정·점수 규칙을 따로 테스트할 수 있고 화면을 갈아치워도 규칙이 안 깨진다."

## 1-3. B 스켈레톤이 주는 것 / 채울 것

스켈레톤 두 파일을 함께 열어 확인.

| 파일                   | 이미 제공                                                                       | 채울 것                                                                                                                      |
| ---------------------- | ------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| `MyTimingBarModel`     | `Score`/`Combo`/`MaxCombo` RP, 존 폭·점수 상수, 생성자(`AddTo`), `Dispose`      | `ZoneOf`·`Judge`·`ApplyHit`·`ResetForNewGame` **본문**                                                                       |
| `MyTimingBarPresenter` | `[SerializeField]` 11개(트랙·마커·버튼·텍스트·속도·횟수) — **UI 연결까지 완료** | **런타임 필드 + 색 상수 4개(None/Good/Perfect/Miss) + `Start()` + 마커/존/판정/라운드 스트림 전부 + `OnDestroy`** (구조부터) |

> ❗ **클리커와 결정적 차이**: 클리커는 `BindInput`/`BindView` 빈 메서드를 줬다. **여기는 그것도 없다.** `Start()`만 비어 있고, 어떤 메서드로 쪼갤지·각 스트림을 어디에 둘지부터 학생이 정한다. 이 문서는 정본과 같은 구조(`BindMarker`/`BindView`/`BindGameLoop` + `PlayOneGameAsync`)로 안내하되, **다른 분리도 정답**임을 강조.
>
> 📌 `[SerializeField]` 11개는 씬에 **이미 연결**되어 있음(`m_TrackImage`·`m_Marker`·`m_HitButton`·`m_StartButton`·`m_ScoreText`·`m_ComboText`·`m_AttemptText`·`m_JudgementText`·`m_StateText`·`m_MarkerSpeed=0.6`·`m_AttemptsPerRound=12`). 학생이 빈 GameObject를 새로 만들지만 않으면 연결은 건드릴 일 없음.

### 1단계. Model 규칙 — 순수 함수부터 (~15분)

> 🔧 **오늘 쓸 부품**
>
> - `ReactiveProperty<int>`: 현재값을 들고, `.Value`가 바뀌면 구독자에게 자동 통지하는 값(`Score`/`Combo`/`MaxCombo`는 제공됨, 읽기·쓰기).
> - (Model엔 R3 스트림이 없다. 여기는 **순수 C#** — 그래서 화면 없이 테스트 가능.)

규칙부터 채운다. 모든 게 위상(0~1) 하나로 돌아간다. `MyTimingBarModel`의 빈 메서드 네 개를 채움. 상수·RP·생성자·`Dispose`는 제공됨(읽기만).

먼저 **위상 → 존**. 트랙을 0~1로 볼 때 중앙(0.5)에서 얼마나 떨어졌나로 판정:

```csharp
// 위상(0~1)이 어느 구간인지 — 순수 함수(입력만으로 결정, 부수효과 없음)
public static Zone ZoneOf(float phase)
{
    float distance = Math.Abs(phase - 0.5f);   // 중앙(0.5)에서의 거리
    if (distance <= PerfectHalfWidth)          // 0.04 이내면 가장 좁은 Perfect
        return Zone.Perfect;
    if (distance <= GoodHalfWidth)             // 0.12 이내면 Good (Perfect를 포함)
        return Zone.Good;
    return Zone.None;                          // 그 밖은 존 밖
}
```

다음 **존 → 판정**(존 enum을 판정 enum으로 옮김):

```csharp
// ZoneOf 결과를 한 번 치기의 판정으로 — 역시 순수 함수
public static Judgement Judge(float phase) =>
    ZoneOf(phase) switch
    {
        Zone.Perfect => Judgement.Perfect,
        Zone.Good => Judgement.Good,
        _ => Judgement.Miss,               // None → Miss
    };
```

핵심 **한 번 치기 반영** — 콤보·최고콤보·점수를 한곳에서 갱신(Model이 콤보를 단일 소유):

```csharp
// 한 번의 치기를 상태에 반영하고 판정을 돌려준다.
public Judgement ApplyHit(float phase)
{
    Judgement judgement = Judge(phase);
    if (judgement == Judgement.Miss)
    {
        Combo.Value = 0;                   // 빗나가면 콤보 끊김 → 구독자(콤보 텍스트)에 자동 통지
        return judgement;
    }

    Combo.Value += 1;                      // 성공 → 콤보 +1
    if (Combo.Value > MaxCombo.Value)
        MaxCombo.Value = Combo.Value;      // 최고 콤보 갱신

    int basePoints = judgement == Judgement.Perfect ? PerfectScore : GoodScore;  // 3 또는 1
    int multiplier = 1 + Combo.Value / ComboPerMultiplier;  // 5콤보마다 배수 +1 (정수 나눗셈)
    Score.Value += basePoints * multiplier;                 // 점수 가산 → 점수 텍스트 자동 갱신
    return judgement;
}
```

마지막 **새 게임 초기화**:

```csharp
// 새 라운드 시작 시 상태를 0으로 (대입 순간 UI도 자동으로 0으로 돌아감)
public void ResetForNewGame()
{
    Score.Value = 0;
    Combo.Value = 0;
    MaxCombo.Value = 0;
}
```

> 📌 짚을 점: 여기엔 R3 스트림이 한 줄도 없다. `ReactiveProperty.Value`에 **대입만** 하면, 나중에 Presenter가 건 구독이 알아서 텍스트를 갱신함. "값이 바뀌면 통지"는 RP가, "어떻게 표시할지"는 Presenter가 — 책임이 갈린다.
>
> 💡 **단위 테스트 가능**의 실감: `ZoneOf(0.5f)==Perfect`, `ZoneOf(0.5f+0.1f)==Good`, `ZoneOf(0f)==None`을 칠판에서 손으로 검산. 화면·프레임 없이 규칙이 맞는지 확인되는 게 Model을 순수 C#로 둔 이득.

**실행·관찰**: 아직 Presenter가 비어 화면 변화는 없음. 컴파일 에러 0만 확인(`Unity_GetConsoleLogs`).

### 2단계. 마커 왕복 — 프레임 스트림 (~15분)

> 🔧 **오늘 쓸 부품**
>
> - `Observable.EveryUpdate()`: 매 프레임 `Unit`을 흘리는 **전역 스트림**(= 게임 루프). 무한이라 **`AddTo` 필수**.
> - `ReactiveProperty<float>`: 마커의 **현재 위상**을 담는 그릇. 잠시 뒤(4단계) 입력 판정이 이 "지금 위상"을 샘플한다.

여기서부터 Presenter. 스켈레톤은 `Start()`만 비어 있으니 **런타임 필드부터 직접 선언**한다(클래스 안, `[SerializeField]`들 아래):

```csharp
private MyTimingBarModel m_Model;          // 규칙 (1단계에서 채운 Model)
private ReactiveProperty<float> m_Phase;   // 마커의 현재 위상(0~1) — 입력이 샘플할 Hot 값
private ReactiveProperty<int> m_Attempt;   // 이번 라운드의 시도 횟수
private float m_UsableWidth;               // 마커가 실제로 움직일 수 있는 가로 폭(픽셀)
```

존 색에 쓸 색 상수도 추가(트랙은 존마다 색이 바뀐다):

```csharp
private static readonly Color NoneColor = new(0.2f, 0.2f, 0.25f, 1f);   // 존 밖(어두움)
private static readonly Color GoodColor = new(0.2f, 0.5f, 0.9f, 1f);    // Good(파랑)
private static readonly Color PerfectColor = new(0.95f, 0.75f, 0.15f, 1f); // Perfect(금색)
private static readonly Color MissColor = new(0.9f, 0.3f, 0.3f, 1f);    // Miss(빨강) — 4단계 판정 표시용
```

`Start()`에서 모델 생성·필드 초기화·마커 폭 계산:

```csharp
private void Start()
{
    m_Model = new MyTimingBarModel();                  // 규칙 객체 생성
    m_Phase = new ReactiveProperty<float>(0f).AddTo(this);   // 위상 0에서 시작 · 수명은 this
    m_Attempt = new ReactiveProperty<int>(0).AddTo(this);
    // 트랙 폭에서 마커 폭을 빼야 마커가 끝에서 트랙 밖으로 안 삐져나감
    m_UsableWidth = m_TrackImage.rectTransform.rect.width - m_Marker.rect.width;

    SpinMarker();   // ↓ 임시: 마커를 계속 왕복시킴(5단계에서 한 판 안으로 옮긴다)
}
```

마커를 매 프레임 움직이는 메서드 — **위상 계산 → RP에 기록 → 화면 위치 반영**:

```csharp
// 마커가 좌우로 끝없이 왕복. 위상은 시간으로 만들고(Mathf.PingPong), 위치는 위상으로 만든다.
private void SpinMarker()
{
    Observable
        .EveryUpdate()                          // 매 프레임 한 번
        .Subscribe(_ =>
        {
            // PingPong(t, 1) = 0→1→0 왕복. m_MarkerSpeed(0.6)는 초당 위상 속도
            float phase = Mathf.PingPong(Time.time * m_MarkerSpeed, 1f);
            m_Phase.Value = phase;              // "지금 위상"을 RP에 기록(4단계 입력이 이걸 샘플)
            // 위상 0~1 → 중앙 기준 좌우 위치(픽셀). 0.5가 중앙(x=0)
            m_Marker.anchoredPosition = new Vector2((phase - 0.5f) * m_UsableWidth, 0f);
        })
        .AddTo(this);                           // ★ 무한 스트림 — 빠뜨리면 씬 나가도 영원히 돎(6단계 누수)
}
```

```
시간:  0초 ----------- 1.67초 ----------- 3.3초 ---->
위상:  0 →→→→→→→→ 1 →→→→→→→→ 0  (PingPong, 좌↔우 왕복)
```

**실행·관찰**: Play → 마커가 트랙 위를 좌우로 왕복.

> 📌 짚을 점: 마커 이동에 `Update()`를 한 글자도 안 썼다. `EveryUpdate()` 한 줄이 게임 루프. "매 프레임 위상을 계산해 마커를 옮긴다"를 **한 번 선언**.
>
> 📌 `SpinMarker`는 **임시 비계**다. 지금은 START 전에도 마커가 도는데, 5단계에서 "한 판 동안만" 돌도록 이 `EveryUpdate` 블록을 `SpinMarker`에서 `PlayOneGameAsync` 안으로 옮긴다(메서드 정의는 그때 삭제). 핵심 변화는 둘 — 시간 기준을 판 시작 기준으로(`Time.time` → `Time.time - startTime`, 이를 위해 `startTime` 한 줄 신설) · 수명 그릇을 한 판짜리로(`AddTo(this)` → `AddTo(play)`). 지금은 단계별로 눈으로 확인하려고 켜 둔 것.

### 3단계. 존 진입 강조 — 상태머신 (~12분)

> 🔧 **오늘 쓸 부품**
>
> - `Select`: 값을 변환(여기선 위상 → 존).
> - `DistinctUntilChanged`: 직전 값과 **다를 때만** 통과(연속 중복 제거) → "진입하는 순간"만 잡는 상태머신.

마커가 존에 **들어오는 순간**에만 트랙 색을 바꾼다. 매 프레임이 아니라 "바뀌는 순간 한 번". `BindMarker` 메서드를 만들고 `Start()`의 `SpinMarker()` 자리를 `BindMarker()` 호출로 바꾼다(마커 이동은 `BindMarker` 안에서 같이 부른다):

```csharp
private void BindMarker()
{
    // 위상이 매 프레임 흘러도, 존이 바뀌는 순간에만 색을 갱신
    m_Phase
        .Select(MyTimingBarModel.ZoneOf)        // 위상 → 존(None/Good/Perfect)   (1단계 순수 함수 재사용)
        .DistinctUntilChanged()                 // 존이 바뀌는 순간만 통과(매 프레임 X)
        .Subscribe(zone => m_TrackImage.color = ColorFor(zone))  // 그 순간 트랙 색 변경
        .AddTo(this);

    SpinMarker();   // 마커 이동(2단계) — 아직 임시
}
```

```
위상→존:   N N N G G P P G N N  (매 프레임)
Distinct:  N - - G - P - G N -   ← 바뀌는 순간만 → 그때만 색칠
```

존 enum을 색으로 바꾸는 헬퍼:

```csharp
// 존 → 트랙 색 (2단계에서 만든 색 상수 사용)
private static Color ColorFor(Zone zone) =>
    zone switch
    {
        Zone.Perfect => PerfectColor,
        Zone.Good => GoodColor,
        _ => NoneColor,
    };
```

`Start()`는 이제:

```csharp
private void Start()
{
    m_Model = new MyTimingBarModel();
    m_Phase = new ReactiveProperty<float>(0f).AddTo(this);
    m_Attempt = new ReactiveProperty<int>(0).AddTo(this);
    m_UsableWidth = m_TrackImage.rectTransform.rect.width - m_Marker.rect.width;

    BindMarker();   // 존 색 바인딩 + 마커 이동
}
```

**실행·관찰**: 마커가 가운데 띠(존)를 지날 때만 트랙 전체가 파랑→금색으로 깜빡이고, 벗어나면 어둡게 돌아온다. 매 프레임이 아니라 **경계를 넘는 순간**에만 색이 바뀐다.

> 📌 짚을 점: `DistinctUntilChanged`가 없으면 매 프레임 같은 색을 다시 칠한다(낭비·깜빡임). "존이 바뀌는 순간"이라는 **이벤트**를 위상 스트림에서 뽑아내는 게 `Select`+`DistinctUntilChanged` 상태머신. 명령형이면 "이전 존 변수 + 매 프레임 비교"가 필요했을 자리.
>
> 📌 **중간 Play 체크포인트(1차시 끝)**: 마커 왕복 + 존 진입 색 변화 두 개가 되면 1차시 합격. `Unity_GetConsoleLogs`로 에러 0 확인 후 마무리.

## 2차시 — 4~6단계

### 4단계. 누른 순간 판정 — 입력 합성 + 샘플링 (~18분, 최고 난도)

> 🔧 **오늘 쓸 부품**
>
> - `Merge`: 여러 스트림을 한 줄로 합침(HIT 버튼 + Space 키 → 하나의 "입력").
> - `WithLatestFrom`: **주도 스트림이 발화하는 순간** 다른 스트림의 **최신값을 샘플**(누른 순간의 마커 위상).
> - `Keyboard.current.spaceKey.wasPressedThisFrame`: Input System으로 Space "눌린 순간" 감지(레거시 Input 금지).

HIT 버튼과 Space를 하나로 합쳐, **누른 순간의 위상**을 Model에 넘겨 판정한다. 입력 합성 메서드부터:

```csharp
// HIT 버튼 클릭과 Space 키를 하나의 입력 스트림으로 (둘 중 무엇이든 "쳤다")
private Observable<Unit> HitStream() =>
    m_HitButton
        .OnClickAsObservable()                  // 버튼 클릭 스트림
        .Merge(                                  // ↔ 합치기
            Observable
                .EveryUpdate()                  // 매 프레임 확인해서…
                .Where(_ => Keyboard.current != null
                    && Keyboard.current.spaceKey.wasPressedThisFrame)  // …Space가 "이번 프레임" 눌렸으면 통과
        );
```

```
HIT:    --c-------c----->
Space:  ------s--------->
Merge:  --c---s---c----->   (둘을 한 입력 스트림으로)
```

판정 표시 헬퍼(점수·콤보는 Model이 갱신, 화면 라벨만 여기서):

```csharp
// 판정 결과를 잠깐 띄움 (Perfect!/Good/Miss + 색)
private void FlashJudgement(Judgement judgement)
{
    (string label, Color color) = judgement switch
    {
        Judgement.Perfect => ("Perfect!", PerfectColor),
        Judgement.Good => ("Good", GoodColor),
        _ => ("Miss", MissColor),
    };
    m_JudgementText.text = label;
    m_JudgementText.color = color;
}
```

이제 **누른 순간 판정**을 임시로 붙여 테스트(역시 5단계에서 한 판 안으로 옮긴다). `Start()` 끝에 임시 한 블록:

```csharp
// 임시(4단계 확인용): 누를 때마다 그 순간 위상으로 판정 — 5단계에서 라운드 안으로 이동
HitStream()
    .WithLatestFrom(m_Phase, (_, phase) => phase)  // 누른 '순간'의 마커 위상을 샘플(주도=입력, 참조=위상)
    .Subscribe(phase => FlashJudgement(m_Model.ApplyHit(phase)))  // 그 위상으로 판정·점수 반영
    .AddTo(this);
```

```
입력:    ----x----------x---->   (주도: 누른 순간)
위상:    .4 .5 .6 .5 .4 .5 .6    (참조: 매 프레임 흐름)
샘플:    ----.5---------.5--->   누른 순간의 위상만 뽑힘 → ApplyHit
```

점수·콤보 텍스트도 지금 바인딩(`BindView`). `Start()`에서 `BindView()` 호출 추가:

```csharp
private void BindView()
{
    // 점수가 바뀌면 점수 텍스트와 일치 (출처가 무엇이든)
    m_Model.Score.Subscribe(score => m_ScoreText.text = $"점수 {score:N0}").AddTo(this);
    // 콤보는 2 이상일 때만 표시
    m_Model.Combo
        .Subscribe(combo => m_ComboText.text = combo >= 2 ? $"{combo} 콤보!" : string.Empty)
        .AddTo(this);
    // 진행도 n / 12
    m_Attempt.Subscribe(n => m_AttemptText.text = $"{n} / {m_AttemptsPerRound}").AddTo(this);
}
```

**실행·관찰**: 마커가 존에 있을 때 HIT(또는 Space)를 누르면 "Perfect!/Good"이 뜨고 점수·콤보가 오른다. 존 밖에서 누르면 "Miss"·콤보 0. (아직 START·라운드는 없음 — 항상 칠 수 있고 진행도는 0/12 그대로.)

> ⚠️ **함정**: `WithLatestFrom`의 인자 순서. **주도 스트림이 앞**(`HitStream()`), 참조가 뒤(`m_Phase`). "누른 순간에 위상을 본다"는 의미와 일치. `CombineLatest`(양쪽 변화 모두 반응)와 헷갈리지 말 것 — 여기선 입력이 칠 때만, 위상은 참조만.
>
> ⚠️ **함정**: Space는 반드시 `wasPressedThisFrame`(눌린 순간 1회). `spaceKey.isPressed`로 하면 누르고 있는 동안 매 프레임 판정돼 점수 폭발.

### 5단계. 라운드 흐름 — 비동기 + 시퀀스 종료 (~20분)

> 🔧 **오늘 쓸 부품**
>
> - `SubscribeAwait(..., AwaitOperation.Drop)`: 값마다 비동기 작업 실행, **진행 중이면 새 값 무시**(START 연타 방지).
> - `UniTask.Delay`: 비동기 대기(카운트다운 1초씩). 단발 비동기는 UniTask 표준.
> - `Take(n)`: n개만 받고 **완료(OnCompleted)** → 라운드 자동 종료.
> - `Share`: 한 원본을 여러 구독이 공유(처리 + 완료대기가 같은 스트림 보게).
> - `WaitAsync`: 스트림 **완료를 await**.
> - 라운드용 `CompositeDisposable`: 한 판 동안만 사는 구독 그릇(판 끝나면 통째 정리).

흩어진 임시 조각(마커 이동·입력 판정)을 **한 판 흐름**으로 묶는다: START → 카운트다운 → N회 → 결과. 이 흐름이 capstone의 핵심.

먼저 카운트다운:

```csharp
private async ValueTask CountdownAsync(CancellationToken ct)
{
    for (int i = 3; i >= 1; i--)
    {
        m_StateText.text = i.ToString();
        await UniTask.Delay(TimeSpan.FromSeconds(1), cancellationToken: ct);  // 1초 대기(취소 가능)
    }
    m_StateText.text = "시작!";
}
```

한 판 전체(여기로 마커 이동·입력 판정이 **이사 온다**):

```csharp
private async ValueTask PlayOneGameAsync(CancellationToken ct)
{
    m_Model.ResetForNewGame();              // 점수·콤보 0 (1단계 Model)
    m_Attempt.Value = 0;
    m_StartButton.interactable = false;     // 진행 중 START 잠금
    m_JudgementText.text = string.Empty;

    // ★ 한 판 동안만 사는 구독 그릇. using 이라 메서드 끝나면 자동 Dispose(판이 끝나면 정리)
    using var play = new CompositeDisposable();

    // [이사 1] 마커 이동 — 판 시작 기준 경과시간으로 계산해 매 판 왼쪽(0)부터 출발
    float startTime = Time.time;
    Observable
        .EveryUpdate()
        .Subscribe(_ =>
        {
            float phase = Mathf.PingPong((Time.time - startTime) * m_MarkerSpeed, 1f);
            m_Phase.Value = phase;
            m_Marker.anchoredPosition = new Vector2((phase - 0.5f) * m_UsableWidth, 0f);
        })
        .AddTo(play);                       // this 가 아니라 play — 판 끝나면 정리

    await CountdownAsync(ct);               // 카운트다운 동안 마커가 움직여 리듬 워밍업

    m_StateText.text = "존에 올 때 Space 또는 HIT!";

    // [이사 2] 라운드 입력: 누른 순간 위상 샘플 → 12회만 받고 완료
    Observable<float> roundHits = HitStream()
        .WithLatestFrom(m_Phase, (_, phase) => phase)  // 누른 순간 위상
        .Take(m_AttemptsPerRound)                      // 12회 받으면 OnCompleted (라운드 종료)
        .Share();                                      // 처리 구독 + 완료대기가 같은 스트림 공유

    roundHits
        .Subscribe(phase =>
        {
            m_Attempt.Value += 1;                      // 진행도 +1
            FlashJudgement(m_Model.ApplyHit(phase));   // 판정·점수 반영
        })
        .AddTo(play);

    await roundHits.WaitAsync(ct);          // 12회가 끝나(완료) 여기로

    m_StateText.text = $"결과 — 점수 {m_Model.Score.Value:N0} · 최고 콤보 {m_Model.MaxCombo.Value}";
    m_StartButton.interactable = true;      // START 다시 활성
    ParkMarker();                           // 마커를 왼쪽 끝에 세움(play 구독은 메서드 반환 시 자동 해제)
}
```

```
START 누름 → SubscribeAwait(Drop)
  카운트다운: 3 → 2 → 1 → 시작!
  입력:  x x x x x x x x x x x x   (12개)
  Take12:──────────────────────|  12번째 뒤 완료 → WaitAsync 다음 줄
  결과 표시 → START 다시 켜짐
(진행 중 START 또 눌러도 → Drop 으로 무시)
```

마커를 왼쪽 끝(존 밖)에 세우는 헬퍼(대기·결과 화면용):

```csharp
private void ParkMarker()
{
    m_Phase.Value = 0f;   // 위상 0 → ZoneOf(0)=None → 트랙 색도 어둡게(BindMarker가 반응)
    m_Marker.anchoredPosition = new Vector2((0f - 0.5f) * m_UsableWidth, 0f);
}
```

START를 한 판에 연결:

```csharp
private void BindGameLoop()
{
    m_StartButton
        .OnClickAsObservable()
        // 누를 때마다 한 판 실행. 진행 중 재클릭은 Drop(무시). ct는 구독 해제 시 자동 취소
        .SubscribeAwait((_, ct) => PlayOneGameAsync(ct), AwaitOperation.Drop)
        .AddTo(this);
}
```

이제 **임시 조각을 치운다**:

1. `SpinMarker()` 메서드와 `BindMarker()` 안의 `SpinMarker()` 호출 삭제(마커 이동은 `PlayOneGameAsync`로 이사함).
2. `Start()`의 임시 `HitStream()...Subscribe` 블록 삭제(라운드로 이사함).
3. `BindMarker()` 끝에 `ParkMarker();` 추가(시작 화면에서 마커를 왼쪽에 세움).

정리된 `BindMarker`·`Start`:

```csharp
private void BindMarker()
{
    m_Phase
        .Select(MyTimingBarModel.ZoneOf)
        .DistinctUntilChanged()
        .Subscribe(zone => m_TrackImage.color = ColorFor(zone))
        .AddTo(this);
    ParkMarker();   // 시작 화면: 마커는 왼쪽 끝(존 밖)에 정지
}

private void Start()
{
    m_Model = new MyTimingBarModel();
    m_Phase = new ReactiveProperty<float>(0f).AddTo(this);
    m_Attempt = new ReactiveProperty<int>(0).AddTo(this);
    m_UsableWidth = m_TrackImage.rectTransform.rect.width - m_Marker.rect.width;

    BindMarker();    // 존 색 바인딩 + 마커 정지
    BindView();      // 점수·콤보·진행도 텍스트
    BindGameLoop();  // START → 한 판
}
```

**실행·관찰**: START → 3·2·1 → 마커가 왼쪽부터 왕복 → 12회 치면 결과(점수·최고콤보) → 마커 정지 → START로 다시 시작. 진행 중 START를 또 눌러도 무시된다.

> ❗ **왜 `play` 그릇이 필요한가**: `PlayOneGameAsync` 안의 `EveryUpdate`(마커)는 **무한 스트림**이라 판이 끝나도 안 멈춘다. `play`에 담고 `using`으로 묶었으니 메서드가 끝나면 자동 정리 → 다음 판에서 마커 루프가 겹쳐 쌓이지 않음. **이게 "라운드용 CompositeDisposable"의 실전.** 영속 구독(존 색·텍스트·START)만 `AddTo(this)`, 한 판짜리(마커·입력)는 `AddTo(play)`로 구분하는 게 핵심.
>
> ⚠️ **함정 — `Share` 누락**: `roundHits`를 두 번 구독한다(`Subscribe` + `WaitAsync`). `Share`가 빠지면 콜드 스트림이 두 번 실행돼 `HitStream`의 `EveryUpdate`(Space 감시)가 **두 벌** 돈다. "두 곳 이상에서 구독하면 `Share`"의 회수 지점.
>
> ⚠️ **함정 — Drop을 빼면**: START 연타 시 판이 겹쳐 실행돼 마커·점수가 꼬인다. `AwaitOperation.Drop`이 "진행 중엔 새 시작 무시"를 한 단어로.

### 6단계. 마무리 점검 — 수명·누수 0·회고 (~15분)

마지막으로 Model 해제를 잇는다:

```csharp
private void OnDestroy() => m_Model?.Dispose();   // 오브젝트 파괴 시 Model의 m_Disposables 정리
```

> 📌 수명 사슬: `OnDestroy` → `m_Model.Dispose()` → Model 내부 `m_Disposables.Dispose()` → `Score`/`Combo`/`MaxCombo` 정리. Presenter의 구독(`AddTo(this)`)은 오브젝트 파괴 시 R3가 자동 해제, 라운드 구독(`AddTo(play)`)은 판이 끝나면 `using`이 해제. `SubscribeAwait`의 `ct`는 구독 해제 시 자동 취소되어 진행 중 라운드도 안전하게 멈춘다.

#### ObservableTracker로 누수 0 검증

1. `Window > Observable Tracker` 열기(`EnableTracking` 켠 상태로 Play).
2. 학생 씬에서 한 판 플레이 → `MyTimingBarPresenter` 오브젝트 **삭제**(또는 씬 비움).
3. Tracker 목록에서 타이밍 바 관련 구독이 **전부 사라지면 합격**. 남으면 `AddTo` 빠진 곳 — StackTrace로 추적.

> 💡 **고의 누수 실험 추천**: `PlayOneGameAsync`의 마커 `EveryUpdate`에서 `.AddTo(play)`를 지우고 한 판 플레이 후 씬을 나가 보면, 프레임 루프가 Tracker에 **영원히 남음**(마커는 안 보여도 매 프레임 계속 돔). 목격 후 `.AddTo(play)` 복구. 전역 스트림은 반드시 수명 그릇에 담는다는 걸 직접 확인.

#### 회고 — 어디에 무엇을 썼나

빈 표 배포, 5분간 학생이 채움.

| 게임 메카닉            | R3                                            | "선언한 의도"                             |
| ---------------------- | --------------------------------------------- | ----------------------------------------- |
| 마커 왕복              | `EveryUpdate` + `ReactiveProperty<float>`     | 매 프레임 위상을 계산해 마커를 옮겨라     |
| 존 진입 강조           | `Select(ZoneOf)` + `DistinctUntilChanged`     | 존이 바뀌는 순간에만 색을 바꿔라          |
| HIT·Space 합성         | `OnClickAsObservable` + `EveryUpdate` `Merge` | 버튼이든 키든 하나의 입력으로 봐라        |
| 누른 순간 위상 샘플    | `WithLatestFrom`                              | 칠 때마다 그 순간의 마커 위치를 가져와라  |
| 점수·콤보              | Model `ReactiveProperty` + `ApplyHit`         | 규칙은 Model이, 표시는 알아서 따라와라    |
| 라운드 종료            | `Take(n)` + `WaitAsync`                       | 12회 받으면 끝내고 그 끝을 기다려라       |
| 카운트다운→플레이→결과 | `SubscribeAwait(Drop)` + `UniTask.Delay`      | 한 판을 잇고, 진행 중 재시작은 무시해라   |
| 수명(영속/라운드)      | `AddTo(this)` · 라운드 `CompositeDisposable`  | 화면 죽으면 다, 판 끝나면 판것만 정리해라 |

> 마무리 요점: `Update()`·코루틴·상태 플래그 0개로 게임 완성. **클리커가 "쌓기"(Interval/Scan)였다면 타이밍 바는 "맞히기"(EveryUpdate/WithLatestFrom)** — 같은 R3로 정반대 리듬을 만들었다. 1부 부품을 새 방식으로 조합한 게 전부.

### `MyTimingBarModel.cs` (채운 부분)

```csharp
// 위상(0~1)이 어느 구간인지 — 순수 함수
public static Zone ZoneOf(float phase)
{
    float distance = Math.Abs(phase - 0.5f);
    if (distance <= PerfectHalfWidth)
        return Zone.Perfect;
    if (distance <= GoodHalfWidth)
        return Zone.Good;
    return Zone.None;
}

// 위상 → 판정 — 순수 함수
public static Judgement Judge(float phase) =>
    ZoneOf(phase) switch
    {
        Zone.Perfect => Judgement.Perfect,
        Zone.Good => Judgement.Good,
        _ => Judgement.Miss,
    };

// 한 번의 치기 반영 (Miss면 콤보 0, 아니면 콤보·점수 가산)
public Judgement ApplyHit(float phase)
{
    Judgement judgement = Judge(phase);
    if (judgement == Judgement.Miss)
    {
        Combo.Value = 0;
        return judgement;
    }

    Combo.Value += 1;
    if (Combo.Value > MaxCombo.Value)
        MaxCombo.Value = Combo.Value;

    int basePoints = judgement == Judgement.Perfect ? PerfectScore : GoodScore;
    int multiplier = 1 + Combo.Value / ComboPerMultiplier;
    Score.Value += basePoints * multiplier;
    return judgement;
}

// 새 게임 초기화
public void ResetForNewGame()
{
    Score.Value = 0;
    Combo.Value = 0;
    MaxCombo.Value = 0;
}
```

### `MyTimingBarPresenter.cs` (전체 — `[SerializeField]` 11개는 제공됨)

```csharp
// ── 추가한 런타임 필드 ──
private MyTimingBarModel m_Model;
private ReactiveProperty<float> m_Phase;
private ReactiveProperty<int> m_Attempt;
private float m_UsableWidth;

private static readonly Color NoneColor = new(0.2f, 0.2f, 0.25f, 1f);
private static readonly Color GoodColor = new(0.2f, 0.5f, 0.9f, 1f);
private static readonly Color PerfectColor = new(0.95f, 0.75f, 0.15f, 1f);
private static readonly Color MissColor = new(0.9f, 0.3f, 0.3f, 1f);

private void Start()
{
    m_Model = new MyTimingBarModel();
    m_Phase = new ReactiveProperty<float>(0f).AddTo(this);
    m_Attempt = new ReactiveProperty<int>(0).AddTo(this);
    m_UsableWidth = m_TrackImage.rectTransform.rect.width - m_Marker.rect.width;

    BindMarker();
    BindView();
    BindGameLoop();
}

// 존 진입 순간만 트랙 색 변경 + 시작 화면 마커 정지
private void BindMarker()
{
    m_Phase
        .Select(MyTimingBarModel.ZoneOf)
        .DistinctUntilChanged()
        .Subscribe(zone => m_TrackImage.color = ColorFor(zone))
        .AddTo(this);
    ParkMarker();
}

private void ParkMarker()
{
    m_Phase.Value = 0f;
    m_Marker.anchoredPosition = new Vector2((0f - 0.5f) * m_UsableWidth, 0f);
}

// Model 상태 → 텍스트
private void BindView()
{
    m_Model.Score.Subscribe(score => m_ScoreText.text = $"점수 {score:N0}").AddTo(this);
    m_Model
        .Combo.Subscribe(combo => m_ComboText.text = combo >= 2 ? $"{combo} 콤보!" : string.Empty)
        .AddTo(this);
    m_Attempt.Subscribe(n => m_AttemptText.text = $"{n} / {m_AttemptsPerRound}").AddTo(this);
}

// START → 한 판 (진행 중 재클릭 Drop)
private void BindGameLoop()
{
    m_StartButton
        .OnClickAsObservable()
        .SubscribeAwait((_, ct) => PlayOneGameAsync(ct), AwaitOperation.Drop)
        .AddTo(this);
}

private async ValueTask PlayOneGameAsync(CancellationToken ct)
{
    m_Model.ResetForNewGame();
    m_Attempt.Value = 0;
    m_StartButton.interactable = false;
    m_JudgementText.text = string.Empty;

    using var play = new CompositeDisposable();

    float startTime = Time.time;
    Observable
        .EveryUpdate()
        .Subscribe(_ =>
        {
            float phase = Mathf.PingPong((Time.time - startTime) * m_MarkerSpeed, 1f);
            m_Phase.Value = phase;
            m_Marker.anchoredPosition = new Vector2((phase - 0.5f) * m_UsableWidth, 0f);
        })
        .AddTo(play);

    await CountdownAsync(ct);

    m_StateText.text = "존에 올 때 Space 또는 HIT!";

    Observable<float> roundHits = HitStream()
        .WithLatestFrom(m_Phase, (_, phase) => phase)
        .Take(m_AttemptsPerRound)
        .Share();

    roundHits
        .Subscribe(phase =>
        {
            m_Attempt.Value += 1;
            FlashJudgement(m_Model.ApplyHit(phase));
        })
        .AddTo(play);

    await roundHits.WaitAsync(ct);

    m_StateText.text = $"결과 — 점수 {m_Model.Score.Value:N0} · 최고 콤보 {m_Model.MaxCombo.Value}";
    m_StartButton.interactable = true;
    ParkMarker();
}

private async ValueTask CountdownAsync(CancellationToken ct)
{
    for (int i = 3; i >= 1; i--)
    {
        m_StateText.text = i.ToString();
        await UniTask.Delay(TimeSpan.FromSeconds(1), cancellationToken: ct);
    }
    m_StateText.text = "시작!";
}

// HIT 버튼 + Space 를 하나의 입력으로
private Observable<Unit> HitStream() =>
    m_HitButton
        .OnClickAsObservable()
        .Merge(
            Observable
                .EveryUpdate()
                .Where(_ => Keyboard.current != null && Keyboard.current.spaceKey.wasPressedThisFrame)
        );

private void FlashJudgement(Judgement judgement)
{
    (string label, Color color) = judgement switch
    {
        Judgement.Perfect => ("Perfect!", PerfectColor),
        Judgement.Good => ("Good", GoodColor),
        _ => ("Miss", MissColor),
    };
    m_JudgementText.text = label;
    m_JudgementText.color = color;
}

private static Color ColorFor(Zone zone) =>
    zone switch
    {
        Zone.Perfect => PerfectColor,
        Zone.Good => GoodColor,
        _ => NoneColor,
    };

private void OnDestroy() => m_Model?.Dispose();
```

## 💡 강사 팁 · 트러블슈팅

- **단계마다 중간 Play.** B는 구조를 학생이 짜므로 한 번에 몰아 치면 어디서 깨졌는지 못 찾는다. 코드 변경 후 `Unity_GetConsoleLogs`로 에러 확인.
- **컴파일은 Unity가** 함(`dotnet build`/`dotnet test` 아님). 저장하면 Unity가 재컴파일.
- **마커가 안 움직임**: 2단계 `EveryUpdate` 구독을 `Start`(또는 `BindMarker`)에서 실제로 호출했는지, `m_UsableWidth`가 0이 아닌지(트랙·마커 연결 확인) 점검.
- **마커가 트랙 밖으로 삐져나감**: `m_UsableWidth = 트랙폭 - 마커폭` 빼먹음(트랙폭만 쓰면 끝에서 절반이 나감).
- **존 색이 매 프레임 깜빡임/안 바뀜**: `DistinctUntilChanged` 누락(매 프레임 재칠) 또는 `Select(ZoneOf)`가 위상이 아닌 다른 값을 봄.
- **Space가 두 번/연속 판정**: `wasPressedThisFrame` 대신 `isPressed`를 씀. 누른 순간 1회만 잡아야 함.
- **점수가 두 배로 오름**: `roundHits`에 `Share` 누락(두 번 구독) 또는 임시 4단계 블록을 5단계에서 안 지움(판정이 두 곳).
- **START 연타 시 마커·점수 꼬임**: `SubscribeAwait`에 `AwaitOperation.Drop` 누락.
- **씬을 나가도 마커 루프가 계속 돔**: `PlayOneGameAsync`의 마커 `EveryUpdate`에 `.AddTo(play)` 누락 — 6단계 고의 누수 실험과 같은 증상.
- **텍스트가 □(두부)**: `R3Lecture-Student-1-Basics`(Pretendard 폰트)를 `R3Lecture-Student-3-TimingBar`보다 **먼저** 임포트했는지 확인.
- **가장 막히는 곳은 5단계(라운드 흐름)**. `SubscribeAwait`+`Take`+`WaitAsync`+`play` 그릇이 한꺼번에 나오므로 마블·코드를 천천히. 4단계 임시 블록을 5단계에서 이사시키는 흐름을 또렷이.
- 학생 코드 꼬이면 `R3Lecture-Student-3-TimingBar` 재임포트로 초기화(작성 중 코드 백업 안내).

## 예상 Q&A

> ❓ **빈 메서드도 없는데 구조를 어떻게 잡나(클리커보다 막막함)**
> 정답 구조는 하나가 아님. 이 문서는 `BindMarker`/`BindView`/`BindGameLoop` + `PlayOneGameAsync`로 안내했지만, "프레임으로 마커, 입력으로 판정, START로 라운드"라는 **세 덩어리**만 분리되면 메서드 이름·개수는 자유. 처음엔 `Start()`에 다 몰아 넣고 돌린 뒤 메서드로 쪼개도 됨.

> ❓ **마커 위상 계산을 Model에 넣어도 되나?**
> 안 됨(권장하지 않음). 프레임·`Time`·`Mathf.PingPong`·`RectTransform`은 Unity 의존이라 Presenter 소관. Model은 "위상(float) → 판정 → 점수" 순수 규칙만 알아야 화면 없이 단위 테스트가 됨. **위상이라는 좌표(0~1)를 경계선**으로, Presenter가 위상을 만들고 Model이 위상을 해석한다.

> ❓ **콤보를 Model에 두는 게 맞나, `Scan`으로 풀까?**
> 둘 다 정답. 이 풀이는 콤보 누적을 **Model이 단일 소유**(`ApplyHit` 안에서)해 테스트가 쉬움. 콤보를 "입력 리듬 해석"으로 보고 Presenter에서 `Scan`으로 접는 풀이도 좋은 토론거리(클리커 콤보가 그 방식).

> ❓ **왜 `WithLatestFrom`이지 `CombineLatest`가 아닌가?** > `CombineLatest`는 **양쪽이 바뀔 때마다** 발화 → 마커가 움직일 때마다 판정돼 버림. `WithLatestFrom`은 **주도 스트림(입력)이 칠 때만** 발화하고 위상은 그 순간 값만 참조 → "누른 순간의 마커 위치"에 정확히 맞음.

> ❓ **라운드 종료를 `Take(n)`과 제한시간 둘 다로?**
> 입문 난도를 위해 정본은 `Take(n)` 하나. 제한시간은 `TakeUntil(Observable.Timer(...))`로 분리 가능하고, "먼저 끝나는 쪽"으로 합치려면 추가 합성이 필요해 난도가 오름 → 도전 과제.

> ❓ **`play`(라운드 CompositeDisposable)와 `AddTo(this)`를 언제 구분하나?**
> 수명이 기준. 화면이 살아 있는 내내 필요한 구독(존 색·텍스트·START 버튼) = `AddTo(this)`. 한 판 동안만 사는 구독(마커 루프·라운드 입력) = `AddTo(play)`. 판이 끝나면 `play`만 정리돼 다음 판이 깨끗이 시작.

> ❓ **`SubscribeAwait`의 `ct`는 어디서 오나?**
> R3가 준다. 구독이 해제(예: `OnDestroy`로 Presenter 파괴)되면 그 `ct`가 자동 취소 → 진행 중이던 `UniTask.Delay`·`WaitAsync`도 취소돼 멈춘다. 그래서 라운드 도중 씬을 나가도 누수·예외 없이 정리됨.

## 이해도 퀴즈

1. 마커 위치(픽셀)와 판정(존)을 어긋나지 않게 묶는 **하나의 좌표**는 무엇이고, 그게 어떤 타입에 담기는가?
2. `Select(ZoneOf)` 뒤에 `DistinctUntilChanged`를 붙이는 이유는?
3. 입력 스트림에 `WithLatestFrom(m_Phase)`를 거는 이유를 `CombineLatest`와 비교해 쓰시오.
4. `PlayOneGameAsync` 안의 마커 `EveryUpdate`를 `AddTo(this)`가 아니라 `AddTo(play)`에 담는 이유는?
5. 라운드가 12회 만에 끝나는 것은 어떤 연산자가, 그 끝을 기다리는 것은 무엇이 담당하는가?

> ✅ **해설**
>
> 1. **위상(0~1)**. `ReactiveProperty<float> m_Phase`에 담겨, 마커 위치 계산과 입력 판정 샘플이 같은 값을 본다(desync 방지).
> 2. 위상은 매 프레임 흐르지만 트랙 색은 **존이 바뀌는 순간**에만 바꾸면 됨. `DistinctUntilChanged`가 연속 중복을 제거해 진입 순간 한 번만 통과시킨다.
> 3. `WithLatestFrom`은 **입력이 칠 때만** 발화하고 위상은 그 순간 최신값만 샘플. `CombineLatest`면 마커가 움직일 때마다도 발화해 매 프레임 판정돼 버린다.
> 4. 마커 `EveryUpdate`는 **무한 스트림**이라 판이 끝나도 안 멈춘다. `play`(한 판 그릇)에 담아 메서드 반환 시 정리해야 다음 판에 루프가 겹치지 않고 누수도 없다.
> 5. 종료는 `Take(12)`(12개 받고 OnCompleted), 그 완료를 기다리는 건 `await roundHits.WaitAsync(ct)`.

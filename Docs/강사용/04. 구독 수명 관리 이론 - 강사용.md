## ✅ 수업 전 체크리스트

- [ ] **균형 잡힌 화법** — "AddTo만 붙이면 누수 끝"이 아님. `-=` 누락이 `AddTo` 누락으로 모양만 바뀐 것. 그래도 가치 있는 이유: ①체인 맨 끝 한 줄이라 리뷰에 띔 ②전 코드베이스 패턴 동일 ③Tracker로 검증 가능.
- [ ] **Tracker 시연 미리 연습** — 핵심 장면은 "누수 만들기 → Tracker에 잡힘 → 프로브 파괴 → AddTo된 것만 사라짐"의 대비. 데모 씬 버튼 순서 손에 익히고 `Window > Observable Tracker` 윈도우도 한 번 열어 둘 것. (대본은 [따라하기 강사용](04.%20구독%20수명%20관리%20따라하기%20-%20강사용.md))
- [ ] **CompositeDisposable 위치** — MonoBehaviour는 AddTo(this)로 충분한 경우가 많음. 꼭 필요한 곳: ①순수 C# 클래스(2부 ClickerModel — 예고) ②라운드마다 구독을 갈아 끼우는 경우(`Clear`).
- [ ] **발행기도 Dispose 대상** — Subject/ReactiveProperty 소유자가 OnDestroy/Dispose에서 정리. 구독자 정리와 발행기 정리는 별개 책임.

## 0. 도입 (~5분)

- 1장 사고 코드(`EveryUpdate` 구독에 AddTo 없음) 띄움 → EnemyHud 파괴 후 구독이 어떻게 되는지 학생이 답하게(영원히 산다).
- 오늘 목표: 누수를 **구조적으로** 막고, **눈으로 확인**하는 법.

**이 차시의 한 문장** — "구독은 IDisposable이고, **Dispose를 잊으면 새는 것은 리액티브도 마찬가지**다. 다만 잊기 어렵게 만드는 표준 도구(AddTo·CompositeDisposable·Tracker)가 있다."

학습 목표: ①"구독 누수"의 정체와 증상 설명 ②AddTo(this)·CompositeDisposable로 수명을 묶음 ③ObservableTracker로 누수를 추적 ④발행기도 Dispose 대상임을 구분.

## 1장. 왜 수명 관리가 1순위인가 (~5분)

- C# `event` 고질병("`OnDestroy`에서 `-=` 누락 시 누수·`MissingReferenceException`") 1차시에서 회수.
- 리액티브도 구독 해제 안 하면 똑같이 샘. 스트림 만들기가 쉬워진 만큼 구독도 많아져 더 위험.
- 사고 코드 같이 읽기:

```csharp
public class EnemyHud : MonoBehaviour
{
    private void Start()
    {
        // 전역 무한 스트림을 구독만 하고 어떤 수명에도 묶지 않았다
        Observable.EveryUpdate()
            .Subscribe(_ => m_DistanceText.text = GetDistance());   // ← 이 오브젝트가 파괴된 뒤에도 영원히 돈다
    }
}
```

- 핵심: `Observable.EveryUpdate()`는 씬과 무관한 전역 스트림. `EnemyHud` 파괴돼도 **구독은 살아 있음.** 콜백이 파괴된 `m_DistanceText` 만지는 순간 `MissingReferenceException`. 예외 안 나도 콜백·캡처 참조가 영원히 메모리에 잔존.

> ❗ 옵저버 패턴의 `-=` 누락 실수가 리액티브에서 사라지는 게 아님. **`Dispose` 누락으로 모양만 바뀜.** 리액티브가 주는 것은 "빠뜨리기 어렵게 만드는 표준 도구" — 이 차시 주제.

## 2장. 구독 = IDisposable (~5분)

- 2차시 Interval 토글(수동 Dispose) 회수 — 이미 한 번 해 봄.
- `Subscribe` 반환값이 구독 그 자체:

```csharp
IDisposable subscription = Observable.Interval(TimeSpan.FromSeconds(1))
    .Subscribe(_ => Debug.Log("tick"));

subscription.Dispose();   // 구독 해제 — tick이 멈춘다. 이것이 -=의 표준화
```

- 칠판: `+= → Subscribe · -= → Dispose`. 모든 수명 도구는 이 한 줄 위에 있음.
- 수동 `Dispose`는 "토글로 켜고 끄는 기능"처럼 **해제 시점을 코드가 직접 정하는** 경우에 씀.

## 3장. AddTo(this) — GameObject 수명에 묶기 (~10분, 메인)

- 대부분의 구독은 "이 컴포넌트가 살아 있는 동안"이 곧 수명. 그걸 한 줄로 선언하는 게 `AddTo`.

```csharp
using R3;
using UnityEngine;

public class ScoreViewer : MonoBehaviour
{
    [SerializeField] private PlayerStat m_Stat;

    private void Start()
    {
        m_Stat.OnScoreChanged
            .Where(x => x % 10 == 0)
            .Subscribe(x => Debug.Log($"점수 : {x}"))
            .AddTo(this);            // 이 컴포넌트가 파괴되면 자동 Dispose
    }
    // OnDestroy에 해제 코드가 한 줄도 없다!
}
```

- ScoreViewer 예제로 **`OnDestroy`에 해제 코드 한 줄도 없음** 강조.
- 1장 사고 코드에 `.AddTo(this)` 한 줄 더해 "수리 완료" 시연.
- **⚠️ 함정**: "AddTo 만능론" — 즉시 교정.

> ⚠️ **`AddTo`는 만능 면죄부가 아님** — **빠뜨리면** 똑같이 누수. `-=` 누락이 `AddTo` 누락으로 바뀐 것뿐. 다만 `AddTo`는 **체인 맨 끝 한 줄**이라 리뷰에 띄고 습관화 쉬움. 팀 규칙: **"`Subscribe`를 보면 그 끝에서 `AddTo`(또는 명시적 보관)를 찾는다."**

- 전달 요점: 코드 리뷰에서 `Subscribe`를 보면 끝에 AddTo 또는 명시적 보관이 있는지 확인.

## 4장. CompositeDisposable — 여러 구독을 한꺼번에 (~5분)

- 구독이 여러 개면 하나하나 변수로 들지 말고 그릇에 담아 한 번에 정리.

```csharp
using R3;
using UnityEngine;

public class BattleHud : MonoBehaviour
{
    private readonly CompositeDisposable m_Disposables = new();

    private void Start()
    {
        hp.Subscribe(UpdateHpBar).AddTo(m_Disposables);       // 그릇에 담기
        mp.Subscribe(UpdateMpBar).AddTo(m_Disposables);
        gold.Subscribe(UpdateGoldText).AddTo(m_Disposables);
    }

    private void OnDestroy()
    {
        m_Disposables.Dispose();   // 전부 한 번에 해제
    }
}
```

- 그릇 비유. `Clear` vs `Dispose` 차이 또렷이:
    - `Clear()` — 담긴 구독 전부 해제하되 그릇은 재사용(예: 라운드마다 구독 갈아 끼울 때).
    - `Dispose()` — 전부 해제하고 그릇도 폐기. 이후 `AddTo`되는 구독은 즉시 해제됨.
- MonoBehaviour 아닌 **순수 C# 클래스**(2부 `ClickerModel`)는 `AddTo(this)` 불가 → `CompositeDisposable`이 표준. 2부 Model에서 실전.

> 💡 성능 민감 코드는 `Disposable.CreateBuilder()`로 모아 `RegisterTo(destroyCancellationToken)`에 거는 저할당 패턴도 있음(R3 README). 동작 동일, 기본은 `AddTo`/`CompositeDisposable`로 충분.

## 5장. ObservableTracker — 누수를 눈으로 (~5분, 이론 토대)

- "안 새고 있겠지"를 **확인**하는 도구가 R3 내장.

```csharp
ObservableTracker.EnableTracking = true;     // 추적 시작 (기본 false)
ObservableTracker.EnableStackTrace = true;   // 구독이 생성된 위치(스택)까지 기록

// 현재 살아 있는 구독을 전부 순회
ObservableTracker.ForEachActiveTask(state =>
{
    Debug.Log($"#{state.TrackingId} {state.FormattedType}");
});
```

- 에디터에서는 **`Window > Observable Tracker`** 윈도우로 같은 내용 GUI 표시 — 씬 나갔는데 안 사라지는 구독 = 누수.
- 누수 점검 루틴: ① Tracker 켬 → ② 의심 씬 출입 → ③ 활성 구독 목록에 그 씬 구독 남았는지 확인 → ④ 남았으면 StackTrace로 생성 지점 찾아 `AddTo`.
- **⚠️ 함정**: 추적 상시 켜기 금지 — 오버헤드 있어 개발·디버그 전용. 출시 빌드에서는 끔.
- (시연 대본은 [따라하기 강사용](04.%20구독%20수명%20관리%20따라하기%20-%20강사용.md).)

## 정리 (~5분)

- 누수 패턴 도감 표 훑고 점검 루틴(켜기→씬 출입→목록→StackTrace) 복기.

| 패턴                                                      | 증상                           | 처방                                                            |
| --------------------------------------------------------- | ------------------------------ | --------------------------------------------------------------- |
| 전역 스트림(`EveryUpdate`/`Interval`) 구독에 `AddTo` 누락 | 오브젝트 파괴 후에도 콜백이 돎 | `.AddTo(this)` — 전역 스트림은 **반드시**                       |
| static/싱글톤의 Observable을 씬 객체가 구독               | 씬 전환 후에도 구독 잔존       | 구독자 쪽에서 `AddTo(this)` (발행자 수명 ≠ 내 수명)             |
| 순수 C# 클래스가 구독을 보관만 하고 Dispose 안 함         | 소유자 파괴 후 잔존            | `CompositeDisposable` + 소유자의 `OnDestroy`/`Dispose`에서 정리 |
| `Subject`/`ReactiveProperty` 발행기 방치                  | 구독자 정리돼도 발행기는 잔존  | 발행기도 `IDisposable` — 소유자가 `Dispose`                     |

## 💡 강사 팁 · 예상 Q&A

> ❓ **AddTo(this)는 어떻게 파괴를 아나?**
> 컴포넌트/GameObject 파괴 시점에 Dispose 호출되도록 R3.Unity가 연결(내부적으로 destroyCancellationToken 등 유니티 수명 신호 활용). 구현 디테일보다 "파괴 시점에 자동 Dispose" 계약이 중요.

> ❓ **씬이 통째로 내려가면 어차피 다 정리되는 것 아닌가?**
> 씬 오브젝트 파괴돼도 **구독 자체는 별개**. 전역 스트림(EveryUpdate·Interval)이나 static 발행자에 걸린 구독은 씬과 무관하게 살아남고, 파괴된 오브젝트 만지는 순간 예외. "발행자 수명 ≠ 내 수명"이면 반드시 묶을 것.

> ❓ **DisposableBag/CreateBuilder는 뭔가?**
> CompositeDisposable의 저할당 변형(R3 README). 동작 개념 동일, 기본기는 AddTo/CompositeDisposable로 충분, 성능 민감 코드에서 찾아 씀.

> ❓ **`OnDestroy`에서 일일이 `-=` 하던 정리는 R3에서 어떻게 바뀌나?** (전환형)
> 구독마다 `.AddTo(this)` 붙이면 `OnDestroy` 시점에 자동 Dispose — `-=` 보일러플레이트 사라짐. 구독 여럿이면 `CompositeDisposable`에 모아 한 번에 Dispose. "수동 -= → AddTo/CompositeDisposable"가 이 차시 핵심 전환. ('기존 방식 → R3' 매핑표는 [00. 강사 사전 학습 가이드](00.%20강사%20사전%20학습%20가이드.md))

## 이해도 퀴즈

1. `Observable.EveryUpdate().Subscribe(...)`에 AddTo가 없을 때 일어나는 일을 두 가지 쓰시오.
2. `CompositeDisposable`의 `Clear()`와 `Dispose()`의 차이를 쓰시오.
3. 순수 C# 클래스(MonoBehaviour 아님)에서 구독을 정리하는 표준 패턴을 코드 골격으로 쓰시오.
4. ObservableTracker로 누수를 찾는 4단계 루틴을 쓰시오.

> ✅ **해설**
>
> 1. ① 구독이 영원히 살아 메모리(콜백+캡처 참조) 누수 ② 콜백이 파괴된 오브젝트를 만지면 `MissingReferenceException`.
> 2. Clear는 담긴 구독만 해제하고 그릇은 재사용 가능. Dispose는 전부 해제 후 그릇도 폐기(이후 AddTo되는 구독은 즉시 해제됨).
> 3. `CompositeDisposable` 필드에 `.AddTo(m_Disposables)`로 모으고, 클래스의 `Dispose()`에서 `m_Disposables.Dispose()`. 소유자(예: Presenter의 OnDestroy)가 Dispose를 호출.
> 4. ①EnableTracking 켜기 ②의심 씬 출입 ③활성 구독 목록 확인 ④남은 구독의 StackTrace로 생성 지점을 찾아 AddTo 보강.

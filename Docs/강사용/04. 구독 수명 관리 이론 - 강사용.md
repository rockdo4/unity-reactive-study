## ✅ 수업 전 체크리스트

- [ ] **정직하게 가르칠 것** — "AddTo만 붙이면 누수 끝"이 아님. `-=`를 빠뜨리는 실수가 `AddTo`를 빠뜨리는 실수로 바뀐 것뿐. 그래도 가치가 있는 이유: ①체인 맨 끝 한 줄이라 리뷰에서 눈에 띔 ②전 코드베이스에서 패턴이 같음 ③Tracker로 검증 가능함. 이 균형 잡힌 화법이 신뢰를 만듦.
- [ ] **Tracker 시연을 미리 연습** — 수업 효과가 가장 큰 장면은 "누수 만들기 → Tracker에 잡힘 → 프로브 파괴 → AddTo된 것만 사라짐"의 대비임. 데모 씬 버튼 순서를 손에 익히고, `Window > Observable Tracker` 윈도우도 한 번 열어 둘 것. (구체 대본은 [따라하기 강사용](04.%20구독%20수명%20관리%20따라하기%20-%20강사용.md))
- [ ] **CompositeDisposable의 위치** — MonoBehaviour에서는 AddTo(this)로 충분한 경우가 많음. CompositeDisposable이 꼭 필요한 곳은 ①순수 C# 클래스(2부 ClickerModel — 예고) ②구독을 라운드마다 갈아 끼우는 경우(`Clear`).
- [ ] **발행기도 Dispose 대상** — Subject/ReactiveProperty를 들고 있는 쪽(소유자)이 OnDestroy/Dispose에서 정리. 구독자 정리와 발행기 정리는 별개 책임임을 구분해 말할 것.

## 0. 도입 (~5분)

> 💡 1장의 사고 코드(`EveryUpdate` 구독에 AddTo 없음)를 띄우고 — "이 EnemyHud가 파괴되면 이 구독은 어떻게 될까요?" 답(영원히 산다)을 학생이 말하게 유도. "오늘은 이걸 **구조적으로** 막고, **눈으로 확인**하는 법을 배웁니다."

**이 차시의 한 문장** — "구독은 IDisposable이고, **Dispose를 잊으면 새는 것은 리액티브도 마찬가지**다. 다만 잊기 어렵게 만드는 표준 도구(AddTo·CompositeDisposable·Tracker)가 있다."

학습 목표: ①"구독 누수"의 정체와 증상 설명 ②AddTo(this)·CompositeDisposable로 수명을 묶음 ③ObservableTracker로 누수를 추적 ④발행기도 Dispose 대상임을 구분.

## 1장. 왜 수명 관리가 1순위인가 (~5분)

- C# `event`의 고질병("`OnDestroy`에서 `-=` 빠뜨리면 누수와 `MissingReferenceException`")을 1차시에서 회수.
- "리액티브도 **구독을 해제하지 않으면 똑같이** 샌다. 오히려 스트림 만들기가 쉬워진 만큼 구독도 많아져서 더 위험하다"를 못 박기.
- 사고 코드를 같이 읽기:

```csharp
public class EnemyHud : MonoBehaviour
{
    private void Start()
    {
        // 전역 무한 스트림을 구독만 하고 어떤 수명에도 묶지 않았다
        Observable.EveryUpdate()
            .Subscribe(_ => m_DistanceText.text = GetDistance());   // ← 이 오브젝트가 파괴된 뒤에도 영원히 돈다
    }
}
```

- 핵심: `Observable.EveryUpdate()`는 씬과 무관한 전역 스트림이라, `EnemyHud`가 파괴돼도 **구독은 살아 있음.** 콜백이 파괴된 `m_DistanceText`를 만지는 순간 `MissingReferenceException`. 운 좋게 예외가 안 나도 콜백과 캡처된 참조가 영원히 메모리에 남음.

> ❗ 옵저버 패턴에서 `-=`를 빠뜨리는 실수가 리액티브에서 사라지는 게 아님. **`Dispose`를 빠뜨리는 실수로 모양이 바뀔 뿐.** 리액티브가 주는 것은 "빠뜨리기 어렵게 만드는 표준 도구"들임 — 그게 이 차시의 주제.

## 2장. 구독 = IDisposable (~5분)

- 2차시 Interval 토글(수동 Dispose)을 회수 — "이미 한 번 해 봤다."
- `Subscribe`의 반환값이 구독 그 자체임:

```csharp
IDisposable subscription = Observable.Interval(TimeSpan.FromSeconds(1))
    .Subscribe(_ => Debug.Log("tick"));

subscription.Dispose();   // 구독 해제 — tick이 멈춘다. 이것이 -=의 표준화
```

- 칠판: `+= → Subscribe · -= → Dispose`. "모든 수명 도구는 이 한 줄 위에 있다."
- 수동 `Dispose`는 "토글로 켜고 끄는 기능"처럼 **해제 시점을 코드가 직접 정하는** 경우에 씀.

## 3장. AddTo(this) — GameObject 수명에 묶기 (~10분, 메인)

- "대부분의 구독은 '이 컴포넌트가 살아 있는 동안'이 곧 수명"임. 그걸 한 줄로 선언하는 게 `AddTo`.

```csharp
using R3;
using UnityEngine;

public class ScoreViewer : MonoBehaviour
{
    [SerializeField] private PlayerStat m_Stat;

    private void Start()
    {
        m_Stat.OnScoreChanged
            .Where(x => x % 10 == 0)
            .Subscribe(x => Debug.Log($"점수 : {x}"))
            .AddTo(this);            // 이 컴포넌트가 파괴되면 자동 Dispose
    }
    // OnDestroy에 해제 코드가 한 줄도 없다!
}
```

- ScoreViewer 예제를 칠판/슬라이드로 짚으며 **`OnDestroy`에 해제 코드가 한 줄도 없음**을 강조.
- 1장의 사고 코드에 `.AddTo(this)` 한 줄을 더해 "수리 완료" 보여 주기.
- **⚠️ 함정**: "AddTo 만능론" — 즉시 교정.

> ⚠️ **`AddTo`는 만능 면죄부가 아님** — `AddTo`를 **빠뜨리면** 똑같이 누수임. `-=`를 빠뜨리는 실수가 `AddTo`를 빠뜨리는 실수로 바뀐 것뿐. 다만 `AddTo`는 **체인 맨 끝의 한 줄**이라 코드 리뷰에서 눈에 띄고 습관화하기 쉬움. 팀 규칙으로 삼을 것: **"`Subscribe`를 보면 그 끝에서 `AddTo`(또는 명시적 보관)를 찾는다."**

- **던질 질문**: "팀 코드 리뷰에서 `Subscribe`를 보면 뭘 확인해야 할까요?" (답: 끝에 AddTo 또는 명시적 보관이 있는가)

## 4장. CompositeDisposable — 여러 구독을 한꺼번에 (~5분)

- 구독이 여러 개면 하나하나 변수로 들지 말고 그릇에 담아 한 번에 정리.

```csharp
using R3;
using UnityEngine;

public class BattleHud : MonoBehaviour
{
    private readonly CompositeDisposable m_Disposables = new();

    private void Start()
    {
        hp.Subscribe(UpdateHpBar).AddTo(m_Disposables);       // 그릇에 담기
        mp.Subscribe(UpdateMpBar).AddTo(m_Disposables);
        gold.Subscribe(UpdateGoldText).AddTo(m_Disposables);
    }

    private void OnDestroy()
    {
        m_Disposables.Dispose();   // 전부 한 번에 해제
    }
}
```

- 그릇 비유. `Clear` vs `Dispose` 차이를 또렷이:
    - `Clear()` — 담긴 구독을 전부 해제하되 그릇은 재사용(예: 라운드마다 구독을 갈아 끼울 때).
    - `Dispose()` — 전부 해제하고 그릇도 폐기. 이후 `AddTo`되는 구독은 즉시 해제됨.
- "MonoBehaviour가 아닌 **순수 C# 클래스**(2부의 `ClickerModel`)는 `AddTo(this)`를 쓸 수 없으므로 `CompositeDisposable`이 표준 — 2부 Model에서 실전."

> 💡 성능이 민감한 곳에서는 `Disposable.CreateBuilder()`로 모아 `RegisterTo(destroyCancellationToken)`에 거는 저할당 패턴도 있음(R3 README). 동작은 같으니 기본은 `AddTo`/`CompositeDisposable`로 충분.

## 5장. ObservableTracker — 누수를 눈으로 (~5분, 이론 토대)

- "'안 새고 있겠지'를 **확인**하는 도구가 R3에 내장됨."

```csharp
ObservableTracker.EnableTracking = true;     // 추적 시작 (기본 false)
ObservableTracker.EnableStackTrace = true;   // 구독이 생성된 위치(스택)까지 기록

// 현재 살아 있는 구독을 전부 순회
ObservableTracker.ForEachActiveTask(state =>
{
    Debug.Log($"#{state.TrackingId} {state.FormattedType}");
});
```

- 에디터에서는 **`Window > Observable Tracker`** 윈도우로 같은 내용을 GUI로 — 씬을 나갔는데 사라지지 않는 구독이 있으면 그게 누수.
- 실전 누수 점검 루틴: ① Tracker를 켠다 → ② 의심 씬에 들어갔다 나온다 → ③ 활성 구독 목록에 그 씬의 구독이 남아 있는지 본다 → ④ 남았다면 StackTrace로 생성 지점을 찾아 `AddTo`를 단다.
- **⚠️ 함정**: "추적 켜 두면 안 되나요?" — 오버헤드가 있어 개발·디버그 전용. 출시 빌드에서는 끔.
- (실제 시연 대본은 [따라하기 강사용](04.%20구독%20수명%20관리%20따라하기%20-%20강사용.md)으로 넘어감.)

## 정리 (~5분)

- 누수 패턴 도감 표를 훑고 점검 루틴(켜기→씬 출입→목록→StackTrace)을 복기.

| 패턴                                                      | 증상                           | 처방                                                            |
| --------------------------------------------------------- | ------------------------------ | --------------------------------------------------------------- |
| 전역 스트림(`EveryUpdate`/`Interval`) 구독에 `AddTo` 누락 | 오브젝트 파괴 후에도 콜백이 돎 | `.AddTo(this)` — 전역 스트림은 **반드시**                       |
| static/싱글톤의 Observable을 씬 객체가 구독               | 씬 전환 후에도 구독 잔존       | 구독자 쪽에서 `AddTo(this)` (발행자 수명 ≠ 내 수명)             |
| 순수 C# 클래스가 구독을 보관만 하고 Dispose 안 함         | 소유자 파괴 후 잔존            | `CompositeDisposable` + 소유자의 `OnDestroy`/`Dispose`에서 정리 |
| `Subject`/`ReactiveProperty` 발행기 방치                  | 구독자 정리돼도 발행기는 잔존  | 발행기도 `IDisposable` — 소유자가 `Dispose`                     |

## 💡 강사 팁 · 예상 Q&A

> ❓ **AddTo(this)는 어떻게 파괴를 아나요?**
> 컴포넌트/GameObject의 파괴 시점에 Dispose가 호출되도록 R3.Unity가 연결함(내부적으로 destroyCancellationToken 등 유니티 수명 신호 활용). 구현 디테일보다 "파괴 시점에 자동 Dispose"라는 계약이 중요.

> ❓ **씬이 통째로 내려가면 어차피 다 정리되는 것 아닌가요?**
> 씬 오브젝트가 파괴돼도 **구독 자체는 별개**임. 전역 스트림(EveryUpdate·Interval)이나 static 발행자에 걸린 구독은 씬과 무관하게 살아남고, 파괴된 오브젝트를 만지는 순간 예외. 그래서 "발행자 수명 ≠ 내 수명"이면 반드시 묶어야 함.

> ❓ **DisposableBag/CreateBuilder는 뭔가요?**
> CompositeDisposable의 저할당 변형(R3 README). 동작 개념은 같으니 기본기는 AddTo/CompositeDisposable로 충분하고, 성능 민감 코드에서 찾아 쓰면 된다고 안내.

> ❓ **`OnDestroy`에서 일일이 `-=` 하던 정리는 R3에서 어떻게 바뀌나요?** (전환형)
> 구독마다 `.AddTo(this)`를 붙이면 `OnDestroy` 시점에 자동 Dispose — `-=`를 손으로 적던 보일러플레이트가 사라짐. 구독이 여럿이면 `CompositeDisposable`에 모아 한 번에 Dispose. "수동 -= → AddTo/CompositeDisposable"가 이 차시의 핵심 전환. ('기존 방식 → R3' 한눈 매핑표는 [00. 강사 사전 학습 가이드](00.%20강사%20사전%20학습%20가이드.md))

## 이해도 퀴즈

1. `Observable.EveryUpdate().Subscribe(...)`에 AddTo가 없을 때 일어나는 일을 두 가지 쓰시오.
2. `CompositeDisposable`의 `Clear()`와 `Dispose()`의 차이를 쓰시오.
3. 순수 C# 클래스(MonoBehaviour 아님)에서 구독을 정리하는 표준 패턴을 코드 골격으로 쓰시오.
4. ObservableTracker로 누수를 찾는 4단계 루틴을 쓰시오.

> ✅ **해설**
>
> 1. ① 구독이 영원히 살아 메모리(콜백+캡처 참조) 누수 ② 콜백이 파괴된 오브젝트를 만지면 `MissingReferenceException`.
> 2. Clear는 담긴 구독만 해제하고 그릇은 재사용 가능. Dispose는 전부 해제 후 그릇도 폐기(이후 AddTo되는 구독은 즉시 해제됨).
> 3. `CompositeDisposable` 필드에 `.AddTo(m_Disposables)`로 모으고, 클래스의 `Dispose()`에서 `m_Disposables.Dispose()`. 소유자(예: Presenter의 OnDestroy)가 Dispose를 호출.
> 4. ①EnableTracking 켜기 ②의심 씬 출입 ③활성 구독 목록 확인 ④남은 구독의 StackTrace로 생성 지점을 찾아 AddTo 보강.

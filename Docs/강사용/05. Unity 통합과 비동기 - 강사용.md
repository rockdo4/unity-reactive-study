# 05. Unity 통합과 비동기 — 강사용

> 학생 교안: [05. Unity 통합과 비동기](../교안/05.%20Unity%20통합과%20비동기.md)
> 데모: 기초 데모 씬 패널 4·5 · `Assets/Scripts/ReactiveBasics/BindingDemo.cs`, `AsyncDemo.cs`

## 1. 강사 사전 학습 · 리마인드

- **이 차시는 1부의 수확기** — 새 개념(프로바이더·에러·비동기)이 많아 보이지만, 실제 수업의 무게는 **UI 바인딩 3패턴**(값→표시 / 파생 상태 / 조건→활성화)에 둘 것. 이게 2부에서 그대로 반복됨.
- **프로바이더는 가볍게** — "시간 연산자가 쓰는 시계를 유니티 것으로 자동 교체해 준다. timeScale 무시 등 변형은 필요할 때 찾는다"까지. DestroyCancellationToken·PlayerLoop 같은 내부로 들어가지 말 것.
- **에러 처리의 핵심 문장** — "R3의 기본은 **OnErrorResume: 에러가 나도 구독은 살아 있다**. 전통 Rx처럼 끊고 싶으면 OnErrorResumeAsFailure." 그리고 **R3에 Retry가 없다**는 사실(소스 확인됨)을 정확히 — 재시도는 FromAsync 내 루프나 Catch 조합.
- **SubscribeAwait 표를 자기 언어로** — Sequential(줄 세움)/Drop(버림)/Switch(갈아탐)/Parallel(동시). 각각 게임 용례 하나씩: 큐 처리/저장 버튼/자동완성/다발 다운로드. 패널 5 데모는 Sequential vs Drop 대비만 보여 줌.
- **UniTask 질문 대비** — 거의 반드시 나옴. "한 번 일어나는 일은 await(UniTask), 여러 번 일어나는 일은 Observable(R3). 같은 생태계라 같이 쓴다."(운영 가이드 4장)
- **안티패턴 절은 빼먹지 말 것** — 1부의 마지막 메시지. "장점만 일부 취하기"가 이 강의의 공식 포지션임.

## 2. 수업 진행

### 도입 (~3분)

> 💡 "지금까지 배운 걸로 UI를 만들면 어떤 모양이 되는지 — 그리고 게임에서 피할 수 없는 두 가지, 에러와 비동기를 오늘 정리합니다."

### 1장. R3.Unity (~6분)

- `EveryUpdate` 예제 — **"전역 스트림이므로 AddTo 필수"** 4차시 회수.
- 프로바이더는 "자동으로 유니티 시계" 한 줄 + 변형(`IgnoreTimeScale`, `~Frame` 연산자)은 카탈로그식 소개.

### 2~3장. uGUI 스트림과 바인딩 3패턴 (~12분, 메인)

- `OnClickAsObservable` + ThrottleFirst — "버튼이 스트림이 되는 순간 3차시의 모든 연산자가 공짜".
- 바인딩 3패턴을 칠판에 번호 붙여 정리:
    1. `값.Subscribe(표시)` — 값→표시
    2. `값.Select(가공).DistinctUntilChanged().Subscribe(표시)` — 파생 상태
    3. `a.CombineLatest(b, 조건).Subscribe(버튼.interactable)` — 조건→활성화
- "진짜 바인딩 아님"(공식 표현: 바인딩처럼 동작하는 Observable) 명시.
- MVP 예고 — "값을 가진 쪽과 보여 주는 쪽을 클래스로 나누면 MVP. 다음 차시(2부)에서 직접."

### 4장. 에러 처리 (~7분)

- OnErrorResume 시맨틱(구독 유지)을 게임 맥락(입력 스트림이 예외 한 번에 죽으면 안 됨)으로.
- Catch / OnErrorResumeAsFailure / IgnoreOnErrorResume 한 줄씩. **Retry 없음** + 대안.
- **⚠️ 함정**: 학생이 UniRx의 `Retry`를 검색해 옴 — "R3엔 없다"를 미리 말해 두면 혼란 방지.

### 5장. 비동기 통합 (~7분)

- `FirstAsync`(await 한 번 — 컷씬/튜토리얼), `FromAsync`(Task→스트림), `SubscribeAwait`+표.
- 취소 연쇄: "AddTo → 구독 해제 → ct 취소"가 공짜로 따라온다.

### 따라하기 — 패널 4·5 (~12분)

패널 4 (바인딩):

1. 피해/회복 버튼 연타 — HP 텍스트·상태 라벨(구간 바뀔 때만 색 변경) 관찰.
2. MP를 20 아래로 — 스킬 버튼이 **저절로** 꺼짐. "끄는 코드를 누가 호출했죠?" (아무도 — 선언했을 뿐)
3. `BindingDemo.cs`에서 3패턴에 번호 매겨 가며 확인.

패널 5 (비동기):

1. `순차 로드` 빠르게 4번 — 줄 서서 4건 모두 처리됨(Sequential).
2. `드롭 로드` 빠르게 4번 — 1건만 처리, 나머지 무시(Drop).
3. 로드 중 Play 정지 — 예외 없이 조용히 취소됨(ct 연쇄)을 언급.

### 정리 — 안티패턴 (~5분)

- 5종(pull→push 강제 · 과도한 체인 · 핫/콜드 혼동 · AddTo 누락 · 전면 도입) 복기.
- **이 강의의 포지션 선언**: "옵저버/이벤트를 버리는 게 아니다. 구독 수명 관리 + UI 바인딩 + 시간/입력 합성, 이득이 확실한 곳부터."

## 3. 예상 Q&A

> ❓ **EveryUpdate로 입력을 받으면 Update()보다 느리지 않나요?** (전환형 — `Update` 폴링 → `EveryUpdate`+`Where`)
> 프레임마다 콜백이 도는 건 같음. 차이는 성능이 아니라 구조(관심사 분리·수명 관리). 성능 수치는 단정하지 말 것(가드레일). 이 행이 '기존 방식 → R3' 한눈 매핑표([운영 가이드 1장](00.%20강의%20운영%20가이드.md))의 `Update` 폴링 행임.

> ❓ **OnErrorResume이 기본이면 에러를 놓치지 않나요?**
> 핸들러를 생략해도 전역 unhandled exception 핸들러(R3.Unity 기본: Debug.LogException)로 가므로 조용히 사라지지 않음. 흐름을 끊어야 하는 에러면 OnErrorResumeAsFailure로 격상.

> ❓ **Switch는 언제 쓰나요?**
> "최신 것만 유효"한 작업 — 검색 자동완성(이전 검색 취소), 마지막 클릭한 아이템의 상세 로드 등. 진행 중인 작업이 ct로 취소된다는 점이 핵심.

> ❓ **UI Toolkit도 되나요?**
> R3는 UI 프레임워크 중립이라 콜백을 FromEvent/Subject로 감싸면 동일하게 적용 가능. 이 강의 데모는 uGUI 기준.

## 4. 이해도 퀴즈

1. UI 바인딩 3패턴을 각각 한 줄 코드 골격으로 쓰시오.
2. R3에서 콜백 예외가 나면 구독이 어떻게 되는가? 전통 Rx와 비교해 쓰시오.
3. 저장 버튼 연타로 중복 저장이 일어나지 않게 하는 SubscribeAwait 옵션은?
4. "모든 C# event를 Observable로 바꾸자"는 제안에 어떻게 답해야 하는가?

> ✅ **해설**
>
> 1. ①`hp.Subscribe(x => text.text = ...)` ②`hp.Select(구간).DistinctUntilChanged().Subscribe(...)` ③`hp.CombineLatest(mp, 조건).Subscribe(can => btn.interactable = can)`.
> 2. R3 기본은 OnErrorResume — 에러를 알리되 구독 유지. 전통 Rx는 OnError로 구독 종료. 끊으려면 OnErrorResumeAsFailure.
> 3. `AwaitOperation.Drop` (처리 중 이벤트를 버림).
> 4. 안티패턴 — pull이 자연스러운 것을 push로 강제하지 말라는 Introduction to Rx의 경고를 들어, 이득이 확실한 곳(수명·바인딩·시간/입력 합성)부터 부분 도입을 제안.

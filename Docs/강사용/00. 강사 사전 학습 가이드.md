# 00. 강사 사전 학습 가이드

> 진행 메모. R3·리액티브를 처음 접하는 강사가 수업 전 스스로 준비하는 가이드.
> 차시별 진행은 강사용 01~06, 일정·평가·배포 등 운영은 [00. 수업 개요](../00.%20수업%20개요.md). 이 문서는 "무엇을 어떤 순서로 공부할지"(1장)와 "강사가 정확히 알고 있어야 할 개념·답변"(2~4장)만.
> Unity 기본기·`Update`·`event`·코루틴(`StartCoroutine`/`yield return`) 경험은 전제.

> 📦 배포 모델: 학생은 클론 안 함. 빈 URP 프로젝트에 R3·UniTask·Input System 설치 후 배포용 `.unitypackage` 임포트(1부 통합 데모 + 2부 클리커, 폰트는 1부 패키지 포함). 저장소 클론은 강사 전용. 로드맵의 "데모 실행"=강사가 저장소에서 직접 씬 생성·관찰.

## 1. 준비 로드맵 (수업 전 필수)

> ℹ️ 순서대로. 각 단계가 다음 토대. 수업 1~2주 전 시작 권장.

### 1주 전 — 개념과 어휘

1. **교안 01 정독** — [01. 리액티브 프로그래밍이란](../교안/01.%20리액티브%20프로그래밍이란.md). 학생에게 가르칠 프레임(`Observable = 옵저버 + Operator + IDisposable`)을 먼저 자기 것으로.
2. **ReactiveX 입문** <https://reactivex.io/intro.html> — "Rx는 옵저버 패턴의 확장" 정의의 1차 출처. 부담되면 훑기만(교안이 핵심 옮겨 둠).
3. **R3 README** <https://github.com/Cysharp/R3> — 전체 그림. 처음엔 ①Basic Usage ②Subscription Management(AddTo) ③Unity 세 곳만. 연산자 표는 사전, 암기 대상 아님.

목표: "리액티브가 뭐냐"에 옵저버 패턴부터 3분간 막힘없이 설명 가능한 상태.

    | -------- | ----------------------------------------------------------------------- | --------------------------------------------------- |
    | 1부 기초 | `ReactiveBasics`(§1 생성·§2 연산자·§3 수명·§4 바인딩·§5 비동기, 완성본) | `Tools > Reactive Study > 01. R3 기초 통합 씬 생성` |

    - ② `ReactiveProperty<int>` 하나를 텍스트에 바인딩
    - ③ `Observable.Interval` 구독을 만들었다가 오브젝트 파괴로 자동 해제되는지 Tracker로 확인
      막히면 교안 02~05 재독. **이 3개 못 하면 라이브 코딩 미루고 완성 데모 시연으로 진행.**

7. **교안 06 따라 클리커 처음부터 끝까지 직접 제작** — 학생이 겪을 일 미리 겪는 단계. 학생은 `ReactiveClicker` 스켈레톤에서 시작 → 강사도 스켈레톤(또는 빈 손)에서 완성본(`ReactiveClicker` 씬·`ClickerModel`/`ClickerPresenter`)까지. 오타 나기 쉬운 곳·오래 걸리는 단계 파악. 본인이 막힌 곳=학생이 막힐 곳 → 메모해 선제 대응.

## 2. 강사가 먼저 잡아야 할 멘탈 모델 5개

다섯 항목을 자기 말로 설명 가능한지 확인. 상세 설명·코드는 교안. 여기선 강사가 머릿속에 가질 "그림"만.

1. **리액티브 = 옵저버 패턴의 확장** — 신비화 금지. "아는 옵저버에 연산자와 표준 해제를 더한 것."
2. **이벤트는 시간 위의 데이터 스트림** — 마블 다이어그램이 공용어. 칠판에 가로선+구슬.
3. **Subscribe는 등록, Dispose는 해제** — `+=`/`-=`의 표준화. 모든 수명 관리가 이 위에.
4. **연산자 합성이 핵심** — 더블클릭·쿨다운·콤보가 상태 플래그 없이 한두 줄로.
5. **만병통치약 아님** — pull이 자연스러운 곳에 강제 금지. "장점만 일부 취하기"가 권장 포지션.

### Unity 관용구 → R3 매핑 (한눈)

학생은 이미 `Update`·`event`·코루틴으로 "반응"을 짜 왔음. 0에서 가르치는 대신 "늘 하던 그것이 무엇으로 바뀌는가"로 다리 놓으면 흡수 빠름. 교안 01·02·03이 각 행을 본문 비교 박스(🔁)로 펼침, 차시별 강사용에 배경·예상 Q&A.

| Unity에서 늘 하던 방식              | R3 대응                                     | 다루는 교안 |
| ----------------------------------- | ------------------------------------------- | ----------- |
| `UnityEvent` / `Action` 콜백        | `Observable` + `Subscribe`                  | 01·05       |
| `Update` 폴링(매 프레임 검사)       | `Observable.EveryUpdate()` + `Where`        | 05          |
| 코루틴 `WaitForSeconds` / 조건 대기 | `Timer` · `Delay` · `Debounce`              | 02·03       |
| `Invoke` / `InvokeRepeating`        | `Timer` · `Interval`                        | 02          |
| bool 플래그 + 수동 전파             | `ReactiveProperty`(현재값 보유 · 자동 통지) | 02·05       |
| 수동 `-=` 해제 / `OnDestroy` 정리   | `AddTo` · `CompositeDisposable`             | 04          |

> 💡 왼쪽 열=학생이 이미 아는 것. 1차시 도입에서 "오른쪽 열을 하나씩 채운다" 예고로 활용 가능. 아는 왼쪽에서 출발하는 게 이 과정의 일관된 교수 전략.

> ℹ️ 이 표는 **'기존 방식 → R3'** 축. UniRx 연산자 이름이 R3에서 어떻게 바뀌는지(`Throttle→Debounce`, `Buffer→Chunk` 등)의 **'UniRx → R3' 이름 매핑**은 축이 다름 → [교안 03 부록](../교안/03.%20연산자%20-%20스트림을%20다듬고%20합성하기.md).

## 3. 검증된 사실 가드레일 (학생 앞에서 단정하지 말 것)

> ❗ 아래는 적힌 표현까지만. 더 단정하면 틀릴 수 있음.

- **버전** — R3 수업 시점 최신 버전 기준. 설치 시 <https://github.com/Cysharp/R3/releases>에서 최신 확인(Releases가 정본 체인지로그).
- **지원 범위** — 공식 문서는 "R3는 **Unity 2021.3 이상** 지원"까지만 밝힘. 특정 유니티 버전·IL2CPP 실동작은 본 자료 비보증(이 저장소의 Unity 6.3 에디터 동작은 확인 대상).
- **UniRx 아카이브** — "아카이브(읽기 전용)되었고 README가 R3 사용 권함"까지만. 정확한 일자는 단정 금지.
- **UniRx 존재 이유** — "공식 Rx.NET이 Unity에서 동작하지 않고 iOS IL2CPP(AOT) 호환성 문제가 있어 재구현"이 정확. 막연히 "Mono 문제"라 단정 금지.
- **R3에 `Retry` 없음** — R3 연산자에 `Retry` 없음(소스 확인 기준). 재시도는 `FromAsync` 내 루프 또는 `Catch` 조합으로 안내.
- **"극적 단순화" 과장 금지** — 정확한 표현은 "더 짧고 선언적으로 표현 가능". 디버깅·학습 비용이라는 대가를 항상 함께.
- **"모든 event를 Rx로 대체" 금지** — _Introduction to Rx_ 안티패턴 경고와 함께만.
- **채택률 수치 금지** — 스타 수·점유율 등 정량 주장 대신 "정식 후속작으로 수렴" 같은 정성 표현만.

## 4. 자주 받는 질문 총괄 (차시 무관)

무관 차시에서도 나오는 공통 질문의 기준 답변. 차시별 세부 Q&A는 강사용 01~06.

> ❓ **UniRx 튜토리얼이 훨씬 많던데 그걸로 배워도 되나?**
> 개념은 같으니 참고는 가능. 단 UniRx는 아카이브됨, 연산자 이름 다름(`Throttle→Debounce`, `Buffer→Chunk`). 새 코드는 R3, 옛 자료는 교안 03 부록 매핑표 거쳐 읽도록 안내.

> ❓ **UniTask랑은 무슨 관계?**
> 다른 도구. UniTask=단발 비동기(async/await)의 고성능 구현, R3=**반복되는 이벤트 스트림** 합성. 같은 Cysharp 생태계라 함께 쓰는 일 많고, R3는 await 통합(교안 05)으로 둘을 잇는다. "한 번 일어나는 일은 await, 여러 번 일어나는 일은 Observable"이 기준선.

> ❓ **성능·GC는 괜찮나?**
> R3는 현대적 재설계로 내부 할당 줄임, 구독 추적도 중앙화. 단 수치 비교 단정 금지(가드레일). 게임플레이 코드 일반 사용에서 문제 드묾, 핫패스(매 프레임 수천 회)에는 어떤 추상화든 신중해야 한다는 일반론으로.

> ❓ **MVVM/MVP 아키텍처를 꼭 같이 써야 하나?**
> 아님. R3는 패턴 없이도 사용 가능. 다만 ReactiveProperty가 Model→View 통지를 자연스럽게 만들어 MVP류와 궁합 좋고, 2부 클리커가 그 최소 예시(Model/Presenter 분리).

# 00. 강사 사전 학습 가이드

> 🎓 이 문서는 **R3·리액티브 프로그래밍을 처음 접하는 강사**가 수업 전에 스스로 준비하기 위한 가이드입니다.
> 차시별 수업 진행 방법은 강사용 01~06 문서를, 강의 일정·평가·배포 등 운영 전반은 [00. 수업 개요](../00.%20수업%20개요.md)를 참조하세요 — 이 문서는 "무엇을 어떤 순서로 공부할지"(1장)와 "강사가 정확히 알고 있어야 할 개념·답변"(2~4장)만 다룹니다.
> Unity 기본기와 `Update`·`event`·코루틴(`StartCoroutine`/`yield return`) 경험은 있다고 가정합니다.

> 📦 배포 모델 메모: 학생은 저장소를 클론하지 않고 **빈 URP 프로젝트에 R3·UniTask·Input System을 설치한 뒤 배포용 `.unitypackage`를 임포트**합니다(1부 통합 데모 + 2부 클리커, 폰트는 1부 패키지에 포함). **저장소 클론은 강사 전용**입니다 — 아래 로드맵의 "데모 실행"은 강사가 저장소에서 직접 씬을 생성·관찰하는 행위입니다.

## 1. 준비 로드맵 (수업 전 필수)

> ℹ️ 순서대로 진행하세요 — 각 단계가 다음 단계의 토대가 됩니다. 수업 1~2주 전에 시작하는 것을 강력히 권장합니다.

### 1주 전 — 개념과 어휘

1. **이 강의의 교안 01을 정독** — [01. 리액티브 프로그래밍이란](../교안/01.%20리액티브%20프로그래밍이란.md). 학생에게 가르칠 바로 그 프레임(`Observable = 옵저버 + Operator + IDisposable`)을 먼저 자기 것으로.
2. **ReactiveX 입문 페이지** <https://reactivex.io/intro.html> — "Rx는 옵저버 패턴의 확장"이라는 정의의 1차 출처. 영어가 부담이면 훑기만 해도 됨(교안이 핵심을 옮겨 둠).
3. **R3 README** <https://github.com/Cysharp/R3> — 전체 그림. 처음에는 ①Basic Usage ②Subscription Management(AddTo) ③Unity 섹션 세 곳만. 연산자 표는 사전처럼 쓰는 것이지 외우는 게 아님.

이 단계의 목표: **"리액티브가 뭐냐"는 질문에 옵저버 패턴부터 시작해 3분 동안 막힘없이 말할 수 있는 상태.**

### 4~5일 전 — 직접 굴려 보기

4. **이 저장소를 클론해 데모를 실행** — 메뉴 `Tools > Reactive Study > Create All Scenes`로 두 데모 씬을 한 번에 생성한 뒤(개별 생성도 가능, 아래 표), `Assets/Scenes/`에서 열어 Play. 교안 02~05를 옆에 두고 통합 씬의 각 섹션을 대조.

    | 데모     | 데모 씬(`Assets/Scenes/`)                                               | 개별 생성 메뉴                                      |
    | -------- | ----------------------------------------------------------------------- | --------------------------------------------------- |
    | 1부 기초 | `ReactiveBasics`(§1 생성·§2 연산자·§3 수명·§4 바인딩·§5 비동기, 완성본) | `Tools > Reactive Study > 01. R3 기초 통합 씬 생성` |
    | 2부 게임 | `ReactiveClicker`(스켈레톤)                                             | `Tools > Reactive Study > 02. 클리커(2부) 씬 생성`  |

    > 💡 씬 파일(`*.unity`)은 git에 커밋되지 않습니다 — 생성기가 정본이라 망가뜨려도 위 메뉴로 언제든 재생성할 수 있습니다. 절차·체크리스트는 [씬 생성·패키지 도구 가이드](../데모/씬%20생성·패키지%20도구%20가이드.md).

5. **데모 코드 통독** — 강사용(완성본) 데모 컴포넌트를 각 줄이 교안의 어느 절인지 확인하며 읽기.

    - `Assets/Scripts/ReactiveBasics/` — `CreateStreamsDemo.cs`(02) · `OperatorsDemo.cs`(03) · `LifetimeDemo.cs`(+`LifetimeProbe`, 04) · `BindingDemo.cs`·`AsyncDemo.cs`(05).
    - `Assets/Scripts/ClickerGame/` — `ClickerModel.cs`·`ClickerPresenter.cs`(06 완성본).
    - 특히 `LifetimeDemo`의 "누수 만들기" 버튼과 `Window > Observable Tracker`는 수업 중 시연 효과가 크므로 손에 익혀 둘 것.

    > 💡 학생 패키지의 코드는 `My` 접두사 버전(`MyCreateStreamsDemo`·`MyOperatorsDemo`·`MyLifetimeDemo`(+`MyLifetimeProbe`)·`MyBindingDemo`·`MyAsyncDemo`·`MyClickerModel`·`MyClickerPresenter`)입니다. **1부(`ReactiveBasics` 통합 씬)·2부(`ReactiveClicker`) 모두 핵심 R3 메서드 본문을 비운 스켈레톤(라이브 코딩용)**이며 주석이 제거돼 있습니다 — 학생은 빈 메서드를 강사와 함께 채웁니다. 강사용 씬(`Assets/Scenes/`)은 접두사 없는 완성본 컴포넌트로 만들어집니다(도달점·대조용).

6. **작은 실험 3개를 직접 작성**(빈 씬에서, 보지 않고):
    - ① 버튼 클릭을 `ThrottleFirst`로 쿨다운 걸기
    - ② `ReactiveProperty<int>` 하나를 텍스트에 바인딩하기
    - ③ `Observable.Interval` 구독을 만들었다가 오브젝트 파괴로 자동 해제되는지 Tracker로 확인하기
      막히면 교안 02~05를 다시. **이 3개를 못 하면 라이브 코딩은 미루고 완성 데모 시연으로 수업할 것.**

### 2~3일 전 — 게임을 한 번 완주

7. **교안 06을 따라 클리커를 처음부터 끝까지 직접 만들기** — 학생이 겪을 일을 미리 겪는 단계. 학생은 `ReactiveClicker` 스켈레톤 패키지에서 시작하므로, 강사도 스켈레톤(또는 빈 손)에서 출발해 완성본(`ReactiveClicker` 씬·`ClickerModel`/`ClickerPresenter`)에 도달해 볼 것. 어디서 오타가 나기 쉬운지, 어느 단계가 오래 걸리는지 몸으로 알게 됨. 본인이 막힌 곳이 학생이 막힐 곳임 — 메모해 두고 수업에서 선제 대응.
8. **(선택) 심화 읽을거리**
    - R3 발표 글(저자 neuecc): <https://neuecc.medium.com/r3-a-new-modern-reimplementation-of-reactive-extensions-for-c-cf29abcc5826> — 왜 3세대를 다시 만들었는지, UniRx와의 차이의 1차 근거.
    - Introduction to Rx: <https://introtorx.com> — 안티패턴 경고("pull을 push로 강제하지 말라")의 출처.
    - UniRx README(아카이브): <https://github.com/neuecc/UniRx> — 학생이 옛 자료를 들고 올 때를 대비한 역사 지식.

## 2. 강사가 먼저 잡아야 할 멘탈 모델 5개

아래 다섯 항목을 **자기 말로 설명할 수 있는지** 확인하세요. 상세 설명·코드는 교안에 있습니다 — 여기서는 강사가 머릿속에 가질 "그림"만 요약합니다.

1. **리액티브 = 옵저버 패턴의 확장** — 새 패러다임 신비화 금지. "이미 아는 옵저버에 연산자와 표준 해제를 더한 것."
2. **이벤트는 시간 위의 데이터 스트림** — 마블 다이어그램이 공용어. 칠판에 가로선+구슬을 그리는 습관.
3. **Subscribe는 등록, Dispose는 해제** — `+=`/`-=`의 표준화. 모든 수명 관리가 이 위에 있음.
4. **연산자 합성이 진짜 가치** — 더블클릭·쿨다운·콤보가 상태 플래그 없이 한두 줄이 되는 것. 이걸 보여 주는 게 강의의 하이라이트.
5. **만병통치약이 아님** — pull이 자연스러운 곳에 강제하지 말 것. "장점만 일부 취하기"가 권장 포지션.

### Unity 관용구 → R3 매핑 (한눈)

학생은 이미 `Update`·`event`·코루틴으로 "반응"을 짜 왔다. 새 개념을 0에서 가르치는 대신 **"늘 하던 그것이 무엇으로 바뀌는가"** 로 다리를 놓으면 흡수가 빠르다. 교안 01·02·03이 각 행을 본문 비교 박스(🔁)로 펼치고, 차시별 강사용에 배경·예상 Q&A를 둔다.

| Unity에서 늘 하던 방식              | R3 대응                                     | 다루는 교안 |
| ----------------------------------- | ------------------------------------------- | ----------- |
| `UnityEvent` / `Action` 콜백        | `Observable` + `Subscribe`                  | 01·05       |
| `Update` 폴링(매 프레임 검사)       | `Observable.EveryUpdate()` + `Where`        | 05          |
| 코루틴 `WaitForSeconds` / 조건 대기 | `Timer` · `Delay` · `Debounce`              | 02·03       |
| `Invoke` / `InvokeRepeating`        | `Timer` · `Interval`                        | 02          |
| bool 플래그 + 수동 전파             | `ReactiveProperty`(현재값 보유 · 자동 통지) | 02·05       |
| 수동 `-=` 해제 / `OnDestroy` 정리   | `AddTo` · `CompositeDisposable`             | 04          |

> 💡 왼쪽 열은 학생이 **이미 아는 것** — 1차시 도입에서 "오른쪽 열을 하나씩 채운다"는 예고로 써도 좋다. 학생이 아는 왼쪽에서 출발하는 것이 이 과정의 일관된 교수 전략이다.

> ℹ️ 이 표는 **'기존 방식 → R3'** 축이다. UniRx 연산자 이름이 R3에서 어떻게 바뀌는지(`Throttle→Debounce`, `Buffer→Chunk` 등)의 **'UniRx → R3' 이름 매핑**은 축이 다르므로 [교안 03 부록](../교안/03.%20연산자%20-%20스트림을%20다듬고%20합성하기.md)을 참조.

## 3. 검증된 사실 가드레일 (학생 앞에서 단정하지 말 것)

> ❗ 수업 중 아래 항목은 **여기 적힌 표현까지만** 말할 것. 더 단정하면 틀릴 수 있음.

- **버전** — 이 자료는 R3 **수업 시점 최신 버전**을 기준으로 함. 설치 시 <https://github.com/Cysharp/R3/releases>에서 최신 버전을 확인할 것(Releases가 정본 체인지로그).
- **지원 범위** — 공식 문서가 밝힌 것은 "R3는 **Unity 2021.3 이상** 지원"까지. 특정 유니티 버전·IL2CPP 실동작을 본 자료가 보증하지 않음(이 저장소의 Unity 6.3 에디터 동작은 확인 대상).
- **UniRx 아카이브** — "아카이브(읽기 전용)되었고 README가 R3 사용을 권한다"까지만. 정확한 일자를 못 박지 말 것.
- **UniRx의 존재 이유** — "공식 Rx.NET이 Unity에서 동작하지 않고 iOS IL2CPP(AOT) 호환성 문제가 있어 재구현"이 정확한 표현. 막연히 "Mono 문제"라고 하지 말 것.
- **R3에 `Retry` 없음** — R3 연산자에 `Retry`가 없음(소스 확인 기준). 재시도는 `FromAsync` 내 루프 또는 `Catch` 조합으로 안내.
- **"극적 단순화" 같은 과장 금지** — 정확한 표현은 "더 짧고 선언적으로 표현할 수 있다". 디버깅·학습 비용이라는 대가를 항상 함께 말할 것.
- **"모든 event를 Rx로 대체" 금지** — _Introduction to Rx_ 의 안티패턴 경고와 함께만 제시.
- **채택률 수치 금지** — 스타 수·점유율 같은 정량 주장 대신 "정식 후속작으로 수렴" 같은 정성 표현만.

## 4. 자주 받는 질문 총괄 (차시 무관)

수업 흐름을 끊지 않도록, 무관 차시에서도 나오는 공통 질문의 **기준 답변**을 모았습니다. 차시별 세부 Q&A는 강사용 01~06 문서를 참조하세요.

> ❓ **UniRx 튜토리얼이 훨씬 많던데 그걸로 배우면 안 되나요?**
> 개념은 같으니 참고는 가능. 단 UniRx는 아카이브됐고 연산자 이름이 다름(`Throttle→Debounce`, `Buffer→Chunk`). 새 코드는 R3로 쓰고, 옛 자료는 교안 03 부록의 매핑표를 거쳐 읽으라고 안내.

> ❓ **UniTask랑은 무슨 관계인가요?**
> 다른 도구임. UniTask는 단발 비동기 작업(async/await)의 고성능 구현, R3는 **반복되는 이벤트 스트림**의 합성. 같은 Cysharp 생태계라 함께 쓰는 일이 많고, R3는 await 통합(교안 05)으로 둘 사이를 잇는다 — "한 번 일어나는 일은 await, 여러 번 일어나는 일은 Observable"이 기준선.

> ❓ **성능은 괜찮나요? GC는요?**
> R3는 현대적 재설계로 내부 할당을 줄였고 구독 추적도 중앙화함. 다만 수치 비교를 단정하지 말 것(가드레일). 게임플레이 코드 기준 일반적 사용에서 문제가 되는 경우는 드물고, 핫패스(매 프레임 수천 회)에는 어떤 추상화든 신중해야 한다는 일반론으로 답할 것.

> ❓ **MVVM/MVP 같은 아키텍처를 꼭 같이 써야 하나요?**
> 아님. R3는 패턴 없이도 쓸 수 있음. 다만 ReactiveProperty가 Model→View 통지를 자연스럽게 만들어 MVP류와 궁합이 좋고, 2부 클리커가 그 최소 예시(Model/Presenter 분리)임.

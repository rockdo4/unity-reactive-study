# Part 1. 개념 브리핑 (~12분)

## 1-1. 오늘 만들 게임 — 완성 스펙

`ReactiveClicker`(완성본) 씬 Play로 1분 시연.

- 클릭 버튼 → 골드 획득(클릭당 골드는 업그레이드로 증가).
- 업그레이드 2종: 클릭 강화 / 도우미 고용(초당 자동 수입) — **살 수 있을 때만 버튼 활성화**.
- 빠른 연속 클릭 → 콤보, 10콤보마다 보너스 골드.
- 피버 버튼: 5초간 클릭 골드 2배, 15초 쿨다운.

> 💡 전달 요점: `Update()` 0개, 코루틴 0개, 상태 플래그 0개로 처음부터 만든다.

## 1-2. Model과 Presenter를 나눈다

| 클래스               | 정체           | 책임                                          | 모르는 것 |
| -------------------- | -------------- | --------------------------------------------- | --------- |
| `MyClickerModel`     | 순수 C# 클래스 | 상태(`ReactiveProperty`)와 규칙(구매·피버 등) | UI        |
| `MyClickerPresenter` | MonoBehaviour  | 입력(버튼)→Model 호출, Model 상태→UI 바인딩   | 게임 규칙 |

Model이 UI를 모르므로 ①규칙 단위 테스트 가능 ②UI 교체에도 Model 불변. 05차시 끝 예고한 **MVP 구조**를 여기서 실제 사용.

> 📌 아키텍처 논쟁 회피(MVP 강의 아님). 명분 한 문장 — "Model이 UI를 모르면, 규칙을 테스트할 수 있고 UI를 갈아치워도 안 깨진다."

## 1-3. 채울 곳은 네 군데 — 부품 미리보기

스켈레톤(`MyClickerModel`·`MyClickerPresenter`) 함께 열어, 필드·게임 규칙·Inspector 연결은 **이미 끝남**, 채울 R3 메서드만 네 개임을 확인.

| 채울 메서드                    | 핵심 부품                                                             | 배운 차시 |
| ------------------------------ | --------------------------------------------------------------------- | --------- |
| `MyClickerModel` **생성자**    | `CombineLatest`(파생 상태) + `Interval`(자동 수입) + `AddTo`(수명)    | 03·02·04  |
| `MyClickerModel.TryStartFever` | `Observable.Timer`(일회성 자동 종료)                                  | 02        |
| `MyClickerPresenter.BindInput` | `Share` + `Merge`+`Scan`+`Debounce`(콤보) + `Where` + `ThrottleFirst` | 03        |
| `MyClickerPresenter.BindView`  | `Subscribe` 바인딩 3패턴 + `CombineLatest`                            | 05        |

> 📌 `Start()`가 이미 `m_Model = new MyClickerModel(); BindInput(); BindView();` 호출, `OnDestroy()`가 `m_Model?.Dispose()` 호출(제공됨). 사이 본문만 채움.

# Part 2. 단계별 구현 따라하기

## 1차시 — 0~3단계

### 0단계. 준비 (~7분)

1. **완성본 시연**(강사 프로젝트): `ReactiveClicker` Play → 클릭→골드, 업그레이드, 콤보, 피버까지. 오늘 만들 결과물 제시.
2. **학생 씬 확인**: `R3Lecture-Student-2-Clicker` 임포트 후 학생용 클리커 씬 Play → 버튼 **무반응** 확인(채울 메서드 빈 채). 골드 텍스트도 갱신 안 됨.
3. **스켈레톤 구조 읽기**: 위 1-3 표대로 "채울 곳은 네 개" 함께 짚기.

> ⚠️ **함정**: 강사 확인용 완성본 씬과 학생 스켈레톤 씬은 **다른 씬·다른 클래스**(`ClickerPresenter` vs `MyClickerPresenter`). 학생이 완성본 씬을 열어 동작한다고 하면 자기 스켈레톤 씬에서 작업 중인지 확인시킬 것.

### 1단계. 골드와 클릭 — 가장 작은 게임 (~12분)

> 🔧 **오늘 쓸 부품**: 02차시 `ReactiveProperty`, 05차시 `Subscribe` 바인딩.

가장 작은 게임만 살림. `MyClickerModel`의 `Gold`·`GoldPerClick` 필드와 `Click()`은 제공됨(읽기만), Presenter `BindInput`/`BindView`에 골드 관련 두 줄만 우선 채움.

`BindInput`에 — 클릭을 Model로:

```csharp
m_ClickButton.OnClickAsObservable()    // 버튼 클릭을 스트림으로
    .Subscribe(_ => m_Model.Click())   // 클릭마다 Model에 위임(규칙은 Model이)
    .AddTo(this);                      // Presenter 파괴 시 자동 해제
```

`BindView`에 — Gold를 텍스트로:

```csharp
// Gold가 바뀔 때마다(클릭이든 자동 수입이든 출처 무관) 텍스트를 값과 일치시킴
m_Model.Gold.Subscribe(gold => m_GoldText.text = $"골드 {gold:N0}").AddTo(this);
```

**실행·관찰**: Play → 클릭하면 골드가 오른다.

> 📌 짚을 점: 골드 텍스트 갱신 코드가 클릭 핸들러에 **없음**. Presenter는 "골드가 바뀌면 텍스트 갱신"을 **한 번 선언**했을 뿐, 출처(클릭이든 자동 수입이든) 무관하게 표시가 항상 값과 일치. 3단계(자동 수입)에서 복선 회수.
>
> 📌 인스펙터 참조 연결(`m_ClickButton`·`m_GoldText`)을 프로젝터로 천천히 — 의외로 많이 막힘. (스켈레톤 씬엔 연결됨, 학생이 빈 GameObject를 새로 만들었다면 직접 연결 필요 안내.)

### 2단계. 업그레이드 — CombineLatest로 "살 수 있나" (~12분)

> 🔧 **오늘 쓸 부품**: 03차시 `CombineLatest`, 05차시 조건→`interactable` 바인딩.

`MyClickerModel` **생성자** 일부 채움. `CanBuyClickUpgrade`/`CanBuyAutoUpgrade` 필드 선언과 `TryBuyClickUpgrade`/`TryBuyAutoUpgrade` 규칙은 제공됨(읽기만). 생성자에서 파생 상태 생성:

```csharp
public MyClickerModel()
{
    // Gold:  --100----------250-->
    // Cost:  --200----------200-->
    // 결과:  --false--------true-->   (둘 중 하나만 바뀌어도 재평가)
    CanBuyClickUpgrade = Gold.CombineLatest(ClickUpgradeCost,  // 골드·비용 최신값을 짝지어
            (gold, cost) => gold >= cost)                     //   살 수 있나? bool로
        .ToReadOnlyReactiveProperty()                         // 현재값 보유 읽기 전용 RP로 굳힘
        .AddTo(m_Disposables);                                // Model 수명에 묶음(누수 방지)

    CanBuyAutoUpgrade = Gold.CombineLatest(AutoUpgradeCost, (gold, cost) => gold >= cost)
        .ToReadOnlyReactiveProperty()
        .AddTo(m_Disposables);

    // (자동 수입 Interval과 RP AddTo는 3단계에서)
}
```

- `CombineLatest`(3차시)가 골드·비용 **최신값 조합**으로 `bool` 스트림 생성, `ToReadOnlyReactiveProperty()`로 "현재값 보유 읽기 전용 프로퍼티"로 굳힘.
- 구매 검증은 제공된 `TryBuy...` 패턴 — **Model이 규칙의 최종 관문**, UI 활성화는 편의일 뿐.

#### 📎 보충 — `ReadOnlyReactiveProperty` · `ToReadOnlyReactiveProperty()` (교안 미수록, 여기서 처음)

교안은 `ReactiveProperty`(02차시)까지만 다룸. `ReadOnlyReactiveProperty`와 `ToReadOnlyReactiveProperty()`는 클리커에서 처음 나오므로 이 단계에서 한 번 짚고 감.

**타입 관계 (R3 1.3 실제 상속)**

```
Observable<T>
  └ ReadOnlyReactiveProperty<T>   // 추상 · 읽기 전용(CurrentValue만, 세터 없음) · 그 자체가 Observable
      └ ReactiveProperty<T>       // .Value 세터 추가 → 읽기·쓰기
```

- `ReactiveProperty<T>` = **읽고 쓰는** RP. `.Value = x`로 변경(02차시 그것 — `Gold`·`IsFever` 등).
- `ReadOnlyReactiveProperty<T>` = RP의 **읽기 전용 부모**. 구독은 똑같이 되고 현재값은 `CurrentValue`로 읽지만, **`.Value` 세터가 없어 밖에서 못 바꿈**.
- `ToReadOnlyReactiveProperty()` = 아무 `Observable<T>`이나 받아 "**현재값을 들고 있는** 읽기 전용 RP"로 굳히는 연산자. 흘러가는 스트림에 "지금 값" 개념과 구독 즉시 replay를 붙여 줌.

**왜 클리커가 이걸 쓰나 (`CanBuyClickUpgrade`)**

- 직접 대입하는 상태가 아니라 골드·비용에서 **계산되는 파생값** → Model이 `.Value =`로 쓸 일이 없음(원천이 둘). 그래서 `CombineLatest`로 파생만 함.
- 그런데 Presenter는 구독 즉시 첫 값을 받아야 버튼 **초기 활성 상태**가 맞고, "지금 살 수 있나"의 현재값도 필요 → 맨 Observable로 두지 않고 `ToReadOnlyReactiveProperty()`로 **현재값 보유 프로퍼티**로 변환.
- 노출 타입을 `ReadOnlyReactiveProperty<bool>`로 둠 → Presenter는 **구독·읽기만, 쓰기 불가**. "이 값은 Model이 계산한다, 너는 못 바꾼다"가 **타입으로 강제**됨(캡슐화).

|                               | `.Value` 쓰기 | 현재값 읽기          | 만드는 법                                                    | 클리커 예                                |
| ----------------------------- | ------------- | -------------------- | ------------------------------------------------------------ | ---------------------------------------- |
| `ReactiveProperty<T>`         | O             | `.Value`             | `new(초기값)`                                                | `Gold`·`GoldPerClick`·`IsFever`          |
| `ReadOnlyReactiveProperty<T>` | X             | `.CurrentValue`/구독 | `Observable.ToReadOnlyReactiveProperty()` (또는 RP 업캐스트) | `CanBuyClickUpgrade`·`CanBuyAutoUpgrade` |

> 💡 한 줄 전달: "RP는 읽기·쓰기, ReadOnlyRP는 읽기 전용. RP가 ReadOnlyRP를 상속하니 안에선 RP로 쓰고 밖엔 ReadOnlyRP로 노출하면 캡슐화가 공짜."
>
> ❗ 수명 주의: `ToReadOnlyReactiveProperty()`는 내부적으로 원본(`CombineLatest`)을 **구독**함 → 그 자체가 정리 대상. 그래서 `.AddTo(m_Disposables)` 필수(3단계 누수 주제와 연결).

`BindInput`에 업그레이드 구매 추가:

```csharp
m_ClickUpgradeButton.OnClickAsObservable()
    .Subscribe(_ =>
    {
        if (m_Model.TryBuyClickUpgrade())   // 구매 성공 시(골드 충분)에만 true
            Log($"클릭 강화! 클릭당 +{m_Model.GoldPerClick.Value:N0}");
    })
    .AddTo(this);

m_AutoUpgradeButton.OnClickAsObservable()
    .Subscribe(_ =>
    {
        if (m_Model.TryBuyAutoUpgrade())    // 실패(골드 부족)면 false → 로그 없음
            Log($"도우미 고용! 초당 +{m_Model.GoldPerSecond.Value:N0}");
    })
    .AddTo(this);
```

`BindView`에 버튼 활성화·비용 표시 추가:

```csharp
// "살 수 있나"(2단계 파생 상태)가 바뀔 때마다 버튼 활성/비활성 — 우리가 끄고 켜지 않음
m_Model.CanBuyClickUpgrade.Subscribe(can => m_ClickUpgradeButton.interactable = can).AddTo(this);
m_Model.CanBuyAutoUpgrade.Subscribe(can => m_AutoUpgradeButton.interactable = can).AddTo(this);

// 비용이 바뀔 때마다(구매로 오를 때마다) 라벨 갱신
m_Model.ClickUpgradeCost.Subscribe(cost => m_ClickUpgradeLabel.text = $"클릭 강화\n비용 {cost:N0}").AddTo(this);
m_Model.AutoUpgradeCost.Subscribe(cost => m_AutoUpgradeLabel.text = $"도우미 고용\n비용 {cost:N0}").AddTo(this);
```

> 📌 `Log(...)`는 스켈레톤에 이미 있는 헬퍼(이벤트 로그 5줄 유지) — 새로 만들 필요 없음.

**실행·관찰**: 골드 10을 모으면 클릭 강화 버튼이 **저절로** 켜지고, 사면 꺼졌다가 다시 모이면 켜진다.

> 📌 활성화를 '호출'한 적 없음 → '선언'만 함.
>
> - 코드 어디에도 버튼 enable/disable 호출 없음
> - `CombineLatest` 한 줄이 골드·비용 변화를 자동 반영

### 3단계. 자동 수입 — Interval과 Model의 수명 (~12분)

> 🔧 **오늘 쓸 부품**: 02차시 `Observable.Interval`, 04차시 `CompositeDisposable`.

도우미 고용 시 1초마다 골드 자동 유입. **Model 안에서 시간이 흐름** → 04차시 구독 수명 관리가 실전 등장. 생성자 완성:

```csharp
public MyClickerModel()
{
    CanBuyClickUpgrade = Gold.CombineLatest(ClickUpgradeCost, (gold, cost) => gold >= cost)
        .ToReadOnlyReactiveProperty()
        .AddTo(m_Disposables);

    CanBuyAutoUpgrade = Gold.CombineLatest(AutoUpgradeCost, (gold, cost) => gold >= cost)
        .ToReadOnlyReactiveProperty()
        .AddTo(m_Disposables);

    // 자동 수입: 1초마다 GoldPerSecond만큼 지급
    Observable.Interval(TimeSpan.FromSeconds(1))       // 1초마다 tick(무한 스트림)
        .Where(_ => GoldPerSecond.Value > 0)           // 도우미를 사야(>0) 통과 — 그전엔 무변화
        .Subscribe(_ => Gold.Value += GoldPerSecond.Value)  // 통과한 tick마다 골드 +초당수입
        .AddTo(m_Disposables);                         // ★ 빠뜨리면 씬 나가도 영원히 돎(6단계 누수)

    // ReactiveProperty들도 전부 그릇에 — Dispose 한 줄로 정리되게
    Gold.AddTo(m_Disposables);
    GoldPerClick.AddTo(m_Disposables);
    GoldPerSecond.AddTo(m_Disposables);
    ClickUpgradeCost.AddTo(m_Disposables);
    AutoUpgradeCost.AddTo(m_Disposables);
    IsFever.AddTo(m_Disposables);
}
```

`BindView`에 스탯 표시 추가 — 두 값을 합쳐 한 줄로:

```csharp
m_Model.GoldPerClick                              // 클릭당 골드가 바뀌거나…
    .CombineLatest(m_Model.GoldPerSecond,         // …초당 골드가 바뀌면 둘의 최신값으로
        (perClick, perSec) => $"클릭당 +{perClick:N0} · 초당 +{perSec:N0}")  // 한 문장 생성
    .Subscribe(text => m_StatsText.text = text)   // 그 문장을 그대로 표시
    .AddTo(this);
```

> ❗ **04차시의 실전**. Model은 MonoBehaviour 아님 → `AddTo(this)` 못 씀 → 순수 C# 표준인 `CompositeDisposable`(`m_Disposables`, 스켈레톤에 선언됨)에 모든 구독·발행기를 담고, 제공된 `Dispose()`가 `m_Disposables.Dispose()` 한 줄로 정리. Presenter의 `OnDestroy → m_Model.Dispose()` 사슬은 스켈레톤에 이미 있음. **빠뜨리면 씬을 나가도 Interval이 영원히 도는** 교과서적 누수 — 6단계에서 Tracker로 확인.

**실행·관찰**: 도우미를 사면 가만히 있어도 골드가 1초마다 오른다.

> 📌 1단계 복선 회수: 자동 수입이 들어와도 **골드 표시 코드는 1단계 그대로**. 출처가 늘어도 바인딩은 불변.

> 📌 **중간 Play 체크포인트(1차시 끝)**: 클릭·업그레이드·자동 수입 세 가지 동작하면 1차시 합격. 컴파일 깨졌으면 다음 차시 막힘 → `Unity_GetConsoleLogs`로 에러 0 전원 확인 후 마무리.

## 2차시 — 4~6단계

### 4단계. 콤보 — Merge + Scan 누적기 (~18분, 최고 난도)

> 🔧 **오늘 쓸 부품**: 03차시 `Share`·`Select`·`Merge`·`Debounce`·`Scan`·`Where`.

빠른 연속 클릭 → 콤보 누적, 0.8초 쉬면 끊김, 10콤보마다 보너스. **순수 연산자 합성만으로** 구현. 콤보는 "입력 리듬에 대한 해석" → **Presenter 소관**(`BindInput`).

**마블 먼저, 코드 나중.** 칠판에 다섯 줄 그린 뒤 코드:

```
clicks:        --o-o-o------------o-->
Select(1):     --1-1-1------------1-->
Debounce(0):   ------------0--------->   (0.8초 잠잠 후)
Merge:         --1-1-1-----0------1-->
Scan:          --1-2-3-----0------1-->   ← 콤보 카운터 완성
```

`BindInput`에서 1단계 클릭 한 줄을 **다음으로 교체**(스켈레톤에 `ComboResetDelay = 800ms`, `ComboBonusUnit = 10` 상수 선언됨):

```csharp
// 클릭 스트림을 본체·콤보·보너스 세 곳이 쓰므로 Share로 구독 하나를 공유
var clicks = m_ClickButton.OnClickAsObservable().Share();

clicks.Subscribe(_ => m_Model.Click()).AddTo(this);   // 본체: 클릭 → 골드

// 콤보 누적기 — 클릭은 +1, 0.8초 잠잠해지면 0으로 리셋
// clicks: --o-o-o------o-->
// Scan:   --1-2-3--0---1-->   (0.8초 멈춤이 리셋 신호)
var combo = clicks
    .Select(_ => 1)                          // 클릭 1회 = +1 신호
    .Merge(clicks.Debounce(ComboResetDelay)  // ↔ 클릭이 0.8초간 없으면…
        .Select(_ => 0))                     // …리셋 신호 0을 한 번 발행
    .Scan(0, (acc, x) =>                     // 직전 콤보(acc)를 들고 있다가
        x == 0 ? 0 : acc + 1)                //   0이 오면 리셋, +1이면 누적
    .Share();                                // 표시·보너스 두 곳이 결과 공유

combo
    .Subscribe(c => m_ComboText.text = c >= 2 ? $"{c} 콤보!" : string.Empty)  // 2콤보부터 표시
    .AddTo(this);

// 콤보 보너스 — 10·20·30…에 도달하는 순간만 통과
combo
    .Where(c => c > 0 && c % ComboBonusUnit == 0)     // 10의 배수에 닿는 순간만
    .Subscribe(c =>
    {
        long bonus = c * m_Model.GoldPerClick.Value;  // 보너스 = 콤보수 × 클릭당 골드
        m_Model.AddBonus(bonus);                      // 입금(제공된 메서드)
        Log($"{c}콤보 달성! 보너스 +{bonus:N0}");
    })
    .AddTo(this);
```

> 📌 짚을 점: Scan 누적값을 0으로 만드는 건 Debounce가 보낸 0(클릭이 0.8초 멈추면). 명령형이라면 "마지막 클릭 시각 저장 + 매 프레임 경과 검사 + 카운터 리셋"이 필요했을 자리.
>
> ⚠️ **함정**: `Share()` 빠뜨림 — 동작은 하나 버튼 클릭 구독이 세 벌. "두 곳 이상에서 쓰면 Share"(3차시) 회수 기회. `Model.AddBonus`는 스켈레톤에 이미 있는 입금 창구 메서드(새로 만들지 말 것).

**실행·관찰**: 연타하면 "n 콤보!"가 오르고, 10콤보에 보너스 로그가 찍힌다.

### 5단계. 피버 — ThrottleFirst 쿨다운과 Timer (~12분)

> 🔧 **오늘 쓸 부품**: 03차시 `ThrottleFirst`, 02차시 `Observable.Timer`, 05차시 바인딩.

피버 버튼 → 5초간 클릭 골드 2배. 쿨다운 15초는 **`ThrottleFirst`가 게이트**(첫 입력만 통과, 이후 구간 무시). 클릭 2배 반영은 제공된 `Click()`(`IsFever.Value ? 2 : 1`)에 이미 있음 → `TryStartFever`와 입력·바인딩만 채움.

`MyClickerModel.TryStartFever`:

```csharp
public bool TryStartFever()
{
    if (IsFever.Value)        // 이미 피버 중이면
        return false;         //   무시(중복 시작 방지)
    IsFever.Value = true;     // 피버 ON → 구독자(버튼색·문구·Click 2배)에 자동 통지
    // FeverDuration(5초) 뒤 한 번 발행하고 끝 → 그때 피버 OFF. Timer는 일회성(02차시)
    Observable.Timer(FeverDuration).Subscribe(_ => IsFever.Value = false).AddTo(m_Disposables);
    return true;
}
```

`BindInput`에 피버 입력 추가(`FeverCooldown = 15초` 상수는 스켈레톤에 선언됨):

```csharp
// 클릭: --c-c-c--------c-->
// 통과: --c------------c-->   (첫 발만, 15초간 나머지 무시)
m_FeverButton.OnClickAsObservable()
    .ThrottleFirst(FeverCooldown)            // 15초 쿨다운 게이트 — 첫 클릭만 통과
    .Subscribe(_ =>
    {
        if (m_Model.TryStartFever())         // 게이트는 통과해도 시작은 Model이 최종 결정
            Log($"피버 시작! {m_Model.FeverDuration.TotalSeconds:0}초간 클릭 2배");
    })
    .AddTo(this);
```

`BindView`에 피버 상태 바인딩 추가:

```csharp
m_Model.IsFever.Subscribe(isFever =>          // 피버 ON/OFF가 바뀔 때마다(구독 즉시 1회 포함)
    {
        m_FeverStateText.text = isFever ? "피버 중! 클릭 2배" : "피버 (15초 쿨다운)";  // 문구 갱신
        if (m_ClickButtonImage != null)
            m_ClickButtonImage.color = isFever ? new Color(1f, 0.6f, 0.2f) : Color.white;  // 버튼색 갱신
    })
    .AddTo(this);
```

> 📌 짚을 점: 쿨다운에 시각 저장 변수·코루틴 안 만듦. "첫 입력만 통과, 15초 무시" 정책이 연산자 한 개(`ThrottleFirst`). 5초 뒤 자동 종료도 `Timer` 하나 — `IsFever` 바인딩(문구+버튼 색)은 바인딩 3패턴 복습.

**실행·관찰**: 피버를 누르면 클릭 버튼이 주황색, 클릭당 골드 2배, 5초 뒤 원상복구. 15초 안에 다시 눌러도 무시된다.

## 예상 Q&A

> ❓ **왜 `CanBuy*`를 그냥 `ReactiveProperty`나 맨 `Observable`로 안 두고 `ReadOnlyReactiveProperty`로?** (교안 미수록)
> 세 가지가 한 번에 풀림. ①파생값이라 Model이 직접 `.Value =`로 쓸 게 아님 → 읽기 전용이 의미에 맞음. ②밖(Presenter)이 실수로 못 바꾸게 타입으로 막음(캡슐화). ③맨 Observable과 달리 **현재값을 보유**해 구독 즉시 첫 값이 와서 버튼 초기 상태가 맞음. `ToReadOnlyReactiveProperty()`가 ①②③를 동시에 줌. 상세·표는 2단계 보충 박스.

> ❓ **콤보를 Model에 두면 안 되는지**
> 둘 다 가능. 이 강의는 "콤보는 **입력의 리듬**에 대한 해석"이라 Presenter에 둠(Model은 입력 장치를 모르는 게 깔끔). 콤보가 게임 규칙(저장돼야 하는 상태)이 되면 Model로 옮기는 게 맞음 — 좋은 토론 주제.

> ❓ **TryStartFever의 Timer 구독이 CompositeDisposable에 계속 쌓이는지**
> 쌓임 — 피버 한 번에 일회성 Timer 구독 하나씩. Timer는 완료되는 스트림이라 실질 누수는 아니고 그릇에 빈 껍데기가 남는 정도(쿨다운 15초 게이트로 횟수 제한됨). "거슬리면 SerialDisposable류로 교체"는 심화 과제로.

> ❓ **골드를 long으로 한 이유 (int면 안 되나?)**
> 이 데모 한 판만 보면 int(약 21억)로도 돌아감. long인 이유는 ①클리커/방치형 장르는 비용·수입이 **지수로 증가**(여기선 ×1.6·×1.8)해 오래 두면 int 상한을 넘김 ②`ReactiveProperty` 대여섯 개라 long의 비용이 사실상 0 ③자릿수 표기(`:N0`)와 함께 장르 관습. 즉 **필수는 아니고 관습+안전**. int로 바꾸려면 `ReactiveProperty<long>`→`<int>`, `Click()` 반환형, `AddBonus` 인자, `(long)Math.Ceiling`→`(int)`만 맞추면 됨(직렬화 필드 아니라 스켈레톤 동기화엔 영향 없음).

> ❓ **저장/로드 방법**
> Model의 ReactiveProperty 값들을 직렬화하면 됨(저장은 `.Value` 읽기, 로드는 `.Value` 대입 — 대입 순간 UI 자동 갱신되는 것도 바인딩의 이점). 강의 범위 밖이라 과제로.

> ❓ **`Update()`에서 매 프레임 검사하던 콤보/쿨다운이 R3에서는 어떻게 바뀌는지** (전환형)
> 콤보는 `Merge`+`Scan`+`Debounce`로, 쿨다운은 `ThrottleFirst`로 **선언 한 줄**이 됨 — "마지막 입력 시각 저장 + 매 프레임 경과 비교 + 플래그 리셋" 명령형 보일러플레이트가 사라짐. ('기존 방식 → R3' 한눈 매핑표는 [00. 강사 사전 학습 가이드](00.%20강사%20사전%20학습%20가이드.md))

## 이해도 퀴즈

1. Model을 순수 C# 클래스로 두는 이점 두 가지를 쓰시오.
2. 콤보 스트림에서 `Debounce`와 `Scan`이 각각 맡는 역할을 쓰시오.
3. Presenter가 파괴될 때 자동 수입 Interval이 멈추기까지의 호출 사슬을 쓰시오.
4. 완성된 클리커에서 "리액티브로 만들 필요가 없었던 것"의 예를 들고 이유를 쓰시오.

> ✅ **해설**
>
> 1. ①UI 없이 규칙을 단위 테스트 가능 ②UI 교체에도 Model 불변(관심사 분리).
> 2. Debounce — 클릭이 0.8초 멈추면 리셋 신호(0)를 발행. Scan — +1/0 신호를 누적해 콤보 수를 유지(0이 오면 리셋).
> 3. `OnDestroy()` → `m_Model.Dispose()` → `m_Disposables.Dispose()` → Interval 구독 해제.
> 4. 예: `TryBuyClickUpgrade`의 비용 계산 — 호출 시점에 한 번 계산하면 끝나는 pull성 로직이라 일반 메서드가 자연스러움(pull을 push로 강제하지 않기).

# Part 1. 개념 브리핑 (~12분)

코드를 치기 전에 칠판/슬라이드로 세 가지만 설명합니다.

## 1-1. 오늘 만들 게임 — 완성 스펙

`ReactiveClicker`(완성본) 씬을 먼저 Play로 1분 시연합니다.

- 클릭 버튼 → 골드 획득(클릭당 골드는 업그레이드로 증가).
- 업그레이드 2종: 클릭 강화 / 도우미 고용(초당 자동 수입) — **살 수 있을 때만 버튼 활성화**.
- 빠르게 연속 클릭하면 콤보 — 10콤보마다 보너스 골드.
- 피버 버튼: 5초간 클릭 골드 2배, 15초 쿨다운.

> 💡 멘트: "오늘과 다음 시간에 이걸 처음부터 만듭니다. **`Update()` 0개, 코루틴 0개, 상태 플래그 0개**로."

## 1-2. Model과 Presenter를 나눈다

| 클래스               | 정체           | 책임                                          | 모르는 것 |
| -------------------- | -------------- | --------------------------------------------- | --------- |
| `MyClickerModel`     | 순수 C# 클래스 | 상태(`ReactiveProperty`)와 규칙(구매·피버 등) | UI        |
| `MyClickerPresenter` | MonoBehaviour  | 입력(버튼)→Model 호출, Model 상태→UI 바인딩   | 게임 규칙 |

Model이 UI를 모르므로 ①규칙을 단위 테스트할 수 있고 ②UI를 갈아치워도 Model이 안 바뀝니다. 05차시 끝에 예고한 **MVP 구조**를 여기서 실제로 씁니다.

> 📌 깊은 아키텍처 논쟁은 피할 것(MVP 강의가 아님). 명분은 한 문장으로 — **"Model이 UI를 모르면, 규칙을 테스트할 수 있고 UI를 갈아치워도 안 깨진다."**

## 1-3. 채울 곳은 네 군데 — 부품 미리보기

스켈레톤(`MyClickerModel`·`MyClickerPresenter`)을 함께 열어, 필드·게임 규칙·Inspector 연결은 **이미 끝나 있고** 우리가 채울 R3 메서드만 네 개임을 확인합니다.

| 채울 메서드                    | 핵심 부품                                                             | 배운 차시 |
| ------------------------------ | --------------------------------------------------------------------- | --------- |
| `MyClickerModel` **생성자**    | `CombineLatest`(파생 상태) + `Interval`(자동 수입) + `AddTo`(수명)    | 03·02·04  |
| `MyClickerModel.TryStartFever` | `Observable.Timer`(일회성 자동 종료)                                  | 02        |
| `MyClickerPresenter.BindInput` | `Share` + `Merge`+`Scan`+`Debounce`(콤보) + `Where` + `ThrottleFirst` | 03        |
| `MyClickerPresenter.BindView`  | `Subscribe` 바인딩 3패턴 + `CombineLatest`                            | 05        |

> 📌 `Start()`가 이미 `m_Model = new MyClickerModel(); BindInput(); BindView();`를 호출하고, `OnDestroy()`가 `m_Model?.Dispose()`를 부릅니다(제공됨). 우리는 그 사이의 본문만 채웁니다.

# Part 2. 단계별 구현 따라하기

## 1차시 — 0~3단계

### 0단계. 준비 (~7분)

1. **완성본 시연**(강사 프로젝트): `ReactiveClicker`에서 Play → 클릭→골드, 업그레이드, 콤보, 피버까지 보여 주기. "오늘 이걸 만든다."
2. **학생 씬 확인**: 학생은 `R3Lecture-Student-2-Clicker`를 임포트하고 학생용 클리커 씬을 Play → 버튼이 **무반응**인 것 확인(채울 메서드가 비어 있으므로). 골드 텍스트도 갱신되지 않음.
3. **스켈레톤 구조 읽기**: 위 1-3 표대로 "채울 곳은 네 개"를 함께 짚기.

> ⚠️ **함정**: 강사 확인용 완성본 씬과 학생 스켈레톤 씬은 **다른 씬·다른 클래스**(`ClickerPresenter` vs `MyClickerPresenter`)입니다. 학생이 완성본 씬을 열어 "되는데요?"라고 하면, 자기 스켈레톤 씬에서 작업 중인지 확인시키세요.

### 1단계. 골드와 클릭 — 가장 작은 게임 (~12분)

> 🔧 **오늘 쓸 부품**: 02차시 `ReactiveProperty`, 05차시 `Subscribe` 바인딩.

먼저 가장 작은 게임만 살립니다. `MyClickerModel`의 `Gold`·`GoldPerClick` 필드와 `Click()`은 이미 제공됨을 확인하고(읽기만), Presenter의 `BindInput`/`BindView`에 골드 관련 두 줄만 우선 채웁니다.

`BindInput`에 — 클릭을 Model로:

```csharp
m_ClickButton.OnClickAsObservable()
    .Subscribe(_ => m_Model.Click())
    .AddTo(this);
```

`BindView`에 — Gold를 텍스트로:

```csharp
m_Model.Gold.Subscribe(gold => m_GoldText.text = $"골드 {gold:N0}").AddTo(this);
```

**실행·관찰**: Play → 클릭하면 골드가 오른다.

> 📌 **여기서 멈추고 짚기**: "골드 텍스트를 갱신하는 코드가 클릭 핸들러에 **없습니다.** Presenter는 '골드가 바뀌면 텍스트를 갱신한다'를 **한 번 선언**했을 뿐이고, 출처(클릭이든 나중의 자동 수입이든)와 무관하게 표시가 항상 값과 맞게 유지됩니다. — 3단계(자동 수입)에서 이 복선을 회수합니다."
>
> 📌 인스펙터 참조 연결(`m_ClickButton`·`m_GoldText`)을 프로젝터로 천천히 — 여기서 의외로 많이 막힘. (스켈레톤 씬엔 연결이 돼 있지만, 학생이 빈 GameObject를 새로 만들었다면 직접 연결해야 함을 안내.)

### 2단계. 업그레이드 — CombineLatest로 "살 수 있나" (~12분)

> 🔧 **오늘 쓸 부품**: 03차시 `CombineLatest`, 05차시 조건→`interactable` 바인딩.

이제 `MyClickerModel` **생성자**의 일부를 채웁니다. `CanBuyClickUpgrade`/`CanBuyAutoUpgrade` 필드 선언과 `TryBuyClickUpgrade`/`TryBuyAutoUpgrade` 규칙은 제공됨(읽기만). 우리는 생성자에서 파생 상태를 만듭니다:

```csharp
public MyClickerModel()
{
    CanBuyClickUpgrade = Gold.CombineLatest(ClickUpgradeCost, (gold, cost) => gold >= cost)
        .ToReadOnlyReactiveProperty()
        .AddTo(m_Disposables);

    CanBuyAutoUpgrade = Gold.CombineLatest(AutoUpgradeCost, (gold, cost) => gold >= cost)
        .ToReadOnlyReactiveProperty()
        .AddTo(m_Disposables);

    // (자동 수입 Interval과 RP AddTo는 3단계에서)
}
```

- `CombineLatest`(3차시)가 골드·비용의 **최신값 조합**으로 `bool` 스트림을 만들고, `ToReadOnlyReactiveProperty()`로 "현재값을 보유한 읽기 전용 프로퍼티"로 굳힙니다.
- 구매 검증은 제공된 `TryBuy...` 패턴 — **Model이 규칙의 최종 관문**이고, UI 활성화는 어디까지나 편의입니다.

`BindInput`에 업그레이드 구매를 추가:

```csharp
m_ClickUpgradeButton.OnClickAsObservable()
    .Subscribe(_ =>
    {
        if (m_Model.TryBuyClickUpgrade())
            Log($"클릭 강화! 클릭당 +{m_Model.GoldPerClick.Value:N0}");
    })
    .AddTo(this);

m_AutoUpgradeButton.OnClickAsObservable()
    .Subscribe(_ =>
    {
        if (m_Model.TryBuyAutoUpgrade())
            Log($"도우미 고용! 초당 +{m_Model.GoldPerSecond.Value:N0}");
    })
    .AddTo(this);
```

`BindView`에 버튼 활성화·비용 표시를 추가:

```csharp
m_Model.CanBuyClickUpgrade.Subscribe(can => m_ClickUpgradeButton.interactable = can).AddTo(this);
m_Model.CanBuyAutoUpgrade.Subscribe(can => m_AutoUpgradeButton.interactable = can).AddTo(this);

m_Model.ClickUpgradeCost.Subscribe(cost => m_ClickUpgradeLabel.text = $"클릭 강화\n비용 {cost:N0}").AddTo(this);
m_Model.AutoUpgradeCost.Subscribe(cost => m_AutoUpgradeLabel.text = $"도우미 고용\n비용 {cost:N0}").AddTo(this);
```

> 📌 `Log(...)`는 스켈레톤에 이미 있는 헬퍼(이벤트 로그 5줄 유지)입니다 — 새로 만들 필요 없음.

**실행·관찰**: 골드 10을 모으면 클릭 강화 버튼이 **저절로** 켜지고, 사면 꺼졌다가 다시 모이면 켜진다.

> 📌 **던질 질문**: "버튼을 켠 게 누구죠?" (아무도 — 선언했을 뿐). 활성화 갱신 코드를 어디에도 "호출"하지 않았다는 점을 음미.

### 3단계. 자동 수입 — Interval과 Model의 수명 (~12분)

> 🔧 **오늘 쓸 부품**: 02차시 `Observable.Interval`, 04차시 `CompositeDisposable`.

도우미를 고용하면 1초마다 골드가 자동으로 들어옵니다. **Model 안에서 시간이 흐르기 시작**하므로 04차시의 구독 수명 관리가 실전으로 등장합니다. 생성자를 완성합니다:

```csharp
public MyClickerModel()
{
    CanBuyClickUpgrade = Gold.CombineLatest(ClickUpgradeCost, (gold, cost) => gold >= cost)
        .ToReadOnlyReactiveProperty()
        .AddTo(m_Disposables);

    CanBuyAutoUpgrade = Gold.CombineLatest(AutoUpgradeCost, (gold, cost) => gold >= cost)
        .ToReadOnlyReactiveProperty()
        .AddTo(m_Disposables);

    // 자동 수입: 1초마다 GoldPerSecond만큼 (도우미를 사기 전에는 0이라 변화 없음)
    Observable.Interval(TimeSpan.FromSeconds(1))
        .Where(_ => GoldPerSecond.Value > 0)
        .Subscribe(_ => Gold.Value += GoldPerSecond.Value)
        .AddTo(m_Disposables);

    // ReactiveProperty들도 전부 그릇에 — Dispose 한 줄로 정리되게
    Gold.AddTo(m_Disposables);
    GoldPerClick.AddTo(m_Disposables);
    GoldPerSecond.AddTo(m_Disposables);
    ClickUpgradeCost.AddTo(m_Disposables);
    AutoUpgradeCost.AddTo(m_Disposables);
    IsFever.AddTo(m_Disposables);
}
```

`BindView`에 스탯 표시를 추가 — 두 값을 합쳐 한 줄로:

```csharp
m_Model.GoldPerClick
    .CombineLatest(m_Model.GoldPerSecond, (perClick, perSec) => $"클릭당 +{perClick:N0} · 초당 +{perSec:N0}")
    .Subscribe(text => m_StatsText.text = text)
    .AddTo(this);
```

> ❗ **여기가 04차시의 실전입니다.** Model은 MonoBehaviour가 아니라 `AddTo(this)`를 못 씁니다 → 순수 C# 클래스의 표준인 `CompositeDisposable`(`m_Disposables`, 스켈레톤에 선언돼 있음)에 모든 구독·발행기를 담고, 제공된 `Dispose()`가 `m_Disposables.Dispose()` 한 줄로 정리합니다. Presenter의 `OnDestroy → m_Model.Dispose()` 사슬은 스켈레톤에 이미 있습니다. **이걸 빠뜨리면 씬을 나가도 Interval이 영원히 도는** 교과서적 누수가 됩니다 — 6단계에서 Tracker로 직접 확인합니다.

**실행·관찰**: 도우미를 사면 가만히 있어도 골드가 1초마다 오른다.

> 📌 1단계 복선 회수: 자동 수입이 들어와도 **골드 표시 코드는 1단계 그대로**다. 출처가 늘어도 바인딩은 그대로.

> 📌 **중간 Play 체크포인트(1차시 끝)**: 클릭·업그레이드·자동 수입 세 가지가 동작하면 1차시 합격. 여기서 컴파일이 깨졌으면 다음 차시 진도가 막히므로, `Unity_GetConsoleLogs`로 에러 0을 전원 확인하고 마무리하세요.

## 2차시 — 4~6단계

### 4단계. 콤보 — Merge + Scan 누적기 (~18분, 최고 난도)

> 🔧 **오늘 쓸 부품**: 03차시 `Share`·`Select`·`Merge`·`Debounce`·`Scan`·`Where`.

빠르게 연속 클릭하면 콤보가 쌓이고, 0.8초 쉬면 끊기고, 10콤보마다 보너스. **순수하게 연산자 합성만으로** 만듭니다. 콤보는 "입력의 리듬에 대한 해석"이므로 **Presenter 소관**(`BindInput`).

**마블을 먼저, 코드를 나중에.** 칠판에 다섯 줄을 그리고 나서 코드를 칩니다:

```
clicks:        --o-o-o------------o-->
Select(1):     --1-1-1------------1-->
Debounce(0):   ------------0--------->   (0.8초 잠잠 후)
Merge:         --1-1-1-----0------1-->
Scan:          --1-2-3-----0------1-->   ← 콤보 카운터 완성
```

`BindInput`에서 1단계의 클릭 한 줄을 **다음으로 교체**합니다(스켈레톤엔 `ComboResetDelay = 800ms`, `ComboBonusUnit = 10` 상수가 이미 선언돼 있음):

```csharp
// 클릭 스트림은 본체·콤보·보너스 세 곳에서 쓰므로 Share로 구독을 공유
var clicks = m_ClickButton.OnClickAsObservable().Share();

clicks.Subscribe(_ => m_Model.Click()).AddTo(this);

// 콤보 — 클릭은 +1, 0.8초 잠잠해지면(Debounce) 0으로 리셋. Merge+Scan 누적기
var combo = clicks
    .Select(_ => 1)
    .Merge(clicks.Debounce(ComboResetDelay).Select(_ => 0))
    .Scan(0, (acc, x) => x == 0 ? 0 : acc + 1)
    .Share();

combo.Subscribe(c => m_ComboText.text = c >= 2 ? $"{c} 콤보!" : string.Empty).AddTo(this);

// 콤보 보너스 — 10·20·30...에 도달하는 순간 보너스 골드 (Model.AddBonus는 제공됨)
combo
    .Where(c => c > 0 && c % ComboBonusUnit == 0)
    .Subscribe(c =>
    {
        long bonus = c * m_Model.GoldPerClick.Value;
        m_Model.AddBonus(bonus);
        Log($"{c}콤보 달성! 보너스 +{bonus:N0}");
    })
    .AddTo(this);
```

> 📌 **던질 질문**: "Scan의 누적값을 0으로 만드는 건 누구죠?" (Debounce가 보낸 0 — 클릭이 0.8초 멈추면). 명령형이라면 "마지막 클릭 시각 저장 + 매 프레임 경과 검사 + 카운터 리셋"이 필요했을 자리입니다.
>
> ⚠️ **함정**: `Share()` 빠뜨림 — 동작은 하지만 버튼 클릭 구독이 세 벌이 됩니다. "두 곳 이상에서 쓰면 Share"(3차시) 회수 기회. `Model.AddBonus`는 스켈레톤에 이미 있는 입금 창구 메서드입니다(새로 만들지 말 것).

**실행·관찰**: 연타하면 "n 콤보!"가 오르고, 10콤보에 보너스 로그가 찍힌다.

### 5단계. 피버 — ThrottleFirst 쿨다운과 Timer (~12분)

> 🔧 **오늘 쓸 부품**: 03차시 `ThrottleFirst`, 02차시 `Observable.Timer`, 05차시 바인딩.

피버 버튼을 누르면 5초간 클릭 골드 2배. 쿨다운 15초는 **`ThrottleFirst`가 게이트**(첫 입력만 통과, 이후 구간 무시). 클릭 2배 반영은 제공된 `Click()`(`IsFever.Value ? 2 : 1`)에 이미 들어 있으니, 우리는 `TryStartFever`와 입력·바인딩만 채웁니다.

먼저 `MyClickerModel.TryStartFever`를 채웁니다:

```csharp
public bool TryStartFever()
{
    if (IsFever.Value)
        return false;
    IsFever.Value = true;
    // FeverDuration 뒤 자동 종료 — Timer는 일회성 스트림 (2차시)
    Observable.Timer(FeverDuration).Subscribe(_ => IsFever.Value = false).AddTo(m_Disposables);
    return true;
}
```

`BindInput`에 피버 입력을 추가(`FeverCooldown = 15초` 상수는 스켈레톤에 선언돼 있음):

```csharp
m_FeverButton.OnClickAsObservable()
    .ThrottleFirst(FeverCooldown)            // 15초 쿨다운 게이트 — 첫 클릭만 통과
    .Subscribe(_ =>
    {
        if (m_Model.TryStartFever())
            Log($"피버 시작! {m_Model.FeverDuration.TotalSeconds:0}초간 클릭 2배");
    })
    .AddTo(this);
```

`BindView`에 피버 상태 바인딩을 추가:

```csharp
m_Model.IsFever.Subscribe(isFever =>
    {
        m_FeverStateText.text = isFever ? "피버 중! 클릭 2배" : "피버 (15초 쿨다운)";
        if (m_ClickButtonImage != null)
            m_ClickButtonImage.color = isFever ? new Color(1f, 0.6f, 0.2f) : Color.white;
    })
    .AddTo(this);
```

> 📌 **여기서 멈추고 짚기**: 쿨다운을 위해 시각 저장 변수도, 코루틴도 안 만들었습니다. "첫 입력만 통과시키고 15초 무시"라는 **정책이 연산자 한 개**(`ThrottleFirst`)입니다. 5초 뒤 자동 종료도 `Timer` 하나 — `IsFever` 바인딩(문구+버튼 색)은 바인딩 3패턴의 복습.

**실행·관찰**: 피버를 누르면 클릭 버튼이 주황색이 되고 클릭당 골드가 2배, 5초 뒤 원상복구. 15초 안에 다시 눌러도 무시된다.

### 6단계. 마무리 점검 — 누수 0 확인과 회고 (~15분)

이제 `BindInput`·`BindView`·`MyClickerModel` 생성자·`TryStartFever`가 모두 채워졌습니다. 모범답안(`ClickerGame/`)과 같은 동작이 되는지 확인하고 마무리합니다.

#### ObservableTracker로 검증 (04차시 실전)

1. `Window > Observable Tracker`를 연다(`EnableTracking`이 켜진 상태에서 Play — 1부 데모 씬을 거쳐 왔거나, 직접 켜 둠).
2. 학생 클리커 씬에서 실컷 플레이 → `MyClickerPresenter` 오브젝트를 **삭제**(또는 씬을 비움).
3. Tracker 목록에서 클리커 관련 구독이 **전부 사라지면 합격.** 남아 있다면 `AddTo(this)`/`AddTo(m_Disposables)`가 빠진 곳 — StackTrace로 추적.

> 💡 **강력 추천 — 고의 누수 실험**: `MyClickerModel` 생성자의 Interval 구독에서 `.AddTo(m_Disposables)`를 지우고 같은 실험 → 씬을 나가도 Interval이 목록에 남습니다. **이 한 번의 목격이 백 마디 경고보다 낫습니다.** (확인 후 `.AddTo` 복구.)

#### 회고 — 어디에 무엇을 썼나

빈 표를 나눠 주고 5분 동안 학생이 채우게 합니다.

| 코드 위치           | 도구                                | "선언한 의도"                               |
| ------------------- | ----------------------------------- | ------------------------------------------- |
| `Gold` 등 상태      | `ReactiveProperty`                  | 값이 바뀌면 알아서 알려라                   |
| 골드/비용/스탯 표시 | `Subscribe` 바인딩                  | 이 값은 항상 이 텍스트와 같게               |
| 구매 버튼 활성화    | `CombineLatest`+`ToReadOnlyRP`      | 골드 ≥ 비용일 때만 살 수 있다               |
| 자동 수입           | `Interval`+`Where`                  | 1초마다, 도우미가 있으면 지급               |
| 콤보                | `Merge`+`Scan`+`Debounce`           | 클릭은 +1, 0.8초 쉬면 리셋                  |
| 콤보 보너스         | `Where(c % 10 == 0)`                | 10단위 도달 순간만                          |
| 피버 쿨다운/지속    | `ThrottleFirst`+`Timer`             | 첫 입력만 통과, 5초 뒤 종료                 |

> 마지막 멘트: "`Update()` 0개, 상태 플래그 0개, 코루틴 0개로 게임 하나가 완성됐습니다. 동시에 — 이 게임의 **모든 로직이 push가 자연스러운 것들**(입력·시간·값 변화)이었습니다. 데미지 공식 계산이나 인벤토리 정렬까지 Observable로 만들 이유는 없습니다."(05차시 안티패턴 회수)

## 예상 Q&A

> ❓ **콤보를 Model에 두면 안 되나요?**
> 둘 다 가능합니다. 이 강의는 "콤보는 **입력의 리듬**에 대한 해석"이라 Presenter에 뒀습니다(Model은 입력 장치를 모르는 게 깔끔). 콤보가 게임 규칙(저장돼야 하는 상태)이 되면 Model로 옮기는 게 맞음 — 좋은 토론 주제.

> ❓ **TryStartFever의 Timer 구독이 CompositeDisposable에 계속 쌓이지 않나요?**
> 쌓입니다 — 피버 한 번에 일회성 Timer 구독 하나씩. Timer는 완료되는 스트림이라 실질 누수는 아니고 그릇에 빈 껍데기가 남는 정도(쿨다운 15초 게이트로 횟수도 제한됨). "거슬리면 SerialDisposable류로 교체"는 심화 과제로.

> ❓ **골드를 long으로 한 이유는?**
> 클리커 장르는 수치 인플레가 심해 int(약 21억)가 실제로 넘칩니다. 자릿수 표기(`N0`)와 함께 장르 관습.

> ❓ **저장/로드는 어떻게 하나요?**
> Model의 ReactiveProperty 값들을 직렬화하면 됩니다(저장은 `.Value` 읽기, 로드는 `.Value` 대입 — 대입 순간 UI가 자동 갱신되는 것도 바인딩의 이점). 이 강의 범위 밖이라 과제로.

> ❓ **`Update()`에서 매 프레임 검사하던 콤보/쿨다운이 R3에서는 어떻게 바뀌나요?** (전환형)
> 콤보는 `Merge`+`Scan`+`Debounce`로, 쿨다운은 `ThrottleFirst`로 **선언 한 줄**이 됩니다 — "마지막 입력 시각 저장 + 매 프레임 경과 비교 + 플래그 리셋"이라는 명령형 보일러플레이트가 사라집니다. ('기존 방식 → R3' 한눈 매핑표는 [00. 강사 사전 학습 가이드](00.%20강사%20사전%20학습%20가이드.md))

## 이해도 퀴즈

1. Model을 순수 C# 클래스로 두는 이점 두 가지를 쓰시오.
2. 콤보 스트림에서 `Debounce`와 `Scan`이 각각 맡는 역할을 쓰시오.
3. Presenter가 파괴될 때 자동 수입 Interval이 멈추기까지의 호출 사슬을 쓰시오.
4. 완성된 클리커에서 "리액티브로 만들 필요가 없었던 것"의 예를 들고 이유를 쓰시오.

> ✅ **해설**
>
> 1. ①UI 없이 규칙을 단위 테스트 가능 ②UI 교체에도 Model 불변(관심사 분리).
> 2. Debounce — 클릭이 0.8초 멈추면 리셋 신호(0)를 발행. Scan — +1/0 신호를 누적해 콤보 수를 유지(0이 오면 리셋).
> 3. `OnDestroy()` → `m_Model.Dispose()` → `m_Disposables.Dispose()` → Interval 구독 해제.
> 4. 예: `TryBuyClickUpgrade`의 비용 계산 — 호출 시점에 한 번 계산하면 끝나는 pull성 로직이라 일반 메서드가 자연스러움(pull을 push로 강제하지 않기).

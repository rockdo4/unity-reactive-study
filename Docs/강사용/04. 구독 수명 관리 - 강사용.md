# 04. 구독 수명 관리 — 강사용

> 학생 교안: [04. 구독 수명 관리](../교안/04.%20구독%20수명%20관리.md)
> 데모: 기초 데모 씬 패널 3 · `Assets/Scripts/ReactiveBasics/LifetimeDemo.cs`

## 1. 강사 사전 학습 · 리마인드

- **이 차시의 한 문장** — "구독은 IDisposable이고, **Dispose를 잊으면 새는 것은 리액티브도 마찬가지**다. 다만 잊기 어렵게 만드는 표준 도구(AddTo·CompositeDisposable·Tracker)가 있다."
- **정직하게 가르칠 것** — "AddTo만 붙이면 누수 끝"이 아님. "`-=` 잊기"가 "`AddTo` 잊기"로 바뀐 것뿐. 그래도 가치가 있는 이유: ①체인 맨 끝 한 줄이라 리뷰에서 눈에 띔 ②전 코드베이스에서 패턴이 같음 ③Tracker로 검증 가능. 이 균형 잡힌 화법이 신뢰를 만듦.
- **Tracker 시연을 미리 연습** — 수업 효과가 가장 큰 장면이 "누수 만들기 → Tracker에 잡힘 → 프로브 파괴 → AddTo된 것만 사라짐"의 대비임. 데모 패널 3의 버튼 순서를 손에 익히고, `Window > Observable Tracker` 윈도우도 한 번 열어 둘 것.
- **CompositeDisposable의 위치** — MonoBehaviour에서는 AddTo(this)로 충분한 경우가 많음. CompositeDisposable이 꼭 필요한 곳은 ①순수 C# 클래스(2부 ClickerModel — 예고) ②구독을 라운드마다 갈아 끼우는 경우(`Clear`).
- **발행기도 Dispose 대상** — Subject/ReactiveProperty를 들고 있는 쪽(소유자)이 OnDestroy/Dispose에서 정리. 구독자 정리와 발행기 정리는 별개 책임임을 구분해 말할 것.

## 2. 수업 진행

### 도입 (~5분)

> 💡 1장의 사고 코드(`EveryUpdate` 구독에 AddTo 없음)를 띄우고 — "이 EnemyHud가 파괴되면 이 구독은 어떻게 될까요?" 답(영원히 산다)을 학생이 말하게 유도. "오늘은 이걸 **구조적으로** 막고, **눈으로 확인**하는 법을 배웁니다."

### 2장. 구독 = IDisposable (~5분)

- 2차시 Interval 토글(수동 Dispose)을 회수 — "이미 한 번 해 봤다."
- 칠판: `+= → Subscribe · -= → Dispose`. "모든 수명 도구는 이 한 줄 위에 있다."

### 3장. AddTo (~10분, 메인)

- ScoreViewer 예제 라이브 코딩. **`OnDestroy`에 해제 코드가 한 줄도 없음**을 강조하며 멈추기.
- 1장의 사고 코드에 `.AddTo(this)` 한 줄을 더해 "수리 완료" 보여 주기.
- **⚠️ 함정**: "AddTo 만능론" — 즉시 교정. "빠뜨리면 똑같이 샌다. 다만 빠뜨린 게 잘 보인다."
- **던질 질문**: "팀 코드 리뷰에서 `Subscribe`를 보면 뭘 확인해야 할까요?" (답: 끝에 AddTo 또는 명시적 보관이 있는가)

### 4장. CompositeDisposable (~5분)

- 그릇 비유. `Clear` vs `Dispose` 차이. "MonoBehaviour가 아닌 클래스의 표준 — 2부 Model에서 실전."

### 5~7장. 따라하기 — 데모 패널 3 + Tracker (~20분, 클라이맥스)

권장 시나리오(이 순서대로):

1. `활성 구독 조회` — 평상시 목록 확인 (데모 자체의 구독들이 보임).
2. `구독 추가` ×3 → 조회 — 카운트 증가 확인.
3. `모두 해제` → 조회 — CompositeDisposable.Clear 효과.
4. `누수 만들기` ×2 → 조회 — **해제 수단이 없는 구독**이 잡힘. 코드에서 "변수도 AddTo도 없음"을 확인.
5. `프로브 파괴` → 조회 — AddTo(this)로 묶인 프로브 구독**만** 사라짐. 누수 구독은 여전히 목록에. **이 대비가 오늘의 그림.**
6. `Window > Observable Tracker` 윈도우로 같은 내용을 GUI로 — StackTrace로 생성 지점 추적까지.

- **⚠️ 함정**: Tracker가 비어 보임 — `EnableTracking`이 데모의 `Awake`에서 켜지므로 **씬 Play 후**에 만들어진 구독만 잡힘. 윈도우를 먼저 열어 두고 Play할 것.
- **⚠️ 함정**: "추적 켜 두면 안 되나요?" — 오버헤드가 있어 개발·디버그 전용. 출시 빌드에서는 끔.

### 정리 (~5분)

- 누수 패턴 도감 표 훑기 + 점검 루틴(켜기→씬 출입→목록→StackTrace) 복기.

## 3. 예상 Q&A

> ❓ **AddTo(this)는 어떻게 파괴를 아나요?**
> 컴포넌트/GameObject의 파괴 시점에 Dispose가 호출되도록 R3.Unity가 연결함(내부적으로 destroyCancellationToken 등 유니티 수명 신호 활용). 구현 디테일보다 "파괴 시점에 자동 Dispose"라는 계약이 중요.

> ❓ **씬이 통째로 내려가면 어차피 다 정리되는 것 아닌가요?**
> 씬 오브젝트가 파괴돼도 **구독 자체는 별개**임. 전역 스트림(EveryUpdate·Interval)이나 static 발행자에 걸린 구독은 씬과 무관하게 살아남고, 파괴된 오브젝트를 만지는 순간 예외. 그래서 "발행자 수명 ≠ 내 수명"이면 반드시 묶어야 함.

> ❓ **DisposableBag/CreateBuilder는 뭔가요?**
> CompositeDisposable의 저할당 변형(R3 README). 동작 개념은 같으니 기본기는 AddTo/CompositeDisposable로 충분하고, 성능 민감 코드에서 찾아 쓰면 된다고 안내.

> ❓ **`OnDestroy`에서 일일이 `-=` 하던 정리는 R3에서 어떻게 바뀌나요?** (전환형)
> 구독마다 `.AddTo(this)`를 붙이면 `OnDestroy` 시점에 자동 Dispose — `-=`를 손으로 적던 보일러플레이트가 사라짐. 구독이 여럿이면 `CompositeDisposable`에 모아 한 번에 Dispose. "수동 -= → AddTo/CompositeDisposable"가 이 차시의 핵심 전환. ('기존 방식 → R3' 한눈 매핑표는 [운영 가이드 1장](00.%20강의%20운영%20가이드.md))

## 4. 이해도 퀴즈

1. `Observable.EveryUpdate().Subscribe(...)`에 AddTo가 없을 때 일어나는 일을 두 가지 쓰시오.
2. `CompositeDisposable`의 `Clear()`와 `Dispose()`의 차이를 쓰시오.
3. 순수 C# 클래스(MonoBehaviour 아님)에서 구독을 정리하는 표준 패턴을 코드 골격으로 쓰시오.
4. ObservableTracker로 누수를 찾는 4단계 루틴을 쓰시오.

> ✅ **해설**
>
> 1. ① 구독이 영원히 살아 메모리(콜백+캡처 참조) 누수 ② 콜백이 파괴된 오브젝트를 만지면 `MissingReferenceException`.
> 2. Clear는 담긴 구독만 해제하고 그릇은 재사용 가능. Dispose는 전부 해제 후 그릇도 폐기(이후 AddTo되는 구독은 즉시 해제됨).
> 3. `CompositeDisposable` 필드에 `.AddTo(m_Disposables)`로 모으고, 클래스의 `Dispose()`에서 `m_Disposables.Dispose()`. 소유자(예: Presenter의 OnDestroy)가 Dispose를 호출.
> 4. ①EnableTracking 켜기 ②의심 씬 출입 ③활성 구독 목록 확인 ④남은 구독의 StackTrace로 생성 지점을 찾아 AddTo 보강.

# 06. 따라하기 — 리액티브 클리커 — 강사용

> 학생 교안: [06. 따라하기 — 리액티브 클리커 만들기](../교안/06.%20따라하기%20—%20리액티브%20클리커%20만들기.md)
> 완성본: `Assets/Scripts/ClickerGame/` · 씬 `Tools > Reactive Study > 2부 클리커 씬 생성`
> 권장 2차시: ①(0~3단계) ②(4~6단계)

## 1. 강사 사전 학습 · 리마인드

- **반드시 한 번 완주하고 들어갈 것** — 운영 가이드 로드맵 7번. 본인이 직접 만들며 막힌 지점을 메모해 두는 것이 이 차시 준비의 90%임.
- **이 수업의 정체** — 새 내용은 없음. 1부의 도구를 **조립**하는 시간. 학생이 "이거 배웠던 건데"를 다섯 번 느끼게 하는 것이 성공 조건. 각 단계 시작 때 "오늘 쓸 부품: ○차시의 ○○"를 선언하고 들어갈 것.
- **Model/Presenter 분리의 명분을 한 문장으로** — "Model이 UI를 모르면, 규칙을 테스트할 수 있고 UI를 갈아치워도 안 깨진다." 이상 깊은 아키텍처 논쟁은 피할 것(MVP 강의가 아님).
- **콤보 스트림(4단계)이 최고 난도** — `Select(1) → Merge(Debounce→0) → Scan(리셋 누적)`. 수업 전에 마블 다이어그램을 종이에 **세 번** 그려 볼 것. 학생 질문이 여기에 몰림.
- **진도 차 대비** — 단계마다 "여기까지의 전체 코드"가 교안에 있음. 뒤처진 학생은 완성본(`ClickerGame/`)과 비교시키되, 복붙으로 건너뛰게 하지는 말 것(타이핑이 학습임).

## 2. 수업 진행

### 1차시 — 0~3단계

**도입 (~5분)**

> 💡 완성본 씬을 먼저 Play로 보여 줌(클릭→골드, 업그레이드, 콤보, 피버까지 1분 시연). "오늘과 다음 시간에 이걸 처음부터 만듭니다. `Update()` 0개, 코루틴 0개로."

**0단계. 준비 (~7분)** — 씬 생성 → 완성 Presenter 비활성화 → `MyClickerPresenter` 오브젝트 + 스크립트 2개 생성. Model/Presenter 역할 표를 칠판에.

- **⚠️ 함정**: 완성 `ClickerPresenter`를 비활성화하지 않고 자기 것을 추가하는 학생 — 버튼이 두 번 반응함. 증상(골드가 2씩 오름)이 보이면 이걸 의심.

**1단계. 골드와 클릭 (~15분)** — 최소 Model + 클릭→Click() + Gold→텍스트 바인딩.

- 멈출 곳: "골드 텍스트 갱신 코드가 클릭 핸들러에 **없다**." 바인딩 선언 한 번으로 출처 무관하게 표시가 맞는다는 것 — 3단계(자동 수입)에서 회수됨.
- 인스펙터 참조 연결을 프로젝터로 천천히(여기서 의외로 많이 막힘).

**2단계. 업그레이드 (~12분)** — `CombineLatest` → `ToReadOnlyReactiveProperty` → `interactable` 바인딩 + `TryBuy` 패턴.

- 멈출 곳: 골드를 모아 버튼이 **저절로** 켜지는 순간. "누가 켰죠?" (아무도 — 선언)
- `TryBuy`가 규칙의 최종 관문(버튼 활성화는 편의)이라는 방어적 설계 한 마디.

**3단계. 자동 수입 (~11분)** — `Interval` + **CompositeDisposable 도입**(4차시 실전).

- 멈출 곳: "Model은 MonoBehaviour가 아니라 AddTo(this)를 못 쓴다 → 그릇(CompositeDisposable)이 표준." Presenter의 OnDestroy→Model.Dispose 사슬 확인.
- 1단계의 복선 회수: 자동 수입이 들어와도 골드 표시 코드는 그대로.

### 2차시 — 4~6단계

**4단계. 콤보 (~18분, 최고 난도)** — `Share` → `Merge`+`Scan` 누적기 → 보너스.

- **마블을 먼저, 코드를 나중에.** 칠판에 클릭/Select(1)/Debounce(0)/Merge/Scan 다섯 줄을 그리고 나서 코드를 침. 교안의 마블 그림을 그대로 판서해도 됨.
- **던질 질문**: "Scan의 누적값을 0으로 만드는 건 누구죠?" (Debounce가 보낸 0 — 클릭이 0.8초 멈추면)
- **⚠️ 함정**: `Share()` 빠뜨림 — 동작은 하지만 버튼 클릭 구독이 세 벌. "두 곳 이상에서 쓰면 Share"(3차시) 회수 기회.

**5단계. 피버 (~12분)** — `ThrottleFirst` 게이트 + `Timer` 종료.

- 멈출 곳: "쿨다운 변수도 코루틴도 없다 — 정책이 연산자 한 개."
- IsFever 바인딩(문구+버튼 색)은 바인딩 3패턴의 복습.

**6단계. Tracker 검증 + 회고 (~15분)**

- 클리커 플레이 → Presenter 오브젝트 삭제 → `Window > Observable Tracker`에서 관련 구독 소멸 확인.
- **강력 추천 — 고의 누수 실험**: Model의 Interval에서 `.AddTo(m_Disposables)`를 지우고 같은 실험 → 씬을 나가도 Interval이 목록에 남음. "이 한 번의 목격이 백 마디 경고보다 낫다"(교안 문구 그대로).
- 회고 표(기능→도구→의도)를 학생이 채우게 함 — 빈 표를 나눠 주고 5분.
- 마지막 멘트: "이 게임의 로직이 전부 push가 자연스러운 것(입력·시간·값 변화)이었다. 데미지 공식 계산까지 Observable로 만들 이유는 없다."(안티패턴 회수)

## 3. 예상 Q&A

> ❓ **콤보를 Model에 두면 안 되나요?**
> 둘 다 가능함. 이 강의는 "콤보는 **입력의 리듬**에 대한 해석"이라 Presenter에 둠(Model은 입력 장치를 모르는 게 깔끔). 콤보가 게임 규칙(저장돼야 하는 상태)이 되면 Model로 옮기는 게 맞음 — 좋은 토론 주제.

> ❓ **TryStartFever의 Timer 구독이 CompositeDisposable에 계속 쌓이지 않나요?**
> 쌓임 — 피버 한 번에 일회성 Timer 구독 하나씩. Timer는 완료되는 스트림이라 실질 누수는 아니고 그릇에 빈 껍데기가 남는 정도(쿨다운 15초 게이트로 횟수도 제한됨). "거슬리면 SerialDisposable류로 교체"는 심화 과제로.

> ❓ **골드를 long으로 한 이유는?**
> 클리커 장르는 수치 인플레가 심해 int(약 21억)가 실제로 넘침. 자릿수 표기(`N0`)와 함께 장르 관습.

> ❓ **저장/로드는 어떻게 하나요?**
> Model의 ReactiveProperty 값들을 직렬화하면 됨(저장은 `.Value` 읽기, 로드는 `.Value` 대입 — 대입 순간 UI가 자동 갱신되는 것도 바인딩의 이점). 이 강의 범위 밖이라 과제로.

## 4. 이해도 퀴즈

1. Model을 순수 C# 클래스로 두는 이점 두 가지를 쓰시오.
2. 콤보 스트림에서 `Debounce`와 `Scan`이 각각 맡는 역할을 쓰시오.
3. Presenter가 파괴될 때 자동 수입 Interval이 멈추기까지의 호출 사슬을 쓰시오.
4. 완성된 클리커에서 "리액티브로 만들 필요가 없었던 것"의 예를 들고 이유를 쓰시오.

> ✅ **해설**
>
> 1. ①UI 없이 규칙을 단위 테스트 가능 ②UI 교체에도 Model 불변(관심사 분리).
> 2. Debounce — 클릭이 0.8초 멈추면 리셋 신호(0)를 발행. Scan — +1/0 신호를 누적해 콤보 수를 유지(0이 오면 리셋).
> 3. `OnDestroy()` → `m_Model.Dispose()` → `m_Disposables.Dispose()` → Interval 구독 해제.
> 4. 예: `TryBuyClickUpgrade`의 비용 계산 — 호출 시점에 한 번 계산하면 끝나는 pull성 로직이라 일반 메서드가 자연스러움(pull을 push로 강제하지 않기).

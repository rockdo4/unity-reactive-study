## 0. 도입 (7분)

**진입점 멘트**: "체력 바 갱신 코드를 어디에 두시겠어요?"

- 폴링(`Update`) 답이 나오면 → "바뀌지 않은 99%의 프레임에도 검사하네요."
- 이벤트 답이 나오면 → "`OnDestroy`에서 `-=` 잊으면요?"

양쪽 다 한계를 보여 주고 시작합니다. 그다음 **학습 목표 4개를 구두로 제시** — ① 리액티브=옵저버의 확장임을 설명 ② 마블 다이어그램 읽기 ③ push/pull 구분 ④ R3 계보.

> 💡 이 차시는 코드를 거의 치지 않습니다. "오늘 본 코드는 전부 **읽기만** 하고, 직접 만지는 건 02차시부터"라고 미리 못 박으면 학생이 받아쓰기 모드에서 듣기 모드로 전환합니다.

## 1. 게임 코드는 "반응"으로 가득하다 (10분)

**핵심 메시지**

- 게임 로직의 대부분은 "무언가 일어나면 → 무언가 한다"임. (체력 바뀌면→체력 바 갱신, 버튼 눌리면→스킬, 적 죽으면→점수·사운드·퀘스트)
- 고전적 두 방법을 대비: **폴링(pull)** vs **이벤트(push)**. 이벤트가 깔끔하지만 규모가 커지면 두 한계가 계속 걸림.

**판서**

```
반응 = 무언가 일어나면 → 무언가 한다
한계 ① 구독 해제 누락 (-= 잊기 쉬움)
한계 ② 이벤트 하나로 표현 안 됨 (더블클릭·연타·검색)
```

**보여줄 코드** — 폴링 vs 이벤트 한 쌍(교안 1절):

```csharp
// 폴링: 매 프레임 "바뀌었나?"를 묻는다 — 바뀌지 않은 99%의 프레임에도
private void Update()
{
    if (m_Hp != m_LastHp) { m_HpText.text = $"HP : {m_Hp}"; m_LastHp = m_Hp; }
}

// 이벤트: 바뀐 쪽이 알려 준다
public event Action<int> HpChanged;
m_Player.HpChanged += hp => m_HpText.text = $"HP : {hp}";
```

**두 한계를 명확히**

- **구독 해제 누락** — `+=`는 `-=`로 손수 해제해야 함. `OnDestroy`에서 빠뜨리면 메모리 누수·`MissingReferenceException`. "잊으면 안 되는데 잊기 쉬운" 수동 작업.
- **합성 불가** — "두 번 빠르게(더블클릭)", "0.5초 안에 세 번(연타)", "타이핑 멈춘 뒤(검색)"는 `event` 하나로 안 됨. 결국 `Update`·코루틴에 플래그·타이머를 직접 쌓음.

**보여줄 코드** — 명령형 더블클릭(읽기만, 부담을 눈으로):

```csharp
// 명령형: 더블클릭 하나에 코루틴 + 상태 플래그 + 타이머 + 조건문이 필요하다
private IEnumerator DetectDoubleClick()
{
    m_Running = true;
    int clickCount = 1;
    float timeLeft = 0.3f;
    while (timeLeft > 0f && clickCount < 2)
    {
        yield return null;
        if (Mouse.current.leftButton.wasPressedThisFrame) clickCount++;
        timeLeft -= Time.deltaTime;
    }
    if (clickCount == 2) Debug.Log("더블 클릭");
    m_Running = false;
}
```

**던질 질문**: "더블클릭 하나에 왜 이렇게 많은 게 필요할까요?"

→ 핵심 질문으로 연결: **이벤트를 "시간 위를 흐르는 데이터 스트림"으로 보고, 구독 수명을 표준으로 관리하며, 연산자로 합성할 수 없을까?** 그 답이 리액티브 프로그래밍.

## 2. 토대: 옵저버 패턴 (8분)

> 옵저버를 이미 배운 반이라면 3분으로 압축. 모르는 반에는 이 절이 사실상 미니 옵저버 수업.

**핵심 메시지**

- **주체(Subject)** — 사건을 일으키는 쪽. 관찰자 명단을 들고 있다가 사건이 나면 전원에게 알림.
- **관찰자(Observer)** — 알림을 받아 반응하는 쪽. 주체에 자신을 **등록(구독)** 해 둠.
- C#은 이 패턴을 `event`로 언어에 내장함.

**보여줄 코드** — `event`가 곧 옵저버임을 보여주는 용도:

```csharp
public class ScoreBoard
{
    public event Action<int> ScoreChanged;   // 관찰자 명단
    public void AddScore(int amount) { m_Score += amount; ScoreChanged?.Invoke(m_Score); }
}

scoreBoard.ScoreChanged += OnScoreChanged;   // 구독
scoreBoard.ScoreChanged -= OnScoreChanged;   // 해제 (잊기 쉬움!)
```

**판서** — "주는 것"과 "못 주는 것"을 표로:

| 옵저버가 주는 것                   | 옵저버가 못 주는 것                         |
| ---------------------------------- | ------------------------------------------- |
| 일대다 알림                        | 해제의 안전망 (수동 `-=`)                   |
| 느슨한 결합 (주체는 관찰자를 모름) | 사건의 합성 (여러 사건·시간 조건 조합 수단) |

## 3. 리액티브 = 옵저버 패턴의 확장 (10분)

**핵심 메시지**

- ReactiveX 공식 정의 인용: "Rx는 옵저버 패턴을 **데이터/이벤트의 시퀀스**를 지원하도록 **확장**하고, 그 시퀀스를 **선언적으로 합성하는 연산자**를 추가한 것." → **대체가 아니라 일반화/확장**.

**판서** — 칠판에 크게:

```
Observable = 옵저버 패턴(주체-관찰자) + Operator(연산자 합성) + IDisposable(구독 수명)
```

**대응 화살표 4개** (이미 아는 옵저버에 이름표 바꿔 달기):

| 옵저버 패턴 | 리액티브    | 비고                |
| ----------- | ----------- | ------------------- |
| 주체        | Observable  | 관찰당할 수 있는 것 |
| 관찰자      | Observer    | Subscribe로 반응    |
| `+=`        | `Subscribe` | 구독                |
| `-=`        | `Dispose`   | 표준화된 해제       |

**옵저버 위에 더해지는 3가지**

1. **연산자 합성("LINQ to events")** — 이벤트 스트림에 `Where`/`Select`/`Merge`를 직접 적용하고 조합함. **이게 단순 옵저버 대비 핵심 차이.** ← 못 박기.
2. **에러·완료 채널** — 값(`OnNext`)뿐 아니라 에러·완료라는 두 시맨틱 추가.
3. **표준화된 해제(`IDisposable`)** — 구독이 곧 `IDisposable`. `-=` 누락을 구조적으로 다룸.

> ⚠️ **예상되는 함정 질문**: "그럼 event는 버리나요?" → 아니라고 즉답. **확장이지 대체가 아니며**, 안 쓸 때 기준은 5차시에서.

## 4. 스트림으로 생각하기 — 마블 다이어그램 (8분)

> 이 절은 **칠판이 중심**. 슬라이드를 읽지 말고 직접 그리세요.

**핵심 메시지**

- 발상 전환: **이벤트는 한 번의 사건이 아니라, 시간 축 위에 늘어선 데이터의 흐름(스트림)이다.** 유니티 **타임라인**을 떠올리게 하면 직관적.
- 마블 다이어그램 = 가로선(시간) + 구슬(`o`, 발행 값) + 연산자 상자.

**판서 ①** — 클릭 스트림을 그리고, `Where` 상자에 통과시키기:

```
in:   --1--2--3--4--5-->
      +--------------------+
      | Where(x => x%2==0) |
      +--------------------+
out:  -----2-----4----->
```

**판서 ②** — 학생에게 `Select(x => x*10)` 출력을 직접 그리게 시키기:

```
in: --1----2----3-->   →   out: --10---20---30-->
```

**판서 ③** — 스트림의 세 가지 끝 구분(완료 `|` / 에러 `X` / 무한 `-->`):

```
complete: --o--o--o--|     (예: Range(1,3))
error:    --o--o--X        (예: 네트워크 실패)
infinite: --o--o--o--o-->  (예: 매 프레임·클릭)
```

> 💡 연산자 문서가 전부 이 그림이라, **읽는 법**만 익히면 처음 보는 연산자도 그림으로 이해함. 그리는 건 이해 확인용, 읽는 게 필수.

## 5. push vs pull — 무엇이 뒤집혔나 (5분)

**핵심 메시지**

- `IEnumerable`과 `Observable`은 데이터 전달 방향만 서로 반대인, 거울처럼 대응되는 관계. 같은 연산자 어휘가 "메모리의 컬렉션"(pull) 대신 "시간을 두고 도착하는 이벤트"(push)에 적용될 뿐 → 그래서 Rx를 **"LINQ to events"**라고 부름.

**판서** — pull/push 대비 표:

| 구분        | `IEnumerable<T>` (pull)         | `Observable<T>` (push)               |
| ----------- | ------------------------------- | ------------------------------------ |
| 데이터 흐름 | 소비자가 **끌어옴**(`MoveNext`) | 생산자가 **밀어 넣음**(`OnNext`)     |
| 주도권      | 소비자 ("다음 값 줘")           | 생산자 ("값 나왔다, 받아라")         |
| 시간        | 무관 — 원할 때 꺼냄             | 본질 — **언제** 오는지가 정보의 일부 |
| 대표 도구   | LINQ                            | Rx 연산자 (같은 이름!)               |

**사기 진작**: "LINQ를 아세요?" → 아는 반이면 "리액티브의 절반은 이미 끝났다." 게임에서 push가 자연스러운 것들(입력·UI·체력/점수·충돌·네트워크)도 한 줄 언급.

## 6~7. 선언적 합성 미리 보기 + Rx 계보 (5분)

**선언적 합성 — 읽기만**

1장 명령형 더블클릭이 R3에서는 이렇게 됨(문법은 3차시):

```csharp
// 선언형: "클릭이 잠잠해질 때까지 모아서, 2개 이상이면 더블클릭"
clickStream
    .Chunk(clickStream.Debounce(TimeSpan.FromMilliseconds(250)))
    .Where(clicks => clicks.Length >= 2)
    .Subscribe(_ => Debug.Log("더블 클릭"));
```

- 비교 포인트는 줄 수가 아님. **명령형 = "어떻게(how)"**(플래그·타이머·분기), **선언형 = "무엇을(what)"**("모아서, 2개 이상이면"이라는 의도가 그대로 코드). 상태 관리 실수의 여지가 함께 사라짐.
- 멘트: "오늘은 '의도가 그대로 코드'라는 것만. 문법은 3차시에." 공짜는 아니라는 단서(연산자 어휘 학습·중간 디버깅 난이도)도 한 줄.

**Rx 계보** — 가드레일 준수(날짜·버전·점유율 단정 금지):

```
Rx.NET(1세대) → UniRx(2세대, 아카이브) → R3(3세대, 이 강의)
```

- UniRx는 현재 **아카이브(읽기 전용)** 상태이고 README가 직접 "R3를 대신 쓰라"고 안내함.
- R3는 같은 저자 계열(neuecc, Cysharp)의 정식 후속작이며 범용 .NET 라이브러리(Godot·Avalonia·WPF 등 지원). 유니티 통합은 `R3.Unity`가 담당, **Unity 2021.3 이상** 지원.

**던질 질문**: "지금 새 프로젝트를 시작하면 UniRx와 R3 중 뭘 골라야 할까요?"

# 🎓 이론 마무리 (2분)

| 개념            | 한 줄 요약                                                       |
| --------------- | ---------------------------------------------------------------- | ------------- |
| 반응으로 가득   | 게임 코드는 "일어나면→한다" — 폴링·수동 event엔 한계(해제·합성)  |
| 옵저버의 확장   | `Observable = 옵저버 + Operator + IDisposable`                   |
| 스트림으로 보기 | 이벤트는 시간 축 위의 데이터 흐름 — 마블(`                       | `/`X`)로 읽음 |
| push vs pull    | `IEnumerable`(pull) ↔ `Observable`(push), Rx = "LINQ to events" |
| 계보            | Rx.NET → UniRx(아카이브) → R3(이 강의)                           |

**마무리 활동**: 체크리스트 6항목 중 2개를 무작위로 골라 옆 사람에게 설명시키기.

**예상 Q&A**

> ❓ **함수형 프로그래밍이랑 같은 건가요?**
> 친척이지만 다름. 리액티브는 "시간 위의 데이터 흐름"이 중심이고, 함수형의 도구(람다·불변·합성)를 많이 빌려 씀. "함수형 reactive programming(FRP)"이라는 학술 용어와 Rx는 엄밀히는 다르다는 논쟁도 있으나 수업에서는 안 들어가는 게 맞음.

> ❓ **이거 배우면 코루틴은 안 써도 되나요?**
> 아님. 코루틴/async는 "한 줄기 절차"를 시간에 걸쳐 진행하는 데 강하고, 리액티브는 "반복 이벤트의 합성"에 강함. 5차시에서 둘을 잇는 법(await 통합)을 배움.

> ❓ **마블 다이어그램 꼭 그려야 하나요?**
> 연산자 문서가 전부 이 그림이라, 읽을 줄 모르면 공식 문서를 못 봄. 그리는 건 이해 확인용이고, 읽는 게 필수.

> ❓ **bool 플래그를 두고 상태가 바뀔 때마다 UI를 갱신하던 건 R3로 어떻게 바뀌나요?** (전환형)
> "현재값을 들고 있다가 바뀌면 자동 통지"하는 `ReactiveProperty`가 그 자리를 대체함(2차시에서 코드로). 지금은 "플래그 + 수동 갱신 호출"이 "값 보유 스트림 + 자동 구독"으로 바뀐다는 예고만 — 개념 차시이므로 가볍게. ('기존 방식 → R3' 한눈 매핑표는 [00. 강사 사전 학습 가이드](00.%20강사%20사전%20학습%20가이드.md))

## 이해도 퀴즈

1. "리액티브는 옵저버 패턴의 확장"을 `Observable = 옵저버 + Operator + IDisposable` 프레임으로 설명하시오.
2. C# `event` 기반 옵저버의 두 가지 고질병을 쓰시오.
3. `--1--2--3--|` 스트림에 `Where(x => x >= 2)`를 적용한 출력 마블을 그리시오.
4. push와 pull의 차이를 `IEnumerable`/`Observable`로 설명하시오.

> ✅ **해설**
>
> 1. 주체/관찰자 구조는 그대로 두고, 그 위에 연산자 합성(스트림 가공)과 표준 해제(구독=IDisposable)를 더한 것.
> 2. ① `-=` 누락 시 누수·`MissingReferenceException` ② 시간·횟수가 얽힌 사건(더블클릭 등)을 합성할 표준 수단이 없음.
> 3. `-----2--3--|` (1만 걸러지고 완료는 유지).
> 4. `IEnumerable`은 소비자가 `MoveNext`로 끌어오고(pull), `Observable`은 생산자가 `OnNext`로 밀어 넣음(push). 값이 "언제" 오는지가 push에서는 정보의 일부.

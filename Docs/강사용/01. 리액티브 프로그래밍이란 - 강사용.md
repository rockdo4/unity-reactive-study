# 01. 리액티브 프로그래밍이란 — 강사용

> 학생 교안: [01. 리액티브 프로그래밍이란](../교안/01.%20리액티브%20프로그래밍이란.md)
> 선수 지식: C# `event`·델리게이트·람다, `MonoBehaviour` 생명주기

## 1. 강사 사전 학습 · 리마인드

> ℹ️ 이 차시는 코드가 아니라 **프레임**을 파는 시간임. 강사가 흔들리면 안 되는 문장 두 개:
>
> 1. "리액티브 프로그래밍은 **옵저버 패턴의 일반화/확장**이다" (ReactiveX 공식 정의 — 신비화 금지)
> 2. "이벤트를 **시간 위를 흐르는 데이터 스트림**으로 본다" (발상 전환의 전부)

- **옵저버 패턴을 먼저 세우는 이유** — 이 강의는 디자인 패턴 수업의 후속이 아니므로 옵저버를 모르는 학생이 있을 수 있음. 2장에서 주체/관찰자/`event`를 자족적으로 다룸. 옵저버 패턴을 이미 아는 학생에게는 "아는 것의 이름표 바꾸기"라고 안심시키고, 모르는 학생에게는 2장이 사실상 미니 옵저버 수업임.
- **마블 다이어그램은 강사가 손으로 그릴 수 있어야 함** — 가로선(시간), 구슬(값), `|`(완료), `✕`(에러). 수업 전 `Where`/`Select`/`Debounce` 세 개를 종이에 그려 볼 것.
- **push/pull 쌍대성** — `IEnumerable`(소비자가 MoveNext로 끌어옴) vs `Observable`(생산자가 OnNext로 밀어 넣음). LINQ를 아는 학생에게는 "같은 연산자를 시간 축에 적용"이 가장 빠른 지름길.
- **역사 파트 가드레일** — [운영 가이드 3장](00.%20강의%20운영%20가이드.md) 준수. 날짜·버전·점유율 단정 금지.

## 2. 수업 진행 (50분)

### 도입 (~7분)

> 💡 **진입점** — "체력 바 갱신 코드를 어디에 두시겠어요?" 폴링(`Update`) 답이 나오면 "바뀌지 않은 99%의 프레임에도 검사하네요", 이벤트 답이 나오면 "`OnDestroy`에서 `-=` 잊으면요?"로 양쪽 다 한계를 보여 주고 시작.

- 학습 목표 4개 구두 제시: ①리액티브=옵저버의 확장임을 설명 ②마블 다이어그램 읽기 ③push/pull 구분 ④R3 계보.

### 1장. 게임 코드는 반응으로 가득하다 (~10분)

- 폴링/이벤트 코드 대비 → 이벤트의 두 한계(해제 누락·합성 불가)를 명확히.
- 명령형 더블클릭 코드를 같이 읽으며 "코루틴+플래그+타이머+조건문" 부담을 눈으로 확인.
- **칠판**: `반응 = 무언가 일어나면 → 무언가 한다` / 한계 두 줄.
- **던질 질문**: "더블클릭 하나에 왜 이렇게 많은 게 필요할까요?"

### 2장. 옵저버 패턴 (~8분)

- 주체/관찰자 → C# `event` 대응. 옵저버를 이미 배운 반이라면 3분으로 압축.
- 옵저버가 "주는 것"(일대다·느슨한 결합)과 "못 주는 것"(해제 안전망·합성)을 표로.

### 3장. 리액티브 = 옵저버의 확장 (~10분)

- ReactiveX 정의 인용 → 칠판에 크게 `Observable = 옵저버 패턴 + Operator + IDisposable`.
- 대응 화살표 4개: 주체→Observable, 관찰자→Observer, `+=`→`Subscribe`, `-=`→`Dispose`.
- 더해지는 3가지(연산자 합성·에러/완료 채널·표준 해제) — **"진짜 가치는 연산자 합성"** 못 박기.
- **⚠️ 함정**: "그럼 event는 버리나요?" — 아니라고 즉답. 확장이지 대체가 아니며, 안 쓸 때 기준은 5차시에서.

### 4장. 마블 다이어그램 (~8분)

- 칠판에 클릭 스트림 → `Where` 상자 → 출력 스트림을 직접 그리기. 학생에게 `Select(x => x*10)` 출력을 그리게 시키기.
- 완료(`|`)/에러(`✕`)/무한(`▶`) 세 가지 끝을 구분.

### 5장. push vs pull (~5분)

- 쌍대성 표. "LINQ를 아세요?" → 아는 반이면 "리액티브의 절반은 끝났다"로 사기 진작.

### 6~7장. 선언적 합성 미리 보기 + 계보 (~5분)

- R3 더블클릭 코드는 **읽기만** — "문법은 3차시에. 오늘은 '의도가 그대로 코드'라는 것만."
- 계보 화살표: `Rx.NET(1) → UniRx(2, 아카이브) → R3(3, 이 강의)`.
- **던질 질문**: "지금 새 프로젝트를 시작하면 UniRx와 R3 중 뭘 골라야 할까요?"

### 마무리 (~2분)

- 체크리스트 6항목 중 2개를 무작위로 골라 옆 사람에게 설명시키기.

## 3. 예상 Q&A

> ❓ **함수형 프로그래밍이랑 같은 건가요?**
> 친척이지만 다름. 리액티브는 "시간 위의 데이터 흐름"이 중심이고, 함수형의 도구(람다·불변·합성)를 많이 빌려 씀. "함수형 reactive programming(FRP)"이라는 학술 용어와 Rx는 엄밀히는 다르다는 논쟁도 있으나 수업에서는 안 들어가는 게 맞음.

> ❓ **이거 배우면 코루틴은 안 써도 되나요?**
> 아님. 코루틴/async는 "한 줄기 절차"를 시간에 걸쳐 진행하는 데 강하고, 리액티브는 "반복 이벤트의 합성"에 강함. 5차시에서 둘을 잇는 법(await 통합)을 배움.

> ❓ **마블 다이어그램 꼭 그려야 하나요?**
> 연산자 문서가 전부 이 그림이라, 읽을 줄 모르면 공식 문서를 못 봄. 그리는 건 이해 확인용이고, 읽는 게 필수.

## 4. 이해도 퀴즈

1. "리액티브는 옵저버 패턴의 확장"을 `Observable = 옵저버 + Operator + IDisposable` 프레임으로 설명하시오.
2. C# `event` 기반 옵저버의 두 가지 고질병을 쓰시오.
3. `──1──2──3──|` 스트림에 `Where(x => x >= 2)`를 적용한 출력 마블을 그리시오.
4. push와 pull의 차이를 `IEnumerable`/`Observable`로 설명하시오.

> ✅ **해설**
>
> 1. 주체/관찰자 구조는 그대로 두고, 그 위에 연산자 합성(스트림 가공)과 표준 해제(구독=IDisposable)를 더한 것.
> 2. ① `-=` 누락 시 누수·`MissingReferenceException` ② 시간·횟수가 얽힌 사건(더블클릭 등)을 합성할 표준 수단이 없음.
> 3. `─────2──3──|` (1만 걸러지고 완료는 유지).
> 4. `IEnumerable`은 소비자가 `MoveNext`로 끌어오고(pull), `Observable`은 생산자가 `OnNext`로 밀어 넣음(push). 값이 "언제" 오는지가 push에서는 정보의 일부.

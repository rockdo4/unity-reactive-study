## 패널 4 — 바인딩 (~12분, 메인)

**① 값 → 표시 (~3분)**

- **조작**: `피해 -10`/`회복 +10`을 눌러 HP 텍스트가 즉시 갱신되는 것 확인. `MP -5`/`MP +5`도 동일.
- **코드 매칭** (`BindingDemo.cs`): `m_Hp.Subscribe(hp => m_HpText.text = $"HP : {hp}")`. "구독 즉시 현재값이 한 번 흐른다 — 그래서 첫 화면부터 100/50으로 맞아 있다. 폴링 없음."
- **멘트**: "갱신을 누가 호출하나요? 버튼 핸들러는 `m_Hp.Value`만 바꿉니다. 표시는 구독이 알아서 따라옵니다."

**② 파생 상태 — 구간 라벨 (~4분)**

- **조작**: HP를 천천히 깎으며 상태 라벨(안전→주의→위험→쓰러짐)과 색이 **구간이 바뀔 때만** 변하는 것을 관찰. 같은 구간 안에서 10씩 깎을 때는 라벨이 안 바뀜.
- **코드 매칭**: `m_Hp.Select(hp => ...구간...).DistinctUntilChanged().Subscribe(...)`. 색까지 `state switch`로 같이 바뀜.
- **멘트**: "`Select`로 숫자를 의미(구간)로 바꾸고, `DistinctUntilChanged`로 '바뀔 때만' 갱신. 매 프레임 if 검사 대신."

**③ 조건 → 활성화 — CombineLatest (~5분, 하이라이트)**

- **조작**: `MP -5`를 반복해 MP를 20 아래로 — 스킬 버튼이 **저절로** 꺼짐. 다시 `MP +5`로 20 이상 만들면 켜짐. HP를 0으로 만들어도 꺼짐(HP>0 조건).
- **던질 질문**: "끄는 코드를 누가 호출했죠?" → (아무도 — 선언했을 뿐)
- **코드 매칭**: `m_Hp.CombineLatest(m_Mp, (hp, mp) => hp > 0 && mp >= SkillMpCost).DistinctUntilChanged().Subscribe(canUse => m_SkillButton.interactable = canUse)`. "HP·MP 둘 중 아무거나 바뀌면 조건이 재평가되고, 결과가 바뀔 때만 `interactable`에 반영."
- **마무리**: `BindingDemo.cs`를 띄워 3패턴에 ①②③ 번호를 매겨 가며 한 번 더 짚기.

```
값→표시:    m_Hp.Subscribe(...)                               (즉시 현재값)
파생 상태:  m_Hp.Select(구간).DistinctUntilChanged().Subscribe(...)
조건→활성화: m_Hp.CombineLatest(m_Mp, 조건).DistinctUntilChanged().Subscribe(interactable)
```

> 💡 MVP 예고 — "값을 가진 쪽(Model)과 보여 주는 쪽(View)을 클래스로 나누면 MVP. 다음 차시(2부) 클리커가 전부 이 3패턴으로 만들어집니다."

## 패널 5 — 비동기 (~8분, 맛보기)

**① Sequential — 줄 세우기 (~3분)**

- **조작**: `순차 로드`를 빠르게 4번 클릭. 로드 1건당 1.5초이므로, 줄 서서 #1~#4가 순서대로 시작·완료되는 로그가 약 6초에 걸쳐 흐름.
- **코드 매칭** (`AsyncDemo.cs`): `.SubscribeAwait(async (_, ct) => { ... await FakeLoadAsync(ct); ... }, AwaitOperation.Sequential)`. "처리 중 들어온 클릭을 큐에 줄 세워 전부 통지 — 하나도 안 버리고 순서 보장, 대신 뒤로 밀림."

**② Drop — 버리기 (~3분)**

- **조작**: `드롭 로드`를 빠르게 4번 클릭. **1건만** 처리되고 처리 중 들어온 나머지는 무시됨. 처리가 끝난 뒤 다시 누르면 또 1건.
- **코드 매칭**: 같은 골격에 `AwaitOperation.Drop`. "저장/로드 버튼 연타 방지에 딱."
- **시간 축으로 대비**:

```
clicks:      --c1-c2-c3--------c4-->
Sequential:  -->c1-->c2-->c3---->c4   (전부 처리, 뒤로 밀림)
Drop:        -->c1--x--x------->c4    (처리 중 c2·c3 버림)
```

**③ 취소 연쇄 (~2분)**

- **조작**: `순차 로드`로 로드가 진행 중일 때 **Play를 정지**. 예외 없이 조용히 취소됨(Console에 에러 없음).
- **멘트**: "`SubscribeAwait`가 넘겨준 `ct`는 구독 해제(여기선 오브젝트 파괴)에 묶여 자동 취소됩니다. `AddTo(this)` → 구독 해제 → 진행 중 작업 취소가 연쇄로. `FakeLoadAsync` 자리에 실제 Addressables 로드나 네트워크 요청이 들어가도 같습니다."

> 💡 "`AwaitOperation` 한 단어로 '동시 입력을 어떻게 다룰지'가 선언됩니다. Switch(최신만)·Parallel(동시)은 표로만 — 데모는 Sequential vs Drop 대비까지."

## 이해도 퀴즈

1. 바인딩 패널에서 "스킬 버튼이 저절로 꺼지는" 동작을 만든 연산자 한 줄을 쓰시오.
2. HP를 90→80으로 바꿔도 상태 라벨이 안 바뀌는 이유를 연산자로 설명하시오.
3. 비동기 패널에서 `순차 로드`와 `드롭 로드`를 각각 4번 빠르게 눌렀을 때 처리되는 건수와 그 차이를 만든 옵션을 쓰시오.
4. 로드 중 Play를 정지해도 예외가 안 나는 이유를 `ct`로 설명하시오.

> ✅ **해설**
>
> 1. `m_Hp.CombineLatest(m_Mp, (hp, mp) => hp > 0 && mp >= SkillMpCost).DistinctUntilChanged().Subscribe(canUse => m_SkillButton.interactable = canUse)`.
> 2. `Select`로 만든 구간 문자열("안전")이 같으므로 `DistinctUntilChanged`가 중복을 걸러 통지하지 않음 — 구간이 바뀔 때만 갱신.
> 3. 순차(Sequential)는 4건 모두 줄 서서 처리, 드롭(Drop)은 처리 중 클릭을 버려 1건만 처리. 차이는 `AwaitOperation` 인자.
> 4. `SubscribeAwait`가 넘긴 `ct`가 구독 해제(오브젝트 파괴)에 묶여 자동 취소되고, R3가 그 취소를 에러가 아닌 정상 종료로 다루므로 Console에 예외가 뜨지 않음.

## 💡 강사 팁 · 예상 Q&A

> ❓ **`SerializableReactiveProperty`는 왜 쓰나요? 그냥 `ReactiveProperty`면 안 되나요?**
> 인스펙터에서 초기값을 편집·직렬화하려면 `SerializableReactiveProperty`여야 함(일반 `ReactiveProperty<T>`는 `[SerializeField]` 직렬화 불가). 데모가 HP 100/MP 50을 인스펙터 기본값으로 들고 있는 게 그 때문. 구독 API는 동일.

> ❓ **`DistinctUntilChanged`를 빼면 어떻게 되나요?**
> 동작은 함 — 다만 같은 구간 안에서 값이 바뀔 때마다(예: HP 90→80, 둘 다 "안전") 라벨 대입과 색 갱신이 매번 일어남. `interactable`도 같은 값을 반복 대입. 결과는 같아도 불필요한 작업이라 "바뀔 때만"을 선언하는 것. 구간 라벨처럼 입력 다양도가 출력보다 클 때 효과가 큼.

> ❓ **로드 중 Play를 멈췄는데 왜 예외가 안 나나요?** > `ct`가 취소되면 `UniTask.Delay(..., cancellationToken: ct)`가 `OperationCanceledException`을 던지지만, R3가 구독 해제 경로의 취소를 정상 종료로 처리하므로 Console에 에러로 뜨지 않음. "취소는 에러가 아니다"가 핵심.

> ❓ **`FakeLoadAsync`가 `UniTask`던데, `Task`와 뭐가 다른가요?**
> UniTask는 Unity에 특화된 무할당(zero-allocation) 비동기 타입으로, `Task`의 경량 대체물임. `await` 사용법은 동일하고 데모 이해엔 차이 없음. R3의 `SubscribeAwait` 람다는 표준 `ValueTask`를 받지만 그 안에서 UniTask를 `await`하는 데는 문제 없음. 깊게 갈 자리 아님 — "이 프로젝트의 단발 비동기는 UniTask" 정도로.

> ❓ **이 바인딩을 Model/View로 나누면 어떻게 되나요?**
> 그게 2부 클리커. `m_Hp` 같은 `ReactiveProperty`들이 `ClickerModel`로, `Subscribe`해서 텍스트에 대입하는 부분이 `ClickerPresenter`로 감. 오늘 본 3패턴이 그대로 옮겨짐.

## 패널 4 — 바인딩 (~12분, 메인)

**① 값 → 표시 (~3분)**

> 🔧 오늘 쓸 부품: 02차시 `ReactiveProperty`·`OnClickAsObservable`, 05차시 `Subscribe` 바인딩.

`MyBindingDemo.Start()`가 `{ }`로 비어 있음(스켈레톤). `[SerializeField]` 필드(`m_Hp`·`m_Mp` `SerializableReactiveProperty`, 버튼·텍스트)와 `OnDestroy`(`m_Hp.Dispose()`/`m_Mp.Dispose()`)는 제공 — `Start()` 본문만 채움. 먼저 **표시(구독)**를 선언:

```csharp
// HP가 바뀔 때마다 텍스트 갱신. 구독 즉시 현재값(100)이 한 번 흘러 첫 화면부터 맞음(폴링 없음)
m_Hp.Subscribe(hp => m_HpText.text = $"HP : {hp}").AddTo(this);
m_Mp.Subscribe(mp => m_MpText.text = $"MP : {mp}").AddTo(this);   // MP도 같은 패턴
```

이어서 **값을 바꾸는 네 버튼**을 잇기 — 핸들러는 `ReactiveProperty.Value`만 바꾸고 표시는 위 구독이 따라옴(같은 `Start()`에 이어서):

```csharp
// 네 버튼 모두 "값만" 바꾼다. 텍스트 갱신 코드는 어디에도 없음 — 위 구독이 알아서 따라온다
m_DamageButton.OnClickAsObservable().Subscribe(_ => m_Hp.Value = Mathf.Max(0, m_Hp.Value - 10)).AddTo(this);    // 피해 -10 (0 아래로는 안 감)
m_HealButton.OnClickAsObservable().Subscribe(_ => m_Hp.Value = Mathf.Min(100, m_Hp.Value + 10)).AddTo(this);    // 회복 +10 (최대 100)
m_UseMpButton.OnClickAsObservable().Subscribe(_ => m_Mp.Value = Mathf.Max(0, m_Mp.Value - 5)).AddTo(this);      // MP -5 (최소 0)
m_RestoreMpButton.OnClickAsObservable().Subscribe(_ => m_Mp.Value = Mathf.Min(50, m_Mp.Value + 5)).AddTo(this); // MP +5 (최대 50)
```

**실행·확인**: Play → `피해 -10`/`회복 +10` 누르면 HP 텍스트 즉시 갱신. `MP -5`/`MP +5`도 동일.

- 요점: 갱신을 직접 호출하지 않음. 버튼 핸들러는 `m_Hp.Value`만 바꾸고, 표시는 방금 선언한 구독이 따라옴.

**② 파생 상태 — 구간 라벨 (~4분)**

> 🔧 오늘 쓸 부품: 03차시 `Select`·`DistinctUntilChanged`, 05차시 파생 상태 바인딩.

같은 `Start()`에 HP를 "의미(구간)"로 바꾸는 구독 함께 작성:

```csharp
// HP: --100--90--80--70--69-->
// 구간:--안전---------?---주의->   (70→69에서만 라벨 바뀜)
m_Hp.Select(hp =>                       // 숫자 HP를 구간 문자열로 변환
        hp <= 0 ? "쓰러짐" : hp < 30 ? "위험" : hp < 70 ? "주의" : "안전")
    .DistinctUntilChanged()             // 같은 구간이 연속되면 무시 → 바뀔 때만 통과
    .Subscribe(state =>
    {
        m_HpStateText.text = $"상태: {state}";
        m_HpStateText.color = state switch  // 구간에 맞춰 색도 함께
        {
            "안전" => Color.green,
            "주의" => Color.yellow,
            _ => Color.red,
        };
    })
    .AddTo(this);
```

- 요점: `Select`로 숫자 → 의미(구간), `DistinctUntilChanged`로 바뀔 때만 갱신. 매 프레임 if 검사 대신.

**실행·확인**: Play → HP 천천히 깎으면 상태 라벨(안전→주의→위험→쓰러짐)·색이 구간 바뀔 때만 변함. 같은 구간 안 10씩 깎을 땐 라벨 안 바뀜.

**③ 조건 → 활성화 — CombineLatest (~5분, 하이라이트)**

> 🔧 오늘 쓸 부품: 03차시 `CombineLatest`, 05차시 조건→`interactable` 바인딩.

`Start()`에 HP·MP 두 값 합쳐 스킬 버튼 활성화 정하는 구독 함께 작성:

```csharp
// MP를 -5씩 깎으면: 50 … 20(20>=20이라 아직 가능) → 15(불가)
// HP:   --100-----100-----100--->
// MP:   --50------20------15---->
// 활성: --true----true----false->   (MP가 20 '미만'이 되어야 꺼짐)
m_Hp.CombineLatest(m_Mp,                       // HP·MP 둘 중 아무거나 바뀌면 둘의 최신값으로
        (hp, mp) => hp > 0 && mp >= SkillMpCost)  // 살아 있고 MP가 비용 이상(>=)인가? bool로
    .DistinctUntilChanged()                    // 결과(true/false)가 바뀔 때만 통과
    .Subscribe(canUse => m_SkillButton.interactable = canUse)  // 버튼 활성/비활성에 반영
    .AddTo(this);
```

마지막으로 **스킬 버튼 클릭 핸들러**(MP를 비용만큼 깎고 로그) — 같은 `Start()`에 이어서:

```csharp
// 스킬 사용 — MP를 비용(20)만큼 깎는다. 그 결과 위 CombineLatest가 다시 평가돼 버튼이 꺼질 수 있다
m_SkillButton
    .OnClickAsObservable()
    .Subscribe(_ =>
    {
        m_Mp.Value -= SkillMpCost;
        m_SkillLogText.text = $"스킬 사용! (MP -{SkillMpCost}, 프레임 {Time.frameCount})";
    })
    .AddTo(this);
```

**실행·확인**: Play → `MP -5` 반복해 MP 20 아래로 → 스킬 버튼 저절로 꺼짐. `MP +5`로 20 이상 → 켜짐. HP 0으로 만들어도 꺼짐(HP>0 조건). 켜졌을 때 스킬 버튼 누르면 MP -20 + 로그, MP가 20 아래면 같은 CombineLatest가 버튼을 즉시 꺼 버림.

- 요점: 끄는 코드를 직접 호출한 적 없음 — 조건을 한 번 **선언**했을 뿐. 스킬 사용으로 MP가 줄면 같은 선언이 알아서 버튼을 끔.

- 마무리: 방금 채운 `Start()`(또는 `BindingDemo.cs`) 띄워 3패턴에 ①②③ 번호 매겨 한 번 더 짚기.

```
값→표시:    m_Hp.Subscribe(...)                               (즉시 현재값)
파생 상태:  m_Hp.Select(구간).DistinctUntilChanged().Subscribe(...)
조건→활성화: m_Hp.CombineLatest(m_Mp, 조건).DistinctUntilChanged().Subscribe(interactable)
```

- MVP 예고: 값 가진 쪽(Model)과 보여 주는 쪽(View)을 클래스로 나누면 MVP. 2부 클리커가 전부 이 3패턴으로 만들어짐.

## 패널 5 — 비동기 (~8분, 맛보기)

**① Sequential — 줄 세우기 (~3분)**

> 🔧 오늘 쓸 부품: 03차시 비동기 연산자 `SubscribeAwait` + `AwaitOperation.Sequential`(`FakeLoadAsync` 헬퍼 제공).

`MyAsyncDemo.Start()`에 순차 로드 버튼 구독 함께 작성(`FakeLoadAsync`는 스켈레톤의 1.5초짜리 가짜 로드 헬퍼, 읽기만):

```csharp
m_SequentialButton.OnClickAsObservable()
    .SubscribeAwait(
        async (_, ct) =>                       // 클릭마다 이 async 람다 실행
        {
            int id = ++m_SequentialCount;
            Log($"[순차] 로드 #{id} 시작...");
            await FakeLoadAsync(ct);           // 1.5초 대기. ct는 구독 해제 시 자동 취소
            Log($"[순차] 로드 #{id} 완료");
        },
        AwaitOperation.Sequential)             // ★ 처리 중 들어온 클릭은 큐에 줄 세워 전부 처리(순서 보장)
    .AddTo(this);
```

**실행·확인**: Play → `순차 로드` 빠르게 4번 클릭. 로드 1건당 1.5초 → 줄 서서 #1~#4 순서대로 시작·완료 로그가 약 6초에 걸쳐 흐름.

**② Drop — 버리기 (~3분)**

> 🔧 오늘 쓸 부품: 03차시 `AwaitOperation.Drop`(①과 같은 골격, 옵션만 교체).

`Start()`에 드롭 로드 버튼 구독 함께 작성 — ①과 같은 골격에 **마지막 인자 하나만** 바꿔 끼움:

```csharp
m_DropButton.OnClickAsObservable()
    .SubscribeAwait(
        async (_, ct) =>                       // ① Sequential과 람다 본문 동일
        {
            int id = ++m_DropCount;
            Log($"[드롭] 로드 #{id} 시작... (처리 중 클릭은 무시)");
            await FakeLoadAsync(ct);
            Log($"[드롭] 로드 #{id} 완료");
        },
        AwaitOperation.Drop)                   // ★ 여기만 다름 — 처리 중 들어온 클릭은 버림
    .AddTo(this);
```

- 저장/로드 버튼 연타 방지에 적합. 두 코드의 차이가 마지막 인자 하나뿐임을 나란히 보여 주기.

**실행·확인**: Play → `드롭 로드` 빠르게 4번 클릭. **1건만** 처리, 처리 중 들어온 나머지는 무시. 처리 끝난 뒤 다시 누르면 또 1건.

- 시간 축 대비:

```
clicks:     --c1--c2-c3-------c4-->
Sequential: --[c1….][c2….][c3….][c4….]->   전부 처리, 뒤로 밀림
Drop:       --[c1….]----------[c4….]->     처리 중 c2·c3 버림
```

**③ 취소 연쇄 (~2분)**

> 🔧 오늘 쓸 부품: 04차시 수명·취소(`AddTo` + 람다가 받는 `ct`) — ①②에서 이미 작성한 `ct`의 효과 확인.

- ①②에서 람다가 받은 `ct`를 `FakeLoadAsync(ct)`로 넘긴 그 한 줄이 여기서 일함(새 코드 없음, 작성한 것 확인하는 단계).

**실행·확인**: `순차 로드`로 로드 진행 중 **Play 정지** → 예외 없이 조용히 취소(Console 에러 없음).

- 요점: `SubscribeAwait`가 넘긴 `ct`는 구독 해제(여기선 오브젝트 파괴)에 묶여 자동 취소. `AddTo(this)` → 구독 해제 → 진행 중 작업 취소로 연쇄. `FakeLoadAsync` 자리에 실제 Addressables 로드·네트워크 요청이 와도 동일.

- `AwaitOperation` 한 단어로 '동시 입력을 어떻게 다룰지' 선언됨. Switch(최신만)·Parallel(동시)은 표로만 — 데모는 Sequential vs Drop 대비까지.

# 🎓 마무리 (~3분)

- 두 패널 한 문장: 왼쪽 = **값이 흐르는 UI**(바인딩 3패턴), 오른쪽 = **비동기를 스트림에 얹는 법**(SubscribeAwait + 취소).
- 1부 닫기(안티패턴 회수): 옵저버/이벤트를 버리는 게 아니라 이득 확실한 곳 — 수명 관리·UI 바인딩·시간/입력 합성 — 부터. 2부에서 이 3패턴으로 게임을 만듦.

## 이해도 퀴즈

1. 바인딩 패널에서 "스킬 버튼이 저절로 꺼지는" 동작을 만든 연산자 한 줄을 쓰시오.
2. HP를 90→80으로 바꿔도 상태 라벨이 안 바뀌는 이유를 연산자로 설명하시오.
3. 비동기 패널에서 `순차 로드`와 `드롭 로드`를 각각 4번 빠르게 눌렀을 때 처리되는 건수와 그 차이를 만든 옵션을 쓰시오.
4. 로드 중 Play를 정지해도 예외가 안 나는 이유를 `ct`로 설명하시오.

> ✅ **해설**
>
> 1. `m_Hp.CombineLatest(m_Mp, (hp, mp) => hp > 0 && mp >= SkillMpCost).DistinctUntilChanged().Subscribe(canUse => m_SkillButton.interactable = canUse)`.
> 2. `Select`로 만든 구간 문자열("안전")이 같으므로 `DistinctUntilChanged`가 중복을 걸러 통지하지 않음 — 구간이 바뀔 때만 갱신.
> 3. 순차(Sequential)는 4건 모두 줄 서서 처리, 드롭(Drop)은 처리 중 클릭을 버려 1건만 처리. 차이는 `AwaitOperation` 인자.
> 4. `SubscribeAwait`가 넘긴 `ct`가 구독 해제(오브젝트 파괴)에 묶여 자동 취소되고, R3가 그 취소를 에러가 아닌 정상 종료로 다루므로 Console에 예외가 뜨지 않음.

## 💡 강사 팁 · 예상 Q&A

> ❓ **`SerializableReactiveProperty`는 왜 쓰나? 그냥 `ReactiveProperty`면 안 되나?**
> 인스펙터에서 초기값 편집·직렬화하려면 `SerializableReactiveProperty`여야 함(일반 `ReactiveProperty<T>`는 `[SerializeField]` 직렬화 불가). 데모가 HP 100/MP 50을 인스펙터 기본값으로 드는 이유. 구독 API는 동일.

> ❓ **`DistinctUntilChanged`를 빼면?**
> 동작은 함 — 다만 같은 구간 안 값 변경마다(예: HP 90→80, 둘 다 "안전") 라벨 대입·색 갱신이 매번 일어남. `interactable`도 같은 값 반복 대입. 결과는 같아도 불필요 → "바뀔 때만"을 선언. 입력 가짓수 > 출력 가짓수일 때 효과 큼.

> ❓ **로드 중 Play 정지인데 왜 예외 안 나나?** > `ct` 취소 시 `UniTask.Delay(..., cancellationToken: ct)`가 `OperationCanceledException`을 던지지만, R3가 구독 해제 경로의 취소를 정상 종료로 처리 → Console에 에러로 안 뜸. "취소는 에러가 아니다"가 핵심.

> ❓ **`FakeLoadAsync`가 `UniTask`던데 `Task`와 뭐가 다른가?**
> UniTask는 Unity 특화 무할당(zero-allocation) 비동기 타입, `Task`의 경량 대체물. `await` 사용법 동일, 데모 이해엔 차이 없음. R3의 `SubscribeAwait` 람다는 표준 `ValueTask`를 받지만 그 안에서 UniTask `await`는 문제없음. 깊게 갈 자리 아님 — "이 프로젝트의 단발 비동기는 UniTask" 정도.

> ❓ **이 바인딩을 Model/View로 나누면?**
> 그게 2부 클리커. `m_Hp` 같은 `ReactiveProperty`가 `ClickerModel`로, `Subscribe`해서 텍스트에 대입하는 부분이 `ClickerPresenter`로 감. 오늘 본 3패턴이 그대로 옮겨짐.

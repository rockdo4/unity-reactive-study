## ✅ 수업 전 체크리스트

- [ ] **이 차시는 1부의 수확기** — 새 개념(프로바이더·에러·비동기)이 많아 보이지만, 실제 수업의 무게는 **UI 바인딩 3패턴**(값→표시 / 파생 상태 / 조건→활성화)에 둘 것. 이게 2부에서 그대로 반복됨.
- [ ] **프로바이더는 가볍게** — "시간 연산자가 쓰는 시계를 유니티 것으로 자동 교체해 준다. timeScale 무시 등 변형은 필요할 때 찾는다"까지. DestroyCancellationToken·PlayerLoop 같은 내부로 들어가지 말 것.
- [ ] **에러 처리의 핵심 문장** — "R3의 기본은 **OnErrorResume: 에러가 나도 구독은 살아 있다**. 전통 Rx처럼 끊고 싶으면 OnErrorResumeAsFailure." 그리고 **R3에 Retry가 없다**는 사실(소스 확인됨)을 정확히 — 재시도는 FromAsync 내 루프나 Catch 조합.
- [ ] **SubscribeAwait 표를 자기 언어로** — Sequential(줄 세움)/Drop(버림)/Switch(갈아탐)/Parallel(동시). 각각 게임 용례 하나씩: 큐 처리/저장 버튼/자동완성/다발 다운로드. 비동기 패널 데모는 Sequential vs Drop 대비만 보여 줌.
- [ ] **비동기는 맛보기 수준으로** — 1부 마지막에 "async/await와 어떻게 만나는지"를 보여 주는 게 목적. UniTask 내부 동작이나 깊은 취소 전파까지 파고들지 말 것. 한 줄 await(FirstAsync), 비동기→스트림(FromAsync), SubscribeAwait의 동시성 선언, 이 세 가지만.
- [ ] **UniTask 질문 대비** — 거의 반드시 나옴. "한 번 일어나는 일은 await(UniTask), 여러 번 일어나는 일은 Observable(R3). 같은 생태계라 같이 쓴다." 데모의 `FakeLoadAsync`도 `async UniTask`로 작성돼 있고, R3의 `SubscribeAwait`(ValueTask 람다) 안에서 그대로 await됨.([00. 강사 사전 학습 가이드](00.%20강사%20사전%20학습%20가이드.md))
- [ ] **안티패턴 절은 빼먹지 말 것** — 1부의 마지막 메시지. "장점만 일부 취하기"가 이 강의의 공식 포지션임.

## 5-1. R3.Unity (6분)

- `EveryUpdate` 예제 — **"전역 스트림이므로 AddTo 필수"** 4차시 회수.
- 🔁 비교 박스 ①: `Update()`에 입력·타이머·상태 검사가 뒤엉키는 그림 → `EveryUpdate().Where(...)`로 관심사마다 독립 파이프라인. "매 프레임 폴링 자체가 사라지는 건 아니다 — 표현이 정리되는 것"을 못 박기(성능 단정 금지, 가드레일).
- 프로바이더는 "자동으로 유니티 시계" 한 줄 + 변형(`IgnoreTimeScale`, `~Frame` 연산자)은 카탈로그식 소개. R3.Unity가 시작 시 기본값(`UnityTimeProvider.Update`/`UnityFrameProvider.Update`)을 자동 등록하므로 설정 없이 유니티 시간 기준으로 돈다는 것까지만.

---

## 5-2~3. uGUI 스트림과 바인딩 3패턴 (12분, 메인)

- 🔁 비교 박스 ②: `onClick.AddListener(...)` → `OnClickAsObservable()`. "버튼이 스트림이 되는 순간 3차시의 모든 연산자가 공짜"(쿨다운=`ThrottleFirst`, 연타=`Chunk`, 합성=`Merge`).
- 🔁 비교 박스 ③: bool 플래그를 두고 바뀔 때마다 갱신 메서드 호출 → `ReactiveProperty` + `Subscribe`(구독 즉시 현재값, 폴링 없음).
- 바인딩 3패턴을 칠판에 번호 붙여 정리:
    1. `값.Subscribe(표시)` — 값→표시
    2. `값.Select(가공).DistinctUntilChanged().Subscribe(표시)` — 파생 상태
    3. `a.CombineLatest(b, 조건).Subscribe(버튼.interactable)` — 조건→활성화
- "진짜 바인딩 아님"(공식 표현: 바인딩처럼 동작하는 Observable) 명시. 유니티에 내장 바인딩 시스템이 없어 값 변화→구독자 통지로 흉내 내는 것이고, UI 갱신은 직접 `Subscribe`해서 대입한다.
- MVP 예고 — "값을 가진 쪽(Model)과 보여 주는 쪽(View)을 클래스로 나누면 MVP. 다음 차시(2부)에서 `ClickerModel`/`ClickerPresenter`로 직접."

---

## 5-4. 에러 처리 (7분)

- OnErrorResume 시맨틱(구독 유지)을 게임 맥락(아이템 하나 파싱 실패로 입력 스트림 전체가 죽으면 안 됨)으로.
- Catch / OnErrorResumeAsFailure / IgnoreOnErrorResume 한 줄씩. **Retry 없음** + 대안(FromAsync 안 루프/지수 백오프, Catch로 대체 스트림).
- 핸들러를 생략해도 R3가 등록한 전역 무시 처리 핸들러(R3.Unity 기본: `Debug.LogException`)로 가므로 조용히 삼켜지지 않는다는 점.
- **⚠️ 함정**: 학생이 UniRx의 `Retry`를 검색해 옴 — "R3엔 없다"를 미리 말해 두면 혼란 방지.

## 5-5. 비동기 통합 (7분, 맛보기)

- `FirstAsync`(await 한 번 — 컷씬/튜토리얼 "아무 키나 누르세요"), `FromAsync`(비동기→스트림), `SubscribeAwait`+표.
- `SubscribeAwait`의 핵심: "처리 중에 또 이벤트가 오면?"을 `AwaitOperation`으로 **선언**한다. 표 4종을 게임 용례와 함께.
- 취소 연쇄: "AddTo → 구독 해제 → ct 취소"가 공짜로 따라온다. `destroyCancellationToken`을 직접 넘기는 자리(FirstAsync 등)에도 같은 원리.
- 깊게 가지 말 것 — 비동기는 맛보기. 코드를 읽고 "이렇게 만난다"만 보이면 충분.

## 💡 강사 팁 · 예상 Q&A

**진행 팁**

- 무게중심은 5-2~3 **바인딩 3패턴** — 2부에서 그대로 반복되므로 여기서 칠판에 번호 붙여 또렷이 남긴다. 프로바이더(5-1)·비동기(5-5)는 맛보기·카탈로그식으로 가볍게.
- "성능이 좋아진다"는 단정 금지(가드레일) — `EveryUpdate`는 매 프레임 폴링 자체가 사라지는 게 아니라 **표현이 정리되는 것**임을 못 박는다.
- 정리의 안티패턴 절은 1부의 마지막 메시지이므로 빼먹지 말 것 — "이득이 확실한 곳부터"가 이 강의의 공식 포지션.

**예상 Q&A**

> ❓ **EveryUpdate로 입력을 받으면 Update()보다 느리지 않나요?** (전환형 — `Update` 폴링 → `EveryUpdate`+`Where`)
> 프레임마다 콜백이 도는 건 같음. 차이는 성능이 아니라 구조(관심사 분리·수명 관리). 성능 수치는 단정하지 말 것(가드레일). 이 행이 '기존 방식 → R3' 한눈 매핑표([00. 강사 사전 학습 가이드](00.%20강사%20사전%20학습%20가이드.md))의 `Update` 폴링 행임.

> ❓ **OnErrorResume이 기본이면 에러를 놓치지 않나요?**
> 핸들러를 생략해도 전역 unhandled exception 핸들러(R3.Unity 기본: Debug.LogException)로 가므로 조용히 사라지지 않음. 흐름을 끊어야 하는 에러면 OnErrorResumeAsFailure로 격상.

> ❓ **Switch는 언제 쓰나요?**
> "최신 것만 유효"한 작업 — 검색 자동완성(이전 검색 취소), 마지막 클릭한 아이템의 상세 로드 등. 진행 중인 작업이 ct로 취소된다는 점이 핵심.

> ❓ **UniTask와 R3, 둘 다 써야 하나요?**
> 역할이 다름. 한 번 일어나는 일(로드 한 건, 화면 전환)은 await(UniTask)가 자연스럽고, 여러 번 일어나는 일(입력·값 변화·시간)은 Observable(R3). 같은 Cysharp 생태계라 섞어 써도 자연스럽다([00. 강사 사전 학습 가이드](00.%20강사%20사전%20학습%20가이드.md)). 이 강의는 R3 중심.

> ❓ **UI Toolkit도 되나요?**
> R3는 UI 프레임워크 중립이라 콜백을 FromEvent/Subject로 감싸면 동일하게 적용 가능. 이 강의 데모는 uGUI 기준.

## 이해도 퀴즈

1. UI 바인딩 3패턴을 각각 한 줄 코드 골격으로 쓰시오.
2. R3에서 콜백 예외가 나면 구독이 어떻게 되는가? 전통 Rx와 비교해 쓰시오.
3. 저장 버튼 연타로 중복 저장이 일어나지 않게 하는 SubscribeAwait 옵션은?
4. "모든 C# event를 Observable로 바꾸자"는 제안에 어떻게 답해야 하는가?

> ✅ **해설**
>
> 1. ①`hp.Subscribe(x => text.text = ...)` ②`hp.Select(구간).DistinctUntilChanged().Subscribe(...)` ③`hp.CombineLatest(mp, 조건).Subscribe(can => btn.interactable = can)`.
> 2. R3 기본은 OnErrorResume — 에러를 알리되 구독 유지. 전통 Rx는 OnError로 구독 종료. 끊으려면 OnErrorResumeAsFailure.
> 3. `AwaitOperation.Drop` (처리 중 이벤트를 버림).
> 4. 안티패턴 — pull이 자연스러운 것을 push로 강제하지 말라는 Introduction to Rx의 경고를 들어, 이득이 확실한 곳(수명·바인딩·시간/입력 합성)부터 부분 도입을 제안.

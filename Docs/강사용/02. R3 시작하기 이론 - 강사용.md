## 2-1. 설치 — NuGet + git URL 두 단계 (7분)

**핵심 메시지**

- R3는 **두 부분**으로 나뉘어 설치된다. 표를 띄우고 절차만 읽는다(이 저장소는 설치 완료 — 실제 설치는 안 함, 시간 절약).

| 구성 요소  | 내용                                     | 설치 경로                     |
| ---------- | ---------------------------------------- | ----------------------------- |
| `R3` 본체  | 범용 .NET 라이브러리(DLL)                | **NuGet** (NuGetForUnity로)   |
| `R3.Unity` | 유니티 통합(AddTo, 프레임 프로바이더 등) | **Package Manager** (git URL) |

**순서 (절차만 읽기)**

1. **NuGetForUnity 설치** — Package Manager > `Install package from git URL...`에
   `https://github.com/GlitchEnzo/NuGetForUnity.git?path=/src/NuGetForUnity`
2. **R3 본체 설치** — 메뉴 `NuGet > Manage NuGet Packages`에서 `R3` 검색 → Install. (`Microsoft.Bcl.TimeProvider` 등 의존 패키지가 자동으로 함께 설치됨)
3. **R3.Unity 설치** — Package Manager > git URL에
   `https://github.com/Cysharp/R3.git?path=src/R3.Unity/Assets/R3.Unity#<버전>`

- "유니티에서 NuGet을 쓰는 표준 통로가 NuGetForUnity"까지가 세트. 이 저장소는 설치가 끝나 있으므로 **절차를 보여 주기만** 하고, `Assets/Packages/`(NuGet DLL)와 Package Manager의 R3.Unity를 에디터에서 직접 보여 줄 것. 직접 시연하고 싶다면 별도 빈 프로젝트를 미리 준비.
- 설치 확인: 아무 스크립트에서 `using R3;`가 컴파일되면 됨.

> ⚠️ **함정** — "버전 어떻게 맞춰요?": git URL 태그(`#<버전>`)와 코어 NuGet 버전을 **동일 버전으로 맞춘다**고 답. 태그 = 버전 고정. 최신 버전 재확인은 [R3 릴리스 페이지](https://github.com/Cysharp/R3/releases)에서, [강사 사전 학습 가이드](00.%20강사%20사전%20학습%20가이드.md) 가드레일대로.

## 2-2. 첫 구독 — Subscribe와 3채널 (8분)

**핵심 메시지**

- "`Subscribe`는 **등록**이지 실행이 아니다. `+=`로 명단에 이름을 올리는 것과 같다." (1차시의 "`+=`는 명단 등록" 비유를 그대로 재사용)
- 학생이 가장 많이 하는 오해가 "Subscribe하면 실행되는 것"임. `Range`는 구독 즉시 흐르니 오해를 강화할 수 있음 — "**`Range`는 구독 순간 다 흘려보내고 끝나는 특수한(콜드) 경우**"라고 명시.

**보여줄 코드** — 가장 작은 R3 코드:

```csharp
using R3;
using UnityEngine;

public class HelloR3 : MonoBehaviour
{
    private void Start()
    {
        Observable
            .Range(1, 3)                            // 1, 2, 3을 내보내는 Observable
            .Subscribe(x => Debug.Log($"받음: {x}")); // 구독 — 값이 올 때 할 일을 등록
    }
}
```

- **`Subscribe`의 반환값은 `IDisposable`** — 구독 그 자체임. `Subscribe`를 `IDisposable` 변수로 받아 보이고 "이게 구독 그 자체, 4차시의 주인공"이라고 못 박는다.

**보여줄 코드** — 풀 버전(3채널). 한 번 보여 주되 "지금은 '있다'만":

```csharp
m_SomeObservable.Subscribe(
    x      => Debug.Log($"OnNext: {x}"),              // 값
    ex     => Debug.LogError($"OnErrorResume: {ex}"), // 에러 — 구독은 끊기지 않음(R3 특징)
    result => Debug.Log($"OnCompleted: {result}")     // 완료 — Result는 Success 또는 Failure
);
```

| 채널            | 의미                   | 마블 기호  |
| --------------- | ---------------------- | ---------- |
| `OnNext`        | 값이 하나 도착         | o          |
| `OnErrorResume` | 에러 발생(스트림 유지) | (R3 고유)  |
| `OnCompleted`   | 스트림 종료(성공/실패) | `\|` / `X` |

> ℹ️ 전통 Rx의 `OnError`는 에러가 나면 **구독을 끊었음**. R3는 이를 `OnErrorResume`으로 바꿔 **에러를 흘려보내되 구독은 유지**하는 것을 기본으로 함(게임에서 입력 스트림이 예외 한 번에 죽으면 곤란하므로).

> ⚠️ **이 차시에서는 "채널이 있다"까지만.** `OnErrorResume`의 진짜 의미(구독 유지)와 끊고 싶을 때 쓰는 도구는 5차시에서 다룸. 여기서 깊이 들어가면 시간이 샘.

## 2-3. 팩토리 — 시간·횟수로 스트림 만들기 (7분)

**핵심 메시지**

- `Observable`의 정적 메서드로 기성 스트림을 만든다. 시간·횟수가 값을 만드는 대표 팩토리들. `Range`/`Timer`/`Interval`/`Return`을 마블과 함께.
- `Interval`은 **멈추는 법(Dispose)** 까지가 한 세트. "Dispose가 곧 해제"를 여기서 처음 보여 주고 4차시에서 회수한다(복선).

**보여줄 코드**:

```csharp
// 1) Range — 1부터 4개(1,2,3,4)를 즉시 내보내고 완료. --1-2-3-4-|
Observable.Range(1, 4)
    .Subscribe(x => Debug.Log($"Range: {x}"), result => Debug.Log($"완료: {result}"));

// 2) Timer — 2초 뒤 한 번 내보내고 완료. ------o| (일회성 지연 실행)
Observable.Timer(TimeSpan.FromSeconds(2))
    .Subscribe(_ => Debug.Log("2초 경과!"));

// 3) Interval — 1초마다 무한히. --o--o--o--> (멈추려면 Dispose)
IDisposable subscription = Observable.Interval(TimeSpan.FromSeconds(1))
    .Subscribe(_ => Debug.Log("tick"));
// ...
subscription.Dispose();   // Dispose가 곧 구독 해제 — Interval이 멈춘다

// 4) Return — 값 하나 내보내고 완료. o|
Observable.Return(42).Subscribe(x => Debug.Log(x));
```

**판서** — 마블:

```
Range --1-2-3-4-|   ·   Timer ----o|   ·   Interval --o--o--o-->
```

- `Timer`/`Interval`은 코루틴 `WaitForSeconds`·`InvokeRepeating`의 리액티브 대응물임. **차이는 반환값이 스트림**이라 뒤에 연산자를 이어 붙일 수 있다는 것(3차시).

> 💡 유니티의 "매 프레임"도 팩토리로 스트림이 됨 — `Observable.EveryUpdate()`. 전역 무한 스트림이라 수명 관리가 특히 중요하므로 4·5차시에서 제대로 다룸.

## 2-4. Subject — 손으로 미는 발행기 (5분)

**핵심 메시지**

- 팩토리는 시간·횟수가 값을 만들었음. **내 코드의 사건**을 스트림으로 내보내려면 `Subject<T>`. `event` 코드와 나란히 띄우고 1:1 대응(선언↔필드, Invoke↔OnNext, +=↔Subscribe).

**보여줄 코드**:

```csharp
public class PlayerStat : MonoBehaviour
{
    private readonly Subject<int> m_ScoreSubject = new();

    // 외부에는 Observable로만 노출 — 바깥에서 OnNext를 못 부르게 캡슐화
    public Observable<int> OnScoreChanged => m_ScoreSubject;

    private int m_Score;

    public void AddScore(int amount)
    {
        m_Score += amount;
        m_ScoreSubject.OnNext(m_Score);   // 알림 발사 (event의 Invoke에 해당)
    }

    private void OnDestroy() => m_ScoreSubject.Dispose();   // 발행기도 IDisposable
}
```

- `Subject<T>`는 **Observer이자 Observable인, 두 역할을 겸한 존재** — `OnNext`로 받기도(밀어 넣기) `Subscribe`로 내주기도 함.
- 옵저버 패턴의 주체를 그대로 옮긴 것: `event Action<int>` 선언 ≈ `Subject<int>` 필드, `Invoke` ≈ `OnNext`.
- **외부에는 `Observable<T>`로 노출**하는 게 관례 — "C#에서 `event`가 클래스 밖에서 `Invoke`되지 않는 것과 같은 캡슐화".

## 2-5. ReactiveProperty · FromEvent (5분)

**핵심 메시지 — ReactiveProperty**

- 게임 상태(체력·골드·레벨)는 "사건"이 아니라 **"현재값이 있는데 가끔 바뀌는 것"**. 이를 위한 전용 도구가 `ReactiveProperty<T>`. Subject와 대비해 차이 3가지를 든다.

```csharp
public class PlayerVitals : MonoBehaviour
{
    public ReactiveProperty<int> Hp { get; } = new(100);   // 초기값 100

    public void TakeDamage(int amount) => Hp.Value -= amount; // .Value 대입 시 자동 통지

    private void OnDestroy() => Hp.Dispose();
}

// 구독하는 쪽 — 구독 "즉시" 현재값(100)이 한 번 흐르고, 이후 바뀔 때마다 흐른다
vitals.Hp.Subscribe(hp => Debug.Log($"HP: {hp}"));
```

`Subject`와의 차이 세 가지:

1. **현재값을 보유** — `Hp.Value`로 언제든 읽을 수 있음.
2. **구독 즉시 현재값이 한 번 흐름** — 늦게 합류한 UI도 첫 프레임부터 올바른 값 표시.
3. **같은 값이면 통지하지 않음** — 기본으로 중복 억제.

> ℹ️ 인스펙터에서 초기값을 편집하고 싶으면 `ReactiveProperty<T>` 대신 **`SerializableReactiveProperty<T>`** 를 씀(일반 버전은 `[SerializeField]` 직렬화 불가). 5차시 바인딩 데모가 이걸 사용함. UI 바인딩(값 변화 → 텍스트 자동 갱신)은 5차시의 주제 — 오늘은 "값을 보유하는 스트림"이라는 정체성만.

**핵심 메시지 — FromEvent**

- 이미 있는 C# `event`나 서드파티 콜백을 스트림으로 감싸는 어댑터. "기존 코드와의 다리, `-=` 자동"이 전부.

```csharp
public event Action<int> LegacyScoreEvent;   // 기존 코드의 event

Observable
    .FromEvent<int>(
        handler => LegacyScoreEvent += handler,   // 구독 시작 시 +=
        handler => LegacyScoreEvent -= handler    // 구독 해제 시 -= (자동!)
    )
    .Subscribe(score => Debug.Log($"점수: {score}"));
```

- 구독을 `Dispose`하면 **`-=`가 자동 호출됨**. 기존 이벤트 기반 코드를 점진적으로 리액티브에 합류시키는 다리.
- 시그니처 문법(`h => e += h`)은 어렵게 느끼므로 "**외워 쓰는 관용구**"라고 안심시킬 것.

> 🔁 **Unity에서 늘 하던 방식 ↔ R3 (Observable 만들기)**
>
> | Unity에서 늘 하던 방식                      | 무엇이 문제였나                                  | R3 대응                                                             |
> | ------------------------------------------- | ------------------------------------------------ | ------------------------------------------------------------------- |
> | `event Action<T>` 선언 + `Invoke`로 통지    | 합성·시간 조건이 안 됨, 해제도 수동              | `Subject<T>` 필드 + `OnNext` — Observable로 노출해 연산자 체인 가능 |
> | `WaitForSeconds` / `InvokeRepeating` 코루틴 | 반환이 스트림이 아니라 뒤에 연산자를 못 이음     | `Timer` / `Interval` — 반환이 스트림이라 연산자 체인 가능 (3차시)   |
> | 기존 `event`에 `+=` / `-=` 수동 해제        | `Dispose` 시점과 `-=` 시점이 따로 놀아 누락 위험 | `FromEvent`로 감싸면 `Dispose` 시 `-=` 자동 호출 — 점진 도입의 다리 |
>
> 메타 축 둘만 기억: **`Invoke`/폴링(push 수동) → `OnNext`/스트림 구독**, **수동 `-=` 해제 → `Dispose`로 수명 결합**.

## 2-6. 콜드 vs 핫 · 값이 흐르는 순서 (5분)

**핵심 메시지 — 콜드/핫**

- 정의 논쟁으로 들어가지 말 것. 두 문장이면 충분:
    - **콜드(cold)** — 구독할 때마다 처음부터 자기 전용 스트림이 흐름. `Range`·`Timer` 같은 팩토리.
    - **핫(hot)** — 스트림이 하나만 흐르고 구독자들은 **흐르는 중간에 합류**. `Subject`·클릭 스트림. 구독 전에 지나간 값은 못 받음.
- "팩토리는 구독마다 새로(콜드), Subject·클릭은 흐르는 데 합류(핫)". `Share()`는 3차시로 미룸.

**판서 — 생성법별로 값이 흐르는 순서**

다섯 가지 생성법은 "**언제·어떻게** 값을 흘리는가"가 각각 다름. 구독 시점 기준 마블:

```
Range(1,4):   -1-2-3-4-|          (구독 즉시 4개 + 완료)
Timer(2s):    ------o-|           (2초 뒤 1개 + 완료)
Interval(1s): --o--o--o--o-->     (무한 — Dispose로만 멈춤)
Subject:      --o----o-o---->     (OnNext 호출 시점마다)
```

| 생성법           | 구독 후 값이 흐르는 방식                                            |
| ---------------- | ------------------------------------------------------------------- |
| `Range(1, 4)`    | 구독 즉시 1~4 연달아 흐르고 `OnCompleted` — **완료가 있는** 유한    |
| `Timer(2초)`     | 2초 뒤 한 번 발행하고 완료 — 일회성                                 |
| `Interval(1초)`  | 1초마다 무한 발행. 스스로 끝나지 않으므로 멈춤은 곧 **`Dispose()`** |
| `Subject.OnNext` | 내가 `OnNext`를 부르는 시점마다 발행 — 손으로 미는 발행기           |
| `FromEvent`      | 감싼 `event`가 발생할 때 수신. 구독을 `Dispose`하면 `-=` 자동       |

- 읽어 둘 핵심 둘: ① **완료(`|`)가 있는 스트림과 없는 스트림의 구분**, ② 무한 스트림(`Interval`)은 `Dispose()`로만 멈춘다(4차시 수명 관리의 복선).

## 3. 예상 Q&A

> ❓ **NuGet 없이 DLL만 복사하면 안 되나요?**
> 기술적으로는 되지만 의존 패키지(Microsoft.Bcl.TimeProvider 등) 누락·버전 충돌을 손으로 관리하게 됨. NuGetForUnity가 그걸 자동화하는 표준 도구.

> ❓ **Subject랑 ReactiveProperty 중 뭘 써야 하나요?**
> "현재값"이 의미 있으면 RP(체력·골드·설정값), 순간의 사건이면 Subject(공격 명중·아이템 드랍). "지금 값이 뭐야?"라는 질문이 성립하면 RP.

> ❓ **Observable이 인터페이스가 아니라 클래스인 이유는?**
> R3가 모든 구독을 베이스 클래스에서 추적·관리하기 위해서(구독 누수 진단에 유리). 깊이는 들어가지 말고 "그래서 Tracker 같은 도구가 가능하다" 정도로(4차시 복선).

> ❓ **코루틴 `WaitForSeconds`로 "N초 뒤 1회 실행"하던 건 R3로 어떻게 바뀌나요?** (전환형)
> `Observable.Timer(TimeSpan.FromSeconds(N))` 한 줄. 코루틴과 달리 반환이 스트림이라 뒤에 연산자를 이어 붙일 수 있고, 구독을 `AddTo`로 묶으면 수명도 자동 정리됨. 기존 스트림에 지연을 끼우는 거면 `Delay`. ('기존 방식 → R3' 한눈 매핑표는 [강사 사전 학습 가이드](00.%20강사%20사전%20학습%20가이드.md))

## 4. 이해도 퀴즈

1. R3 설치가 두 단계(NuGet + git URL)인 이유를 쓰시오.
2. `Subscribe`가 "실행"이 아니라 "등록"인 이유를 `+=` 비유로 설명하시오.
3. `Interval`로 만든 무한 스트림을 멈추는 코드를 쓰시오.
4. `Subject<int>`와 `ReactiveProperty<int>`의 차이를 두 가지 이상 쓰시오.

> ✅ **해설**
>
> 1. R3 본체는 범용 .NET 라이브러리라 NuGet으로 배포되고, 유니티 전용 통합(AddTo·프로바이더 등)은 별도 패키지 R3.Unity로 UPM(git URL) 배포되기 때문.
> 2. 명단에 이름을 올리는 것과 같음 — 콜백은 값이 흐를 때 실행됨. (`Range`처럼 구독 즉시 흐르는 콜드 스트림은 등록과 동시에 값이 오는 특수 경우.)
> 3. `IDisposable sub = Observable.Interval(...).Subscribe(...); sub.Dispose();`
> 4. RP는 현재값을 보유(`.Value`), 구독 즉시 현재값 1회 통지, 같은 값 중복 억제. Subject는 셋 다 없음.

## 💡 강사 팁

- 설치 절(2-1)은 절차만 빠르게 — 실제 설치는 안 함. 시간이 빠듯하면 표 + 함정 한 줄로 압축 가능.
- 2-2의 "Subscribe는 등록"이 이 차시의 정신적 기둥 — `Range`의 "즉시 흐르는 콜드" 특수성을 여기서 분명히 끊어 두면 이후 혼란이 줄어듦.
- `Interval`의 `Dispose`는 4차시 복선이므로 "해제 방법이 있다"까지만, 깊이 들어가지 말 것.
- 질문은 절 종료 시점마다 모아서 — 코드 깊이가 필요한 질문은 "따라하기에서 직접 본다"로 연결.

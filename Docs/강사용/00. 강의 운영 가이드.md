# 00. 강의 운영 가이드 — 강사용

이 문서는 **R3·리액티브 프로그래밍을 처음 접하는 강사**를 기준으로 씀. 1장(사전 학습 로드맵)을 수업 1~2주 전에 시작하는 것을 강력 권장. 차시별 진행은 각 차시 강사용 문서를 따름.

## 1. 강사 사전 학습 로드맵 (수업 전 필수)

> ℹ️ 순서대로 할 것 — 각 단계가 다음 단계의 토대가 됨.

### 1주 전 — 개념과 어휘

1. **이 강의의 교안 01을 정독** — [01. 리액티브 프로그래밍이란](../교안/01.%20리액티브%20프로그래밍이란.md). 학생에게 가르칠 바로 그 프레임(`Observable = 옵저버 + Operator + IDisposable`)을 먼저 자기 것으로.
2. **ReactiveX 입문 페이지** <https://reactivex.io/intro.html> — "Rx는 옵저버 패턴의 확장"이라는 정의의 1차 출처. 영어가 부담이면 훑기만 해도 됨(교안이 핵심을 옮겨 둠).
3. **R3 README** <https://github.com/Cysharp/R3> — 전체 그림. 처음에는 ①Basic Usage ②Subscription Management(AddTo) ③Unity 섹션 세 곳만. 연산자 표는 사전처럼 쓰는 것이지 외우는 게 아님.

이 단계의 목표: **"리액티브가 뭐냐"는 질문에 옵저버 패턴부터 시작해 3분 동안 막힘없이 말할 수 있는 상태.**

### 4~5일 전 — 직접 굴려 보기

4. **이 저장소를 클론해 데모를 실행** — `Tools > Reactive Study > 1부 기초 데모 씬 생성` 후 Play. 패널 1~5를 전부 눌러 보며 교안 02~05를 옆에 두고 대조.
5. **데모 코드 통독** — `Assets/Scripts/ReactiveBasics/`의 5개 파일. 각 줄이 교안의 어느 절인지 확인. 특히 `LifetimeDemo`의 "누수 만들기" 버튼과 `Window > Observable Tracker`는 수업 중 시연 효과가 크므로 손에 익혀 둘 것.
6. **작은 실험 3개를 직접 작성**(빈 씬에서, 보지 않고):
    - ① 버튼 클릭을 `ThrottleFirst`로 쿨다운 걸기
    - ② `ReactiveProperty<int>` 하나를 텍스트에 바인딩하기
    - ③ `Observable.Interval` 구독을 만들었다가 오브젝트 파괴로 자동 해제되는지 Tracker로 확인하기
      막히면 교안 02~05를 다시. **이 3개를 못 하면 라이브 코딩은 미루고 완성 데모 시연으로 수업할 것.**

### 2~3일 전 — 게임을 한 번 완주

7. **교안 06을 따라 클리커를 처음부터 끝까지 직접 만들기** — 학생이 겪을 일을 미리 겪는 단계. 어디서 오타가 나기 쉬운지, 어느 단계가 오래 걸리는지 몸으로 알게 됨. 본인이 막힌 곳이 학생이 막힐 곳임 — 메모해 두고 수업에서 선제 대응.
8. **(선택) 심화 읽을거리**
    - R3 발표 글(저자 neuecc): <https://neuecc.medium.com/r3-a-new-modern-reimplementation-of-reactive-extensions-for-c-cf29abcc5826> — 왜 3세대를 다시 만들었는지, UniRx와의 차이의 1차 근거.
    - Introduction to Rx: <https://introtorx.com> — 안티패턴 경고("pull을 push로 강제하지 말라")의 출처.
    - UniRx README(아카이브): <https://github.com/neuecc/UniRx> — 학생이 옛 자료를 들고 올 때를 대비한 역사 지식.

### 강사가 먼저 잡아야 할 멘탈 모델 5개

1. **리액티브 = 옵저버 패턴의 확장** — 새 패러다임 신비화 금지. "이미 아는 옵저버에 연산자와 표준 해제를 더한 것."
2. **이벤트는 시간 위의 데이터 스트림** — 마블 다이어그램이 공용어. 칠판에 가로선+구슬을 그리는 습관.
3. **Subscribe는 등록, Dispose는 해제** — `+=`/`-=`의 표준화. 모든 수명 관리가 이 위에 있음.
4. **연산자 합성이 진짜 가치** — 더블클릭·쿨다운·콤보가 상태 플래그 없이 한두 줄이 되는 것. 이걸 보여 주는 게 강의의 하이라이트.
5. **만병통치약이 아님** — pull이 자연스러운 곳에 강제하지 말 것. "장점만 일부 취하기"가 권장 포지션.

## 3. 검증된 사실 가드레일 (학생 앞에서 단정하지 말 것)

> ❗ 수업 중 아래 항목은 **여기 적힌 표현까지만** 말할 것. 더 단정하면 틀릴 수 있음.

- **버전** — 이 자료는 R3 **수업 시점 최신 버전**을 기준으로 함. 설치 시 <https://github.com/Cysharp/R3/releases>에서 최신 버전을 확인할 것(Releases가 정본 체인지로그).
- **지원 범위** — 공식 문서가 밝힌 것은 "R3는 **Unity 2021.3 이상** 지원"까지. 특정 유니티 버전·IL2CPP 실동작을 본 자료가 보증하지 않음(이 저장소의 Unity 6.3 에디터 동작은 확인 대상).
- **UniRx 아카이브** — "아카이브(읽기 전용)되었고 README가 R3 사용을 권한다"까지만. 정확한 일자를 못 박지 말 것.
- **UniRx의 존재 이유** — "공식 Rx.NET이 Unity에서 동작하지 않고 iOS IL2CPP(AOT) 호환성 문제가 있어 재구현"이 정확한 표현. 막연히 "Mono 문제"라고 하지 말 것.
- **R3에 `Retry` 없음** — R3 연산자에 `Retry`가 없음(소스 확인 기준). 재시도는 `FromAsync` 내 루프 또는 `Catch` 조합으로 안내.
- **"극적 단순화" 같은 과장 금지** — 정확한 표현은 "더 짧고 선언적으로 표현할 수 있다". 디버깅·학습 비용이라는 대가를 항상 함께 말할 것.
- **"모든 event를 Rx로 대체" 금지** — _Introduction to Rx_ 의 안티패턴 경고와 함께만 제시.
- **채택률 수치 금지** — 스타 수·점유율 같은 정량 주장 대신 "정식 후속작으로 수렴" 같은 정성 표현만.

## 4. 자주 받는 질문 총괄 (차시 무관)

> ❓ **UniRx 튜토리얼이 훨씬 많던데 그걸로 배우면 안 되나요?**
> 개념은 같으니 참고는 가능. 단 UniRx는 아카이브됐고 연산자 이름이 다름(`Throttle→Debounce`, `Buffer→Chunk`). 새 코드는 R3로 쓰고, 옛 자료는 교안 03 부록의 매핑표를 거쳐 읽으라고 안내.

> ❓ **UniTask랑은 무슨 관계인가요?**
> 다른 도구임. UniTask는 단발 비동기 작업(async/await)의 고성능 구현, R3는 **반복되는 이벤트 스트림**의 합성. 같은 Cysharp 생태계라 함께 쓰는 일이 많고, R3는 await 통합(교안 05)으로 둘 사이를 잇는다 — "한 번 일어나는 일은 await, 여러 번 일어나는 일은 Observable"이 기준선.

> ❓ **성능은 괜찮나요? GC는요?**
> R3는 현대적 재설계로 내부 할당을 줄였고 구독 추적도 중앙화함. 다만 수치 비교를 단정하지 말 것(가드레일). 게임플레이 코드 기준 일반적 사용에서 문제가 되는 경우는 드물고, 핫패스(매 프레임 수천 회)에는 어떤 추상화든 신중해야 한다는 일반론으로 답할 것.

> ❓ **MVVM/MVP 같은 아키텍처를 꼭 같이 써야 하나요?**
> 아님. R3는 패턴 없이도 쓸 수 있음. 다만 ReactiveProperty가 Model→View 통지를 자연스럽게 만들어 MVP류와 궁합이 좋고, 2부 클리커가 그 최소 예시(Model/Presenter 분리)임.

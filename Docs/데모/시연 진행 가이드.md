# 데모 시연 진행 가이드

수업 중 데모를 **조작 → 화면/Console → 연산자 핵심 → 멘트 → 교안 연결** 순서로 진행하는 대본. 씬 구성은 [구성 가이드](구성%20가이드.md), 씬 생성은 [자동 생성 가이드](자동%20생성%20가이드.md)를 참조.

## 공통 사전 준비 (수업 전)

1. [자동 생성 가이드](자동%20생성%20가이드.md)로 두 씬을 미리 생성해 둔다.
2. `Window > Observable Tracker`를 열어 도킹해 둔다(4차시 핵심 도구). Console도 열어 둔다.
3. Game 뷰 해상도를 16:9로(패널이 기준 1920×1080에 맞춰 배치됨).

## 2차시 — 패널 1 "Observable 만들기"

- **조작**: `Range(1, 4)` → `Timer 2초` → `Interval 시작`(다시 눌러 중지) → `Subject.OnNext` → `event 발생 (FromEvent)` 순으로 누른다.
- **화면/Console**: 로그 패널에 `Range OnNext: 1~4`와 `OnCompleted`, 2초 뒤 `Timer 발행!`, 1초 간격 `Interval tick`, `Subject 수신`, `FromEvent 수신: 점수…`.
- **연산자 핵심**: 완료(`|`)가 있는 스트림(`Range`/`Timer`)과 없는 스트림(`Interval`)의 구분. `Interval`은 **중지 = `Dispose()`**.

```
Range(1,4):   ─1─2─3─4─|
Timer(2s):    ──────●─|
Interval(1s): ──●──●──●──▶   (중지 = Dispose)
```

- **멘트**: "구독은 등록일 뿐, 값을 **언제** 흘리는지가 생성법마다 다르다."
- **교안 연결**: [교안 02 §3·§4·§6](../교안/02.%20R3%20시작하기%20—%20Observable%20만들기.md).

## 3차시 — 패널 2 "연산자"

- **조작**: 화면을 **빠르게 두 번 클릭**(더블클릭) → 검색창에 타이핑 후 멈춤 → `탭` 버튼 빠르게 3번 → 다시 천천히.
- **화면/Console**: 더블클릭 시에만 반응, 검색은 입력 멈춘 0.5초 뒤·같은 검색어는 무시, 500ms 안에 3번이면 "3연타!", 쿨다운은 2초에 한 번.
- **연산자 핵심**: `EveryUpdate→Where→Share` + `Chunk(Debounce 250ms)→Where(≥2)`(더블클릭), `Debounce(500ms)→DistinctUntilChanged`(검색), `Chunk(500ms,3)`(연타), `ThrottleFirst(2s)`(쿨다운).

```
탭:           ──t─t─t────────t──▶
ThrottleFirst:──t────────────t──▶   (먼저 한 번, 구간 무시)
Debounce:     ──────────●───────▶   (잠잠해지면 마지막)
```

- **🔻 고의 실패 시연(UniRx Throttle)**: 학생이 검색해 온 UniRx 예제의 `Throttle`을 그대로 입력 → **컴파일 에러**를 Console에서 보여 준다. R3에서는 `Throttle → Debounce`(최대 함정), `Buffer → Chunk`. [교안 03 부록](../교안/03.%20연산자%20—%20스트림을%20다듬고%20합성하기.md)의 이름 매핑표로 복구. (강사용 03 참조)
- **멘트**: "백 마디 설명보다 컴파일 에러 한 번이 매핑을 각인시킨다."
- **교안 연결**: [교안 03 §4](../교안/03.%20연산자%20—%20스트림을%20다듬고%20합성하기.md).

## 4차시 — 패널 3 "구독 수명" (핵심 시연)

`Window > Observable Tracker`를 띄운 채 진행.

- **조작 ①(관리)**: `구독 추가`를 3번 → `활성 구독 조회`(카운트 증가 확인) → `모두 해제` → 다시 `활성 구독 조회`(줄어듦). 멘트: "`CompositeDisposable`에 담으면 한 번에 정리된다."
- **조작 ②(AddTo 자동 해제)**: `활성 구독 조회`로 프로브 구독 확인 → `프로브 파괴` → 다시 조회 → **프로브 구독이 사라짐**. Console: `[LifetimeProbe] 파괴됨 — 구독도 자동 해제`. 멘트: "`AddTo(this)`는 오브젝트 파괴에 수명을 묶는다."
- **🔻 고의 실패 시연(누수)**: `누수 만들기`를 누른다. 이 구독은 `AddTo`도 변수 보관도 없는 전역 `Interval(5s)` 구독 — **핸들이 없어 해제할 방법이 없다**. `활성 구독 조회`/`Observable Tracker`에서 카운트가 계속 남고, Console에 `누수 구독 #N tick (해제 불가!)`가 5초마다 영원히 찍힌다.
    - **눈으로 보이기**: `모두 해제`를 눌러도 누수 구독은 사라지지 않는다(관리 구독만 정리됨) — Tracker에 남아 대비됨.
    - **복구**: 누수 구독은 코드로 회수 불가 → **Play 정지**가 유일한 정리. 교훈: "그래서 처음부터 `AddTo`/`Dispose`로 묶어야 한다." (대조: 프로브는 파괴만으로 깨끗이 사라졌다.)

```
관리 구독:  AddTo(CompositeDisposable) → "모두 해제"로 정리 ✅
프로브:     AddTo(this)               → 파괴 시 자동 해제 ✅
누수:       (없음)                    → Tracker에 영구 잔류 ❌ → AddTo가 답
```

- **교안 연결**: [교안 04](../교안/04.%20구독%20수명%20관리.md).

## 5차시 — 패널 4·5 "바인딩·비동기"

- **패널 4 조작**: `피해 -10`/`회복 +10`, `MP -5/+5`, `스킬 사용`. HP/MP 텍스트가 즉시 갱신, 상태 라벨(안전/주의/위험)은 구간이 바뀔 때만, HP>0 & MP≥20일 때만 스킬 버튼 활성.
    - **연산자 핵심**: `ReactiveProperty.Subscribe`(즉시 현재값), `Select→DistinctUntilChanged`(구간 라벨), `CombineLatest`(여러 값 조건 → `interactable`).
- **패널 5 조작**: `순차 로드`를 빠르게 4번 → 줄 서서 4건 모두 처리(Sequential). `드롭 로드`를 빠르게 4번 → 처리 중 클릭은 버려짐(Drop). (로드 1건당 1.5초)

```
클릭:        ──c1─c2─c3────────c4──▶
Sequential:  ──▶c1──▶c2──▶c3────▶c4   (전부 처리, 뒤로 밀림)
Drop:        ──▶c1──╳──╳───────▶c4    (처리 중 무시)
```

- **멘트**: "`AwaitOperation` 한 단어로 '동시 입력을 어떻게 다룰지'가 선언된다. `ct`는 구독 해제 시 자동 취소."
- **교안 연결**: [교안 05](../교안/05.%20Unity%20통합과%20비동기.md).

## 6차시 — `ReactiveClicker`(완성본)

- **조작**: `클릭!` 연타로 콤보(0.8초 멈추면 리셋) → 10·20콤보에서 보너스 → `클릭 강화`/`도우미 고용` 구매(골드 부족 시 버튼 꺼짐) → `피버!`(15초 쿨다운, 5초간 2배).
- **확인**: 자동수입(초당)이 도우미 고용 후 증가. **Play 정지 직전 `Observable Tracker`에서 누수 0** 확인 — 모든 구독이 `AddTo`로 묶여 있음(2부 실습 제출물의 "Tracker 누수 0 스크린샷").
- **교안 연결**: [교안 06](../교안/06.%20따라하기%20—%20리액티브%20클리커%20만들기.md).

## (선택) 결정론 테스트와 짝짓기

패널 3의 누수를 "눈으로" 보였다면, 같은 시간 의존 동작을 **테스트로도** 잡을 수 있음을 한 줄 곁들여도 좋다. R3는 `ObservableSystem.DefaultTimeProvider`를 가짜 시계로 바꾸면 `Debounce`/`Throttle`/`Interval`을 실시간 대기 없이 결정론적으로 검증할 수 있다(가상 시간 `Advance`). 현재 비-시간 규칙은 `Assets/Scripts/ClickerGame/Tests/ClickerModelTests.cs`에 EditMode 테스트로 들어 있고, 시간 의존 결정론 테스트는 후속 과제다(테스트 헬퍼 패키지 호환성은 같은 파일 헤더 주석 참조).

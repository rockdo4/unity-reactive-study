# 05. Unity 통합과 비동기

R3가 **유니티의 시간·프레임·UI**와 어떻게 맞물리는지(R3.Unity), 값 변화를 UI에 바인딩하는 정석 패턴, 에러를 다루는 법, 그리고 `async/await`와의 통합까지 익히는 장.

## 1. R3.Unity가 더해 주는 것 - 시간과 프레임의 다리

R3 본체는 범용 .NET 라이브러리라 유니티의 `Time.time`도 게임 루프도 모름. 그 다리가 `R3.Unity` 패키지임.

- **프레임 스트림** - `Observable.EveryUpdate()`: 매 프레임 발행하는 전역 스트림. `Update()` 함수의 리액티브 대응물.
- **타임/프레임 프로바이더** - `Timer`/`Interval`/`Debounce` 같은 시간 연산자가 내부에서 쓰는 "시계"를 유니티 것으로 교체함. R3.Unity가 시작 시 자동으로 기본값을 등록하므로(`UnityTimeProvider.Update` / `UnityFrameProvider.Update`) **아무 설정 없이 유니티 시간 기준으로 동작**함.
- **변형 프로바이더** - timeScale 무시(`UnityTimeProvider.UpdateIgnoreTimeScale`) 등이 필요하면 연산자의 `TimeProvider` 인자로 골라 끼움. 일시정지 중에도 돌아야 하는 UI 타이머 같은 곳에서 씀.
- **프레임 단위 연산자** - `DelayFrame`, `DebounceFrame`, `ThrottleFirstFrame`처럼 시간 대신 **프레임 수**로 동작하는 변형이 있음. "다음 프레임에", "3프레임 모아서" 같은 게임 특유의 요구와 맞음.
- 그 외 함께 쓰는 것들 - `AddTo`(구독 수명 관리), `SerializableReactiveProperty`(인스펙터에 노출되는 ReactiveProperty), uGUI 확장(아래), `Window > Observable Tracker`(구독 추적 창).

```csharp
// 매 프레임 입력 검사 - Update() 함수 없이. 전역 스트림이므로 AddTo 필수!
Observable.EveryUpdate()
    .Where(_ => Keyboard.current != null && Keyboard.current.spaceKey.wasPressedThisFrame)
    .Subscribe(_ => Jump())
    .AddTo(this);
```

> 💡 `Update()` 함수와 뭐가 다른가 - 함수 하나에 입력·타이머·상태 검사가 뒤엉키는 대신, **관심사마다 독립된 파이프라인**으로 분리되고 각자 수명을 가짐. 다만 매 프레임 폴링 자체가 사라지는 건 아님 - "표현이 정리되는 것"임.

## 2. uGUI 이벤트를 스트림으로

R3.Unity가 uGUI에 확장 메서드를 더해 줌.

```csharp
// Button - 가장 많이 쓰는 형태
m_Button.OnClickAsObservable()
    .ThrottleFirst(TimeSpan.FromSeconds(1))
    .Subscribe(_ => Fire())
    .AddTo(this);

// 그 외 UnityEvent 일반 - AsObservable()로 감싼다
m_InputField.onValueChanged.AsObservable()   // TMP_InputField의 UnityEvent<string>
    .Debounce(TimeSpan.FromMilliseconds(500))
    .Subscribe(text => Search(text))
    .AddTo(this);

m_Slider.onValueChanged.AsObservable()
    .Subscribe(v => SetVolume(v))
    .AddTo(this);
```

`onClick.AddListener(...)`와 다른 점: 이벤트가 **스트림**이 됐으므로 쿨다운(`ThrottleFirst`)·연타(`Chunk`)·합성(`Merge`) 같은 모든 연산자를 그대로 쓸 수 있음.

## 3. ReactiveProperty → UI 바인딩

게임 UI의 기본 문제: "값이 바뀌면 표시를 갱신하라." `ReactiveProperty`와 연산자를 합치면 이렇게 됨.

```csharp
public class PlayerVitalsUI : MonoBehaviour
{
    [SerializeField] private SerializableReactiveProperty<int> m_Hp = new(100);
    [SerializeField] private SerializableReactiveProperty<int> m_Mp = new(50);
    [SerializeField] private TextMeshProUGUI m_HpText;
    [SerializeField] private TextMeshProUGUI m_StateText;
    [SerializeField] private Button m_SkillButton;

    private void Start()
    {
        // 1) 값 → 텍스트. 구독 즉시 현재값이 흐르므로 첫 화면부터 올바름. 폴링 없음
        m_Hp.Subscribe(hp => m_HpText.text = $"HP : {hp}").AddTo(this);

        // 2) 파생 상태 - 구간이 바뀔 때만 갱신 (Select + DistinctUntilChanged)
        m_Hp.Select(hp => hp < 30 ? "위험" : hp < 70 ? "주의" : "안전")
            .DistinctUntilChanged()
            .Subscribe(state => m_StateText.text = state)
            .AddTo(this);

        // 3) 여러 값에 걸친 조건 - 버튼 활성화 (CombineLatest)
        m_Hp.CombineLatest(m_Mp, (hp, mp) => hp > 0 && mp >= 20)
            .DistinctUntilChanged()
            .Subscribe(can => m_SkillButton.interactable = can)
            .AddTo(this);
    }
}
```

이 세 줄 패턴(값→표시 / 파생 상태 / 조건→활성화)이 리액티브 UI의 9할임.

> ⚠️ **"진짜 데이터 바인딩"은 아님** - 공식 표현으로 ReactiveProperty는 "바인딩처럼 동작하는 Observable". 유니티에 내장 바인딩 시스템이 없어 값 변화→구독자 통지로 바인딩을 흉내 내는 것. UI 갱신은 위처럼 **직접 `Subscribe`해서 대입**함.

> 💡 이 구조에서 "값을 가진 쪽(Model)"과 "표시하는 쪽(View)"을 클래스로 분리하면 그대로 **MVP(Model-View-Presenter)** 가 됨.

## 4. 에러 처리 - 스트림은 쉽게 죽지 않는다

### OnErrorResume - R3의 기본기

전통 Rx에서는 콜백에서 예외가 나면 `OnError`로 빠지며 **구독이 끊겼음**. 게임에서는 치명적임(아이템 하나 파싱 실패로 입력 스트림 전체가 죽는 식). R3는 기본 에러 채널을 **`OnErrorResume`** 으로 바꿈 - 에러를 알리되 **구독은 유지**.

```csharp
itemStream.Subscribe(
    item => Inventory.Add(item),
    ex   => Debug.LogError($"아이템 처리 실패(스트림은 계속): {ex.Message}")
).AddTo(this);
```

에러 핸들러를 생략하면 R3가 등록한 **전역 무시 처리 핸들러**(R3.Unity 기본: `Debug.LogException`)로 감 - 조용히 삼켜지지 않음.

### 흐름을 바꾸는 도구들

```csharp
// Catch - 에러가 "완료(실패)"로 끝난 스트림을 대체 스트림으로 잇는다
primarySource.Catch((Exception ex) => fallbackSource);

// OnErrorResumeAsFailure - OnErrorResume을 "실패 완료"로 격상 (전통 Rx처럼 끊고 싶을 때)
source.OnErrorResumeAsFailure();

// IgnoreOnErrorResume - 에러를 의도적으로 무시 (정말 무시해도 되는 곳에만)
source.IgnoreOnErrorResume(ex => Debug.LogWarning(ex.Message));
```

> ⚠️ **R3에는 UniRx의 `Retry` 연산자가 없음.** 재시도가 필요한 작업(네트워크 등)은 보통 ①`FromAsync` 안에서 루프/지수 백오프로 재시도하거나 ②`Catch`로 대체 스트림을 잇는 방식으로 구성함. "에러 나면 알아서 N번 재구독" 한 줄은 없다고 기억할 것.

## 5. async/await 통합

R3는 `async/await`와 양방향으로 통함.

### Observable → await: 한 번의 사건을 기다리기

```csharp
// "다음 클릭 한 번"을 await - 컷씬 진행, 튜토리얼 "아무 키나 누르세요"
await m_Button.OnClickAsObservable().FirstAsync(destroyCancellationToken);

// 스트림이 완료될 때까지 대기
await someStream.WaitAsync(ct);
```

### Task → Observable: 비동기 작업을 스트림으로

```csharp
// 비동기 함수 하나를 Observable로 - 결과가 스트림으로 흘러 연산자를 이어 붙일 수 있다
Observable.FromAsync(async ct => await LoadPlayerDataAsync(ct))
    .Subscribe(data => Apply(data))
    .AddTo(this);
```

### SubscribeAwait - 스트림 처리 자체가 비동기일 때

"버튼 클릭 → 비동기 로드"처럼 콜백 안에서 `await`가 필요하면 `SubscribeAwait`. **처리 중에 또 이벤트가 오면?** 을 `AwaitOperation`으로 선언함.

```csharp
m_LoadButton.OnClickAsObservable()
    .SubscribeAwait(async (_, ct) =>
    {
        await LoadAsync(ct);          // ct: 구독 해제(오브젝트 파괴) 시 자동 취소
    }, AwaitOperation.Drop)           // 처리 중 들어온 클릭은 버림 - 연타 방지
    .AddTo(this);
```

| AwaitOperation | 처리 중 새 이벤트가 오면     | 용례                          |
| -------------- | ---------------------------- | ----------------------------- |
| `Sequential`   | 줄 세워 순서대로 처리(기본)  | 순서 보장이 필요한 큐         |
| `Drop`         | 버림                         | 저장/로드 버튼 연타 방지      |
| `Switch`       | 진행 중인 걸 취소하고 갈아탐 | 최신 검색어만 유효한 자동완성 |
| `Parallel`     | 동시에 처리                  | 독립적인 다발 작업            |

### 취소 - CancellationToken은 공짜로 따라온다

`SubscribeAwait`/`FromAsync`가 넘겨주는 `ct`는 **구독이 해제되면 자동으로 취소**됨. 구독을 `AddTo(this)`로 묶어 뒀다면 "오브젝트 파괴 → 구독 해제 → 진행 중이던 비동기 작업 취소"가 연쇄로 일어남. 유니티의 `destroyCancellationToken`을 직접 넘기는 자리(`FirstAsync` 등)에도 같은 원리가 적용됨.

시간 축에서 보면 차이가 분명함 - 클릭이 처리 속도보다 빠를 때 `Sequential`은 밀려든 클릭을 **큐에 줄 세워 전부** 통지하고(순서 보장, 대신 지연 누적), `Drop`은 처리 중에 들어온 클릭을 **버림**(연타 무시).

```
클릭:        ──c1─c2─c3────────c4──▶
Sequential:  ──▶c1──▶c2──▶c3────▶c4    (하나도 안 버리고 순서대로, 뒤로 밀림)
Drop:        ──▶c1──╳──╳───────▶c4     (처리 중 c2·c3 버림)
```

## 6. 안티패턴 - 언제 리액티브를 _안_ 쓰나

반드시 챙길 경고.

- **pull을 억지로 push로 만들지 말 것** - _Introduction to Rx_ 의 경고: "본래 pull(끌어오기)에 자연스러운 것을 억지로 push로 만드는 것은 실수다." 한 번 읽으면 끝인 값, 단순 1:1 호출까지 Observable로 감싸지 말 것. **모든 C# `event`를 Rx로 대체하려는 시도 자체가 안티패턴.**
- **과도한 연산자 체인** - 한 파이프라인에 연산자가 7~8개 이어지면 의도가 아니라 퍼즐이 됨. 중간 스트림에 **이름을 붙여**(지역 변수) 쪼갤 것.
- **핫/콜드 혼동** - 같은 콜드 스트림을 여러 곳에서 구독해 사이드 이펙트가 두 번 돌게 하는 실수. 한 스트림을 두 곳 이상에서 쓰면 `Share()`.
- **`AddTo` 누락** - 구독 누수. 전역 스트림은 반드시 묶을 것.
- **디버깅·학습 비용** - 체인은 스택트레이스가 길고, 팀 전체가 연산자 어휘를 공유해야 함. 도입은 "구독 수명 관리 + UI 바인딩 + 시간/입력 합성"처럼 **이득이 확실한 곳부터 부분적으로**.

## 7. 정리 & 체크리스트

- R3.Unity = 유니티와의 다리 - `EveryUpdate`, 타임/프레임 프로바이더, `~Frame` 연산자, uGUI 확장, `AddTo`, Tracker.
- UI 바인딩 3패턴 - **값→표시 / Select+DistinctUntilChanged(파생 상태) / CombineLatest(조건→활성화)**.
- ReactiveProperty는 "바인딩처럼 동작하는 Observable" - 진짜 내장 바인딩이 아님.
- 에러는 기본 `OnErrorResume`(구독 유지). 필요시 `Catch`/`OnErrorResumeAsFailure`. **R3에 `Retry`는 없음.**
- 비동기 통합 - `FirstAsync`(await 한 번), `FromAsync`(Task→스트림), `SubscribeAwait`+`AwaitOperation`(Sequential/Drop/Switch/Parallel), 취소는 구독 해제에 연동.
- 안티패턴 5종 - pull→push 강제 · 과도한 체인 · 핫/콜드 혼동 · AddTo 누락 · 무분별한 전면 도입.

### 체크리스트

- [ ] `Observable.EveryUpdate()`와 `Update()` 함수의 관계, 그리고 왜 `AddTo`가 필수인지 설명할 수 있음
- [ ] UI 바인딩 3패턴을 코드로 쓸 수 있음
- [ ] `OnErrorResume`이 전통 `OnError`와 어떻게 다른지(구독 유지) 설명할 수 있음
- [ ] R3에 `Retry`가 없다는 것과 재시도의 대안(FromAsync 루프, Catch)을 말할 수 있음
- [ ] `AwaitOperation`의 Sequential/Drop/Switch 차이를 용례와 함께 말할 수 있음
- [ ] "모든 event를 Rx로 바꾸지 말라"는 경고의 근거를 설명할 수 있음

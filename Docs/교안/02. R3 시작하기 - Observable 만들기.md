# 02. R3 시작하기 - Observable 만들기

R3로 첫 코드를 작성하는 장. **구독(Subscribe)의 정확한 의미**를 잡은 뒤, Observable을 만드는 다섯 가지 방법(팩토리 · Subject · ReactiveProperty · FromEvent)을 익힘.

## 1. 설치

R3를 새 프로젝트에 설치하는 절차는 다음과 같음.

R3는 **두 부분**으로 나뉘어 설치됨.

| 구성 요소  | 내용                                     | 설치 경로                     |
| ---------- | ---------------------------------------- | ----------------------------- |
| `R3` 본체  | 범용 .NET 라이브러리(DLL)                | **NuGet** (NuGetForUnity로)   |
| `R3.Unity` | 유니티 통합(AddTo, 프레임 프로바이더 등) | **Package Manager** (git URL) |

순서:

1. **NuGetForUnity 설치** - Package Manager > `Install package from git URL...`에
   `https://github.com/GlitchEnzo/NuGetForUnity.git?path=/src/NuGetForUnity`
2. **R3 본체 설치** - 메뉴 `NuGet > Manage NuGet Packages`에서 `R3` 검색 → Install. (`Microsoft.Bcl.TimeProvider` 등 의존 패키지가 자동으로 함께 설치됨)
3. **R3.Unity 설치** - Package Manager > git URL에
   `https://github.com/Cysharp/R3.git?path=src/R3.Unity/Assets/R3.Unity#<버전>`

> ⚠️ 왜 둘로 나뉘나 - R3 본체는 유니티 전용이 아니라 **범용 .NET 라이브러리**라서 NuGet으로 배포됨. 유니티에서 NuGet을 쓰는 표준 통로가 NuGetForUnity임. `#<버전>`처럼 태그를 붙이면 그 버전으로 고정됨(**태그 = 버전 고정, 코어 NuGet 버전과 일치시킬 것**). 최신 버전은 [R3 릴리스 페이지](https://github.com/Cysharp/R3/releases)에서 확인.

설치 확인: 아무 스크립트에서 `using R3;`가 컴파일되면 됨.

## 2. 첫 구독 - Subscribe와 Observer 계약

가장 작은 R3 코드:

```csharp
using R3;
using UnityEngine;

public class HelloR3 : MonoBehaviour
{
    private void Start()
    {
        Observable
            .Range(1, 3)                            // 1, 2, 3을 내보내는 Observable
            .Subscribe(x => Debug.Log($"받음: {x}")); // 구독 - 값이 올 때 할 일을 등록
    }
}
```

짚을 것 두 가지.

- **`Subscribe`는 "등록"이지 "실행"이 아님.** `+=`로 명단에 이름을 올리는 것과 같음. 실제 콜백은 스트림에 값이 흐를 때 돎. (단, `Range`처럼 구독 즉시 값을 내보내는 Observable도 있음 - 콜드/핫은 뒤에서 다룸.)
- **`Subscribe`의 반환값은 `IDisposable`** - 구독 그 자체임. 지금은 흘려보냈지만, 구독 수명 관리의 주인공이 됨.

### 값 · 에러 · 완료 - 세 채널

Observer는 값만 받는 게 아님. `Subscribe`의 풀 버전:

```csharp
m_SomeObservable.Subscribe(
    x      => Debug.Log($"OnNext: {x}"),            // 값
    ex     => Debug.LogError($"OnErrorResume: {ex}"), // 에러 - 구독은 끊기지 않음(R3 특징)
    result => Debug.Log($"OnCompleted: {result}")    // 완료 - Result는 Success 또는 Failure
);
```

| 채널            | 의미                   | 마블 기호     |
| --------------- | ---------------------- | ------------- |
| `OnNext`        | 값이 하나 도착         | o             |
| `OnErrorResume` | 에러 발생(스트림 유지) | (R3 고유)     |
| `OnCompleted`   | 스트림 종료(성공/실패) | `\|` 또는 `X` |

> ℹ️ 전통 Rx의 `OnError`는 에러가 나면 **구독을 끊었음**. R3는 이를 `OnErrorResume`으로 바꿔 **에러를 흘려보내되 구독은 유지**하는 것을 기본으로 함(게임에서 입력 스트림이 예외 한 번에 죽으면 곤란하므로).

## 3. 팩토리 - 시간·횟수에서 스트림 만들기

`Observable`의 정적 메서드로 기성 스트림을 만들 수 있음. 시간·횟수가 값을 만드는 대표 팩토리들임.

```csharp
// 1) Range - 1부터 4개(1,2,3,4)를 즉시 내보내고 완료. --1-2-3-4-|
Observable.Range(1, 4)
    .Subscribe(x => Debug.Log($"Range: {x}"), result => Debug.Log($"완료: {result}"));

// 2) Timer - 2초 뒤 한 번 내보내고 완료. ------o| (일회성 지연 실행)
Observable.Timer(TimeSpan.FromSeconds(2))
    .Subscribe(_ => Debug.Log("2초 경과!"));

// 3) Interval - 1초마다 무한히. --o--o--o--> (멈추려면 Dispose)
IDisposable subscription = Observable.Interval(TimeSpan.FromSeconds(1))
    .Subscribe(_ => Debug.Log("tick"));
// ...
subscription.Dispose();   // Dispose가 곧 구독 해제 - Interval이 멈춘다

// 4) Return - 값 하나 내보내고 완료. o|
Observable.Return(42).Subscribe(x => Debug.Log(x));
```

`Timer`/`Interval`은 코루틴 `WaitForSeconds`·`InvokeRepeating`의 리액티브 대응물임. 차이는 **반환값이 스트림**이라서 뒤에 연산자를 이어 붙일 수 있다는 것.

> 💡 유니티의 "매 프레임"도 팩토리로 스트림이 됨 - `Observable.EveryUpdate()`. 전역 무한 스트림이라 수명 관리가 특히 중요함.

## 4. Subject - 손으로 미는 발행기

팩토리는 시간·횟수가 값을 만들었음. **내 코드의 사건**을 스트림으로 내보내려면 `Subject<T>`를 씀.

```csharp
using R3;
using UnityEngine;

public class PlayerStat : MonoBehaviour
{
    private readonly Subject<int> m_ScoreSubject = new();

    // 외부에는 Observable로만 노출 - 바깥에서 OnNext를 못 부르게 캡슐화
    public Observable<int> OnScoreChanged => m_ScoreSubject;

    private int m_Score;

    public void AddScore(int amount)
    {
        m_Score += amount;
        m_ScoreSubject.OnNext(m_Score);   // 알림 발사 (event의 Invoke에 해당)
    }

    private void OnDestroy() => m_ScoreSubject.Dispose();   // 발행기도 IDisposable
}
```

- `Subject<T>`는 **Observer이자 Observable인 "양역"** - `OnNext`로 받기도 하고(밀어 넣기), `Subscribe`로 내주기도 함.
- 옵저버 패턴의 주체를 그대로 옮긴 것: `event Action<int>` 선언 ≈ `Subject<int>` 필드, `Invoke` ≈ `OnNext`.
- **외부에는 `Observable<T>`로 노출**하는 게 관례. C#에서 `event`가 클래스 밖에서 `Invoke`되지 않는 것과 같은 캡슐화임.

### 콜드 vs 핫 - 미리 한 번만

- **콜드(cold)** - 구독할 때마다 처음부터 자기 전용 스트림이 흐름. `Range`, `Timer` 같은 팩토리가 대표.
- **핫(hot)** - 스트림이 하나만 흐르고, 구독자들은 **흐르는 중간에 합류**함. `Subject`, 클릭 스트림이 대표. 구독 전에 지나간 값은 못 받음.

지금은 "팩토리는 콜드, Subject는 핫" 정도만. 이 구분이 실수로 이어지는 지점(같은 콜드 스트림을 두 번 구독)은 `Share()`로 풀 수 있음.

## 5. ReactiveProperty - 값을 보유하는 스트림

게임 상태(체력·골드·레벨)는 "사건"이 아니라 **"현재값이 있는데, 가끔 바뀌는 것"**임. 이를 위한 전용 도구가 `ReactiveProperty<T>`.

```csharp
using R3;
using UnityEngine;

public class PlayerVitals : MonoBehaviour
{
    public ReactiveProperty<int> Hp { get; } = new(100);   // 초기값 100

    public void TakeDamage(int amount)
    {
        Hp.Value -= amount;        // .Value에 대입하면 구독자에게 자동 통지 (같은 값이면 통지 없음)
    }

    private void OnDestroy() => Hp.Dispose();
}

// 구독하는 쪽
vitals.Hp.Subscribe(hp => Debug.Log($"HP: {hp}"));
// 구독하는 "즉시" 현재값(100)이 한 번 흐르고, 이후 바뀔 때마다 흐른다
```

`Subject`와의 차이 세 가지:

1. **현재값을 보유** - `Hp.Value`로 언제든 읽을 수 있음.
2. **구독 즉시 현재값이 한 번 흐름** - 늦게 합류한 UI도 첫 프레임부터 올바른 값을 표시.
3. **같은 값이면 통지하지 않음** - 기본으로 중복 억제.

> ℹ️ 인스펙터에서 초기값을 편집하고 싶으면 `ReactiveProperty<T>` 대신 **`SerializableReactiveProperty<T>`** 를 씀(일반 버전은 `[SerializeField]` 직렬화 불가).

UI 바인딩(값 변화 → 텍스트 자동 갱신)은 별도 주제. 여기서는 "값을 보유하는 스트림"이라는 정체성만 잡음.

## 6. FromEvent - 기존 이벤트를 스트림으로

이미 있는 C# `event`나 서드파티 콜백을 스트림으로 감싸는 어댑터.

```csharp
public event Action<int> LegacyScoreEvent;   // 기존 코드의 event

Observable
    .FromEvent<int>(
        handler => LegacyScoreEvent += handler,   // 구독 시작 시 +=
        handler => LegacyScoreEvent -= handler    // 구독 해제 시 -= (자동!)
    )
    .Subscribe(score => Debug.Log($"점수: {score}"));
```

`FromEvent`가 좋은 이유: 구독을 `Dispose`하면 **`-=`가 자동으로 호출됨**. 기존 이벤트 기반 코드를 점진적으로 리액티브에 합류시키는 다리 역할임.

## 7. 생성법별로 값이 흐르는 순서

다섯 가지 생성법은 "**언제·어떻게** 값을 흘리는가"가 각각 다름. 구독(`Subscribe`) 시점을 기준으로 모양을 정리하면:

| 생성법           | 구독 후 값이 흐르는 방식                                                  |
| ---------------- | ------------------------------------------------------------------------- |
| `Range(1, 4)`    | 구독 즉시 1~4가 연달아 흐르고 `OnCompleted` - **완료가 있는** 유한 스트림 |
| `Timer(2초)`     | 2초 뒤 한 번 발행하고 완료 - 일회성                                       |
| `Interval(1초)`  | 1초마다 무한 발행. 스스로 끝나지 않으므로 멈춤은 곧 **`Dispose()`**       |
| `Subject.OnNext` | 내가 `OnNext`를 부르는 시점마다 발행 - 손으로 미는 발행기                 |
| `FromEvent`      | 감싼 `event`가 발생할 때 수신. 구독을 `Dispose`하면 `-=` 자동             |

같은 "구독"이라도 마블로 보면 흐름의 모양이 다름:

```
Range(1,4):   -1-2-3-4-|          (구독 즉시 4개 + 완료)
Timer(2s):    ------o-|           (2초 뒤 1개 + 완료)
Interval(1s): --o--o--o--o-->     (무한 - Dispose로만 멈춤)
Subject:      --o----o-o---->     (OnNext 호출 시점마다)
```

읽어 둘 핵심 둘 - ① **완료(`|`)가 있는 스트림과 없는 스트림의 구분**, ② 무한 스트림(`Interval`)은 `Dispose()`로만 멈춘다는 것(그래서 수명 관리가 중요함).

## 8. 정리 & 체크리스트

- 설치는 2단계: NuGet으로 `R3` 본체 + git URL로 `R3.Unity`.
- `Subscribe`는 **등록**. 반환값 `IDisposable`이 구독 그 자체고, `Dispose()`가 해제.
- Observer는 3채널 - `OnNext`(값) · `OnErrorResume`(에러, 구독 유지) · `OnCompleted(Result)`(완료).
- 만들기 5종 - 팩토리(`Range`/`Timer`/`Interval`...), `Subject`(손으로 발행), `ReactiveProperty`(값 보유 + 변경 통지), `FromEvent`(기존 이벤트 감싸기), 그리고 유니티 전용(`EveryUpdate` 등).
- 콜드(구독마다 새로)와 핫(흐르는 데 합류)의 감만 잡으면 됨.

### 체크리스트

- [ ] R3 설치가 왜 NuGet과 git URL 두 단계인지 설명할 수 있음
- [ ] "Subscribe는 등록"의 의미를 `+=` 비유로 설명할 수 있음
- [ ] `Interval`을 멈추는 방법이 `Dispose()`임을 코드로 쓸 수 있음
- [ ] `Subject`와 `ReactiveProperty`의 차이 3가지(현재값 보유·구독 즉시 통지·중복 억제)를 말할 수 있음
- [ ] `FromEvent`가 `-=`를 자동화한다는 것을 설명할 수 있음
- [ ] 외부에 `Subject`를 그대로 노출하지 않고 `Observable<T>`로 노출하는 이유를 말할 수 있음

# 01. 리액티브 프로그래밍이란

1부 첫 차시. 코드보다 **개념**이 주인공임. 리액티브 프로그래밍이 무엇이고(옵저버 패턴의 일반화/확장), 이벤트를 **시간 위를 흐르는 데이터 스트림**으로 본다는 게 무슨 뜻인지, 마블 다이어그램 읽는 법까지를 잡고 감. R3 코드는 다음 차시부터.

## 1. 게임 코드는 "반응"으로 가득하다

게임 로직의 대부분은 "무언가 일어나면 → 무언가 한다"임.

- 체력이 바뀌면 → 체력 바를 갱신한다
- 버튼이 눌리면 → 스킬을 쓴다
- 적이 죽으면 → 점수를 올리고, 사운드를 재생하고, 퀘스트를 갱신한다

이걸 코드로 만드는 고전적인 방법이 둘 있음.

**방법 1 — 폴링(polling, 끌어오기).** 매 프레임 값을 검사함.

```csharp
private void Update()
{
    // 매 프레임 "바뀌었나?"를 묻는다 — 바뀌지 않은 99%의 프레임에도
    if (m_Hp != m_LastHp)
    {
        m_HpText.text = $"HP : {m_Hp}";
        m_LastHp = m_Hp;
    }
}
```

**방법 2 — 이벤트(event, 밀어내기).** 바뀐 쪽이 알려 줌.

```csharp
public event Action<int> HpChanged;          // 발행자
m_Player.HpChanged += hp => m_HpText.text = $"HP : {hp}";   // 구독자
```

이벤트 방식이 깔끔하지만, 규모가 커지면 두 가지가 계속 걸림.

- **구독 해제 누락** — `+=`로 등록한 핸들러는 `-=`로 손수 해제해야 함. `OnDestroy`에서 `-=`를 빠뜨리면 파괴된 객체를 이벤트가 강하게 붙들어 **메모리 누수**, 파괴된 핸들러가 호출돼 `MissingReferenceException`이 남. "잊으면 안 되는데 잊기 쉬운" 수동 작업임.
- **이벤트 하나로 표현 안 되는 것** — "두 번 빠르게(더블클릭)", "0.5초 안에 세 번(연타)", "타이핑이 멈춘 뒤에만(검색)"은 `event` 하나로 안 됨. 결국 `Update`·코루틴에 상태 플래그와 타이머를 직접 쌓게 됨.

더블클릭 하나를 명령형으로 짜 보면 부담이 드러남.

```csharp
using System.Collections;
using UnityEngine;
using UnityEngine.InputSystem;

// 명령형: 더블클릭 하나에 코루틴 + 상태 플래그 + 타이머 + 조건문이 필요하다
public class DoubleClickImperative : MonoBehaviour
{
    private bool m_Running;

    private void Update()
    {
        if (Mouse.current.leftButton.wasPressedThisFrame && !m_Running)
            StartCoroutine(DetectDoubleClick());
    }

    private IEnumerator DetectDoubleClick()
    {
        m_Running = true;
        int clickCount = 1;
        float timeLeft = 0.3f;

        while (timeLeft > 0f && clickCount < 2)
        {
            yield return null;
            if (Mouse.current.leftButton.wasPressedThisFrame) clickCount++;
            timeLeft -= Time.deltaTime;
        }

        if (clickCount == 2) Debug.Log("더블 클릭");
        m_Running = false;
    }
}
```

핵심 질문 — **이벤트를 "시간 위를 흐르는 데이터 스트림"으로 보고, 구독 수명을 표준으로 관리하며, 연산자로 합성할 수 없을까?** 그 답이 리액티브 프로그래밍임.

## 2. 토대: 옵저버 패턴

리액티브를 정의하려면 먼저 그 토대인 **옵저버 패턴**을 알아야 함.

- **주체(Subject)** — 사건을 일으키는 쪽. 관찰자 명단을 들고 있다가 사건이 나면 명단의 전원에게 알림.
- **관찰자(Observer)** — 알림을 받아 반응하는 쪽. 주체에 자신을 **등록(구독)** 해 둠.

C#에서는 언어가 이 패턴을 `event`로 내장함.

```csharp
// 주체: 점수가 바뀌면 알린다
public class ScoreBoard
{
    public event Action<int> ScoreChanged;            // 관찰자 명단
    private int m_Score;

    public void AddScore(int amount)
    {
        m_Score += amount;
        ScoreChanged?.Invoke(m_Score);                // 명단 전원에게 통지
    }
}

// 관찰자: 등록하고, 다 쓰면 반드시 해제한다
scoreBoard.ScoreChanged += OnScoreChanged;            // 구독
scoreBoard.ScoreChanged -= OnScoreChanged;            // 해제 (잊기 쉬움!)
```

옵저버 패턴이 주는 것: **일대다 알림**과 **느슨한 결합**(주체는 관찰자가 누군지 모름). 못 주는 것: **해제의 안전망**(수동 `-=`)과 **사건의 합성**(여러 사건·시간 조건을 조합하는 표준 수단).

## 3. 리액티브 프로그래밍 = 옵저버 패턴의 확장

> ℹ️ **공식 정의 (ReactiveX)** — "Rx는 옵저버 패턴을 **데이터/이벤트의 시퀀스**를 지원하도록 **확장**하고, 그 시퀀스를 **선언적으로 합성하는 연산자**를 추가한 것." 옵저버의 대체가 아니라 **일반화/확장**임.

리액티브 프로그래밍(Reactive Programming)은 **데이터의 흐름과 변화의 전파**를 중심으로 프로그램을 짜는 패러다임. 그 대표 구현이 ReactiveX(Rx) 계열 라이브러리이고, 유니티에서 쓰는 최신 구현이 **R3**임.

한 줄로 외울 프레임:

```
Observable = 옵저버 패턴(주체-관찰자) + Operator(연산자 합성) + IDisposable(구독 수명)
```

### 3대 등장인물

| 이름             | 역할                                                       | 옵저버 패턴 대응     |
| ---------------- | ---------------------------------------------------------- | -------------------- |
| **Observable**   | 값/사건의 시퀀스를 발행하는 쪽. "관찰당할 수 있는 것"      | 주체(Subject)        |
| **Observer**     | Observable을 **구독(Subscribe)** 해 값이 올 때 반응하는 쪽 | 관찰자(Observer)     |
| **Subscription** | 구독 그 자체. 손에 쥐고 있다가 **Dispose하면 해제**        | `-=`에 해당 (표준화) |

옵저버 패턴과의 대응: 주체→Observable, 관찰자→Observer, `+=`→`Subscribe`, `-=`→`Dispose`. **이미 아는 옵저버 패턴에 이름표를 바꿔 단 것**에서 출발함.

### 옵저버 패턴 위에 더해지는 3가지

1. **연산자 합성("LINQ to events")** — `IEnumerable`에 LINQ를 쓰듯, 이벤트 스트림에 `Where`/`Select`/`Merge`를 직접 적용하고 조합함. **이게 단순 옵저버 대비 진짜 가치.**
2. **에러·완료 채널** — 값(`OnNext`)뿐 아니라 **에러**와 **완료**라는 두 시맨틱이 추가됨. "이 스트림은 여기서 끝났다/실패했다"를 표준으로 표현.
3. **표준화된 해제(`IDisposable`)** — 구독이 곧 `IDisposable` 객체라서 `Dispose()`가 곧 구독 해제. `-=` 누락 문제를 구조적으로 다룰 수 있게 됨.

## 4. 스트림으로 생각하기 — 마블 다이어그램

리액티브의 핵심 발상 전환: **이벤트는 한 번의 사건이 아니라, 시간 축 위에 늘어선 데이터의 흐름(스트림)이다.**

영상 편집 툴이나 유니티 **타임라인**을 떠올리면 됨. 클릭·키 입력·체력 변화가 전부 "시간 위의 점들"이고, 그 점들의 나열이 스트림임.

이걸 그림으로 그린 것이 **마블 다이어그램(marble diagram)**. 가로선이 시간, 구슬(●)이 발행된 값임.

```
시간 →
클릭 스트림:   ──●────●─●──────●──▶
                 클릭  클릭 클릭   클릭
```

연산자는 "스트림이 들어와서 다른 스트림이 나가는" 상자로 그림. 예를 들어 `Where`(조건 필터):

```
입력:   ──1──2──3──4──5──▶
        ┌─────────────────┐
        │ Where(x => x%2==0) │
        └─────────────────┘
출력:   ─────2─────4─────▶
```

`Select`(변환):

```
입력:   ──1────2────3──▶
        ┌──────────────┐
        │ Select(x => x*10) │
        └──────────────┘
출력:   ──10───20───30─▶
```

스트림의 끝은 두 가지 기호로 표현함 — 완료(`|`)와 에러(`✕`).

```
완료되는 스트림:  ──●──●──●──|     (예: Range(1,3) — 3개 내보내고 완료)
에러난 스트림:    ──●──●──✕        (예: 네트워크 요청 실패)
무한 스트림:      ──●──●──●──●──▶  (예: 매 프레임, 클릭 — 끝이 없음)
```

> 💡 마블 다이어그램은 Rx 세계의 공용어임. 연산자 문서가 전부 이 그림으로 설명되므로, **읽는 법**만 익혀 두면 처음 보는 연산자도 그림으로 이해할 수 있음.

## 5. push vs pull — 무엇이 뒤집혔나

`IEnumerable`과 `Observable`은 정확히 **쌍대(dual)** 관계임.

| 구분        | `IEnumerable<T>` (pull)               | `Observable<T>` (push)                      |
| ----------- | ------------------------------------- | ------------------------------------------- |
| 데이터 흐름 | 소비자가 **끌어옴** (`MoveNext` 호출) | 생산자가 **밀어 넣음** (`OnNext` 호출)      |
| 주도권      | 소비자 ("다음 값 줘")                 | 생산자 ("값 나왔다, 받아라")                |
| 시간        | 무관 — 원할 때 꺼냄                   | 본질 — 값이 **언제** 오는지가 정보의 일부   |
| 대표 도구   | LINQ (`Where`, `Select`...)           | Rx 연산자 (`Where`, `Select`... 같은 이름!) |

LINQ를 안다면 리액티브의 절반은 이미 아는 것임. 같은 연산자 어휘가 "이미 메모리에 있는 컬렉션"(pull) 대신 "시간을 두고 도착하는 이벤트"(push)에 적용될 뿐. 그래서 Rx를 **"LINQ to events"** 라고 부름.

게임에서 push가 자연스러운 것들: 입력, UI 이벤트, 체력/점수 변화, 충돌, 네트워크 메시지. 전부 "언제 올지 모르는" 데이터고, 폴링 대신 밀어 받는 쪽이 맞음.

## 6. 선언적 합성의 가치 — 미리 보기

1장의 명령형 더블클릭이 R3에서는 이렇게 됨(자세한 문법은 3차시에서).

```csharp
// 선언형: "클릭이 잠잠해질 때까지 모아서, 2개 이상이면 더블클릭"
clickStream
    .Chunk(clickStream.Debounce(TimeSpan.FromMilliseconds(250)))
    .Where(clicks => clicks.Length >= 2)
    .Subscribe(_ => Debug.Log("더블 클릭"));
```

비교 포인트는 줄 수가 아님. **명령형은 "어떻게(how)"** — 플래그를 세우고, 타이머를 깎고, 분기함. **선언형은 "무엇을(what)"** — "모아서, 2개 이상이면"이라는 의도가 그대로 코드가 됨. 상태 플래그·타이머·`while`이 사라졌으므로 **상태 관리 실수의 여지**도 함께 사라짐.

> ⚠️ 공짜는 아님 — 연산자 어휘를 익혀야 하고, 체인 중간을 디버깅하기는 콜백 한 줄보다 어려움. 언제 쓰고 언제 안 쓰는지는 이 강의 내내 짚음(특히 5차시).

## 7. Rx 생태계와 R3

리액티브를 검색하면 여러 이름이 나옴. 족보만 정리해 두면 혼동이 없음.

- **ReactiveX(Rx)** — 패러다임이자 API 표준. .NET에서 출발해 RxJava, RxJS 등 수십 언어로 퍼짐.
- **Rx.NET(1세대)** — .NET 공식 구현. 유니티에서는 동작하지 않는 부분과 iOS IL2CPP(AOT) 호환성 문제가 있었음.
- **UniRx(2세대)** — 그 문제를 풀려고 유니티용으로 재구현한 라이브러리. 오랫동안 사실상 표준이었으나 현재 **아카이브(읽기 전용)** 상태이고, README가 직접 "R3를 대신 쓰라"고 안내함.
- **R3(3세대)** — 같은 저자 계열(neuecc, Cysharp)의 정식 후속작. 유니티 전용이 아닌 범용 .NET 라이브러리이며(Godot·Avalonia·WPF 등도 지원), 유니티 통합은 `R3.Unity` 패키지가 담당. **Unity 2021.3 이상** 지원.

이 강의는 **R3 1.3.1** 기준임. 옛 UniRx 튜토리얼을 읽을 때 필요한 이름 매핑은 3차시 부록에 정리함.

## 8. 정리 & 체크리스트

- 게임 코드는 "반응"으로 가득하고, 폴링과 수동 `event` 관리에는 한계가 있음(해제 누락·합성 불가).
- 리액티브 프로그래밍 = **옵저버 패턴의 일반화/확장** (ReactiveX 공식 정의). `Observable = 옵저버 패턴 + Operator + IDisposable`.
- 이벤트를 **시간 위의 데이터 스트림**으로 봄. 마블 다이어그램(시간 축 + 구슬 + `|`/`✕`)으로 읽음.
- `IEnumerable`(pull)과 `Observable`(push)은 쌍대 — Rx는 **"LINQ to events"**.
- 선언형 합성의 가치는 "어떻게"가 아니라 **"무엇을"을 적는 것** — 상태 플래그·타이머가 사라짐.
- 계보: Rx.NET(1세대) → UniRx(2세대, 아카이브) → **R3(3세대, 이 강의 기준)**.

### 체크리스트

- [ ] 폴링과 이벤트(푸시)의 차이를 설명할 수 있음
- [ ] 옵저버 패턴의 주체/관찰자와 C# `event`의 `+=`/`-=` 대응을 앎
- [ ] "리액티브는 옵저버 패턴의 확장"을 `Observable = 옵저버 + Operator + IDisposable` 프레임으로 설명할 수 있음
- [ ] 마블 다이어그램에서 시간 축·값·완료(`|`)·에러(`✕`)를 읽을 수 있음
- [ ] push와 pull의 차이, "LINQ to events"의 뜻을 말할 수 있음
- [ ] Rx.NET → UniRx → R3 계보와, 새 프로젝트에서 R3를 쓰는 이유를 말할 수 있음

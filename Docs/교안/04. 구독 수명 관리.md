# 04. 구독 수명 관리

구독 수명을 관리하는 장. 구독은 곧 `IDisposable`이고, 해제하지 않은 구독은 누수가 됨. `AddTo`로 GameObject 수명에 묶고, `CompositeDisposable`로 묶음 해제하고, `ObservableTracker`로 누수를 추적하는 법을 익힘.

## 1. 왜 수명 관리가 중요한가

C# `event`의 흔한 문제 - `OnDestroy`에서 `-=`를 빠뜨리면 누수와 `MissingReferenceException`. 리액티브도 구독을 해제하지 않으면 똑같이 샘. 오히려 스트림 만들기가 쉬워진 만큼 구독도 많아져서, 수명 관리를 표준화하지 않으면 더 위험함.

전형적인 누수 코드:

```csharp
public class EnemyHud : MonoBehaviour
{
    private void Start()
    {
        // 전역 무한 스트림을 구독했는데 아무 데도 묶지 않았다
        Observable.EveryUpdate()
            .Subscribe(_ => m_DistanceText.text = GetDistance());   // ← 이 오브젝트가 파괴된 뒤에도 영원히 돈다
    }
}
```

이 `EnemyHud`가 파괴되면? 구독은 살아 있음. `Observable.EveryUpdate()`는 씬과 무관한 전역 스트림이라, 콜백이 파괴된 `m_DistanceText`를 만지는 순간 `MissingReferenceException`. 운 좋게 예외가 안 나도 콜백과 캡처된 참조가 영원히 메모리에 남음.

> 옵저버 패턴의 "`-=` 잊기"가 리액티브에서 사라지는 게 아니라 "Dispose 잊기"로 바뀜. 리액티브는 이를 잊기 어렵게 만드는 표준 도구를 제공함.

## 2. 구독 = IDisposable

`Subscribe`의 반환값이 구독 그 자체임.

```csharp
IDisposable subscription = Observable.Interval(TimeSpan.FromSeconds(1))
    .Subscribe(_ => Debug.Log("tick"));

subscription.Dispose();   // 구독 해제 - tick이 멈춘다. 이것이 -=의 표준화
```

수명 관리의 모든 도구는 이 한 줄("Dispose가 해제다") 위에 세워짐. 수동 `Dispose`는 "토글로 켜고 끄는 기능"(예: Interval 토글 버튼)처럼 해제 시점을 코드가 직접 정하는 경우에 씀.

## 3. AddTo(this) - GameObject 수명에 묶기

대부분의 구독은 "이 컴포넌트가 살아 있는 동안"이 곧 수명임. 그걸 한 줄로 선언하는 게 `AddTo`.

```csharp
using R3;
using UnityEngine;

public class ScoreViewer : MonoBehaviour
{
    [SerializeField] private PlayerStat m_Stat;

    private void Start()
    {
        m_Stat.OnScoreChanged
            .Where(x => x % 10 == 0)
            .Subscribe(x => Debug.Log($"점수 : {x}"))
            .AddTo(this);            // 이 컴포넌트가 파괴되면 자동 Dispose
    }
    // OnDestroy에 해제 코드가 한 줄도 없다!
}
```

`AddTo(this)`는 구독의 `IDisposable`을 이 GameObject/Component의 파괴 시점에 자동 호출되도록 등록함. 앞에서 본 누수 코드도 `.AddTo(this)` 한 줄이면 안전해짐.

> ⚠️ `AddTo`를 빠뜨리면 똑같이 누수임. "`-=` 잊기"가 "`AddTo` 잊기"로 바뀐 것뿐. 다만 `AddTo`는 체인 맨 끝의 한 줄이라 빠뜨렸는지 눈에 띔.

## 4. 여러 구독을 한꺼번에 - CompositeDisposable

구독이 여러 개면 하나하나 변수로 들지 말고 그릇에 담아 한 번에 정리함.

```csharp
using R3;
using UnityEngine;

public class BattleHud : MonoBehaviour
{
    private readonly CompositeDisposable m_Disposables = new();

    private void Start()
    {
        hp.Subscribe(UpdateHpBar).AddTo(m_Disposables);       // 그릇에 담기
        mp.Subscribe(UpdateMpBar).AddTo(m_Disposables);
        gold.Subscribe(UpdateGoldText).AddTo(m_Disposables);
    }

    private void OnDestroy()
    {
        m_Disposables.Dispose();   // 전부 한 번에 해제
    }
}
```

- `Clear()` - 담긴 구독을 전부 해제하되 그릇은 재사용(예: 라운드마다 구독을 갈아 끼울 때).
- `Dispose()` - 전부 해제하고 그릇도 폐기. 이후 `AddTo`되는 구독은 즉시 해제됨.
- MonoBehaviour가 아닌 **순수 C# 클래스**는 `AddTo(this)`를 쓸 수 없으므로 `CompositeDisposable`이 표준임.

> 💡 성능이 민감한 곳에서는 `Disposable.CreateBuilder()`로 모아 `RegisterTo(destroyCancellationToken)`에 거는 저할당 패턴도 있음(R3 README). 동작은 같으니 기본은 `AddTo`/`CompositeDisposable`로 충분.

## 5. ObservableTracker - 누수를 눈으로 잡기

구독이 새고 있지 않은지 확인하는 도구가 R3에 내장됨.

```csharp
ObservableTracker.EnableTracking = true;     // 추적 시작 (기본 false)
ObservableTracker.EnableStackTrace = true;   // 구독이 생성된 위치(스택)까지 기록

// 현재 살아 있는 구독을 전부 순회
ObservableTracker.ForEachActiveTask(state =>
{
    Debug.Log($"#{state.TrackingId} {state.FormattedType}");
});
```

- 유니티 에디터에서는 **`Window > Observable Tracker`** 윈도우로 같은 내용을 GUI로 봄 - 씬을 나갔는데 사라지지 않는 구독이 있으면 그게 누수임.
- 추적에는 오버헤드가 있으므로 개발·디버그 중에만 켤 것.

실전 누수 점검 루틴: ① Tracker를 켠다 → ② 의심 씬에 들어갔다 나온다 → ③ 활성 구독 목록에 그 씬의 구독이 남아 있는지 본다 → ④ 남았다면 StackTrace로 생성 지점을 찾아 `AddTo`를 단다.

## 6. 누수 패턴 정리

| 패턴                                                      | 증상                           | 처방                                                            |
| --------------------------------------------------------- | ------------------------------ | --------------------------------------------------------------- |
| 전역 스트림(`EveryUpdate`/`Interval`) 구독에 `AddTo` 누락 | 오브젝트 파괴 후에도 콜백이 돎 | `.AddTo(this)` - 전역 스트림은 반드시                           |
| static/싱글톤의 Observable을 씬 객체가 구독               | 씬 전환 후에도 구독 잔존       | 구독자 쪽에서 `AddTo(this)` (발행자 수명 ≠ 내 수명)             |
| 순수 C# 클래스가 구독을 보관만 하고 Dispose 안 함         | 소유자 파괴 후 잔존            | `CompositeDisposable` + 소유자의 `OnDestroy`/`Dispose`에서 정리 |
| `Subject`/`ReactiveProperty` 발행기 방치                  | 구독자 정리돼도 발행기는 잔존  | 발행기도 `IDisposable` - 소유자가 `Dispose`                     |

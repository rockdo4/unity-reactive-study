# 06. 따라하기 — 리액티브 클리커 만들기 (2부)

1부에서 배운 것**만으로** 작은 게임을 완성하는 따라하기 수업(2차시 권장). 쿠키클리커풍 미니 게임 "리액티브 클리커"를 5단계로 만들며, 각 단계가 1부의 어느 차시를 쓰는지 확인함.

> 데모: `06_ReactiveClicker` 씬(완성본·확인용) · 학생 패키지 `R3Lecture-Student-06-Clicker`(스켈레톤 — 직접 채워 만든다) · 코드 `Assets/Scripts/ClickerGame/`

**완성 스펙**

- 클릭 버튼을 누르면 골드 획득 (클릭당 골드는 업그레이드로 증가)
- 업그레이드 2종: 클릭 강화 / 도우미 고용(초당 자동 수입) — **살 수 있을 때만 버튼 활성화**
- 빠르게 연속 클릭하면 콤보 — 10콤보마다 보너스 골드
- 피버 버튼: 5초간 클릭 골드 2배, 15초 쿨다운

| 기능           | 사용하는 도구                         | 배운 차시 |
| -------------- | ------------------------------------- | --------- |
| 골드/스탯      | `ReactiveProperty<long>`              | 02        |
| UI 자동 갱신   | `Subscribe` 바인딩                    | 05        |
| 구매 가능 여부 | `CombineLatest` → `interactable`      | 03·05     |
| 자동 수입      | `Observable.Interval`                 | 02        |
| 콤보           | `Merge` + `Scan` + `Debounce`         | 03        |
| 피버 쿨다운    | `ThrottleFirst` + `Timer`             | 03        |
| 수명 관리      | `AddTo(this)` + `CompositeDisposable` | 04        |

> 완성본: `Assets/Scripts/ClickerGame/` (`ClickerModel.cs`, `ClickerPresenter.cs`). 막히면 비교하되, **타이핑은 직접** 할 것.

## 0단계. 준비 — 패키지 임포트와 설계

### 무엇을 채우는 수업인가

이번 수업은 **빈 파일에서 전부 작성하는** 방식이 아니라, **스켈레톤의 비어 있는 핵심 메서드만 채우는** 방식이다. 게임의 뼈대(필드·상수·게임 규칙·UI 배선·수명 관리)는 이미 들어 있고, 학생은 **R3의 심장에 해당하는 부분**만 직접 채운다.

1. `R3Lecture-Student-06-Clicker` 패키지를 임포트하고 `06_ReactiveClicker` 씬을 연다. UI(클릭 버튼·골드 텍스트·업그레이드 버튼 등)가 이미 배치돼 있고, 그 UI가 **`MyClickerPresenter`에 모두 연결**돼 있다. 스크립트 `MyClickerModel.cs`·`MyClickerPresenter.cs`도 함께 들어 있다.
2. 두 스크립트에는 R3 핵심 메서드 — **`MyClickerModel`의 생성자 일부·`TryStartFever()`**, **`MyClickerPresenter`의 `BindInput()`·`BindView()`** — 만 비어 있다. 아래 단계대로 그 빈 메서드를 채워 게임을 완성한다.
3. 강사 프로젝트의 완성본 `06_ReactiveClicker` 씬(과 `Assets/Scripts/ClickerGame/`)은 **확인·비교용**이다(학생 배포 아님). 막히면 이걸 펴서 대조한다.

> 💡 이미 채워져 있는 것 vs. 직접 채우는 것
>
> - **이미 있음(확인만)** — `Gold`/`GoldPerClick` 등 모든 `ReactiveProperty` 필드, 비용 상수, `FeverDuration`, `m_Disposables`, 게임 규칙 메서드(`Click`/`AddBonus`/`TryBuyClickUpgrade`/`TryBuyAutoUpgrade`), `Dispose()`. Presenter 쪽은 모든 `[SerializeField]` 필드, `m_LogLines`, `m_Model`, `Start()`, `Log()`, `OnDestroy()`.
> - **직접 채움** — Model 생성자의 파생 상태(`CanBuy*`)·자동 수입·`AddTo`, `TryStartFever()`, Presenter의 `BindInput()`·`BindView()`.

### 설계 — Model과 Presenter를 나눈다

| 클래스               | 정체           | 책임                                          | 모르는 것 |
| -------------------- | -------------- | --------------------------------------------- | --------- |
| `MyClickerModel`     | 순수 C# 클래스 | 상태(`ReactiveProperty`)와 규칙(구매·피버 등) | UI        |
| `MyClickerPresenter` | MonoBehaviour  | 입력(버튼)→Model 호출, Model 상태→UI 바인딩   | 게임 규칙 |

Model이 UI를 모르므로 ①규칙을 단위 테스트할 수 있고 ②UI를 갈아치워도 Model이 안 바뀜. 5차시 끝에 예고한 **MVP 구조**를 여기서 실제로 씀. 스켈레톤도 이 두 클래스로 나뉘어 있으니, 채우면서 "어느 책임이 어디에 있는지"를 함께 확인하자.

## 1단계. 골드와 클릭 — 가장 작은 게임

먼저 게임의 가장 작은 형태부터 머릿속에 그려 보자. 클릭하면 골드가 오르고, 그 골드가 텍스트에 자동으로 비치는 흐름이다.

### Model — 상태는 ReactiveProperty로 (이미 채워져 있음, 확인만)

`MyClickerModel.cs`에는 상태가 전부 `ReactiveProperty`로 선언돼 있고, `Click()`도 들어 있다. **다시 타이핑하지 말고 읽어서 의미만 확인**하자.

```csharp
public ReactiveProperty<long> Gold { get; } = new(0);
public ReactiveProperty<long> GoldPerClick { get; } = new(1);

public long Click()
{
    long gain = GoldPerClick.Value * (IsFever.Value ? FeverMultiplier : 1);
    Gold.Value += gain;        // .Value 대입 → 구독자(UI)에게 자동 통지
    return gain;
}
```

> `Click()`은 처음부터 피버 배수(`IsFever.Value ? FeverMultiplier : 1`)까지 반영된 모습으로 제공된다. 5단계에서 `IsFever`가 켜지면 자연히 2배가 된다 — 1단계에선 `IsFever`가 `false`라 그냥 1배다.

### Presenter — 입력을 Model로, Model을 UI로 (여기서 채우기 시작)

`MyClickerPresenter.cs`의 `Start()`는 이미 `m_Model = new MyClickerModel();` 후 `BindInput()`·`BindView()`를 부른다. 우리가 채울 곳은 그 **비어 있는 `BindInput()`·`BindView()`** 다.

가장 먼저 클릭→골드, 골드→텍스트의 두 가닥을 잇는다. (콤보·피버·업그레이드는 뒤 단계에서 같은 메서드에 살을 붙인다.)

```csharp
private void BindInput()
{
    // 입력 → Model
    m_ClickButton.OnClickAsObservable()
        .Subscribe(_ => m_Model.Click())
        .AddTo(this);
}

private void BindView()
{
    // Model → UI (값이 바뀔 때만, 폴링 없음)
    m_Model.Gold
        .Subscribe(gold => m_GoldText.text = $"골드 {gold:N0}")
        .AddTo(this);
}
```

### 확인

UI는 패키지 씬에서 이미 `MyClickerPresenter`의 각 `[SerializeField]`에 연결돼 있다(`m_ClickButton`·`m_GoldText` 등). **Play → 클릭하면 골드가 오른다.**

> 💡 짚기 — `m_GoldText`를 갱신하는 코드는 클릭 핸들러에 없음. Presenter는 "골드가 바뀌면 텍스트를 갱신한다"를 **한 번 선언**했을 뿐이고, 출처(클릭이든 나중의 자동 수입이든)와 무관하게 항상 맞는 표시가 됨. 이게 바인딩의 힘.

## 2단계. 업그레이드 — CombineLatest로 "살 수 있나"

### Model — 파생 상태를 생성자에서 채운다

`ClickUpgradeCost` 필드와 `CanBuyClickUpgrade` 프로퍼티 선언, 그리고 `TryBuyClickUpgrade()`는 **이미 스켈레톤에 있다**(확인만). 비어 있는 건 **생성자에서 `CanBuyClickUpgrade`를 파생**시키는 부분이다.

```csharp
// 이미 선언돼 있음 (확인만):
public ReactiveProperty<long> ClickUpgradeCost { get; } = new(10);
public ReadOnlyReactiveProperty<bool> CanBuyClickUpgrade { get; }   // 파생 상태

// 이미 있음 (확인만): 구매 검증은 TryBuy... 패턴
public bool TryBuyClickUpgrade()
{
    long cost = ClickUpgradeCost.Value;
    if (Gold.Value < cost)
        return false;
    Gold.Value -= cost;
    GoldPerClick.Value += 1;
    ClickUpgradeCost.Value = (long)Math.Ceiling(cost * ClickUpgradeCostGrowth);   // 비용 인상
    return true;
}
```

비어 있는 생성자에 파생 한 줄을 채운다(자동 수입·`AddTo`는 3단계에서 마저 채운다):

```csharp
public MyClickerModel()
{
    CanBuyClickUpgrade = Gold
        .CombineLatest(ClickUpgradeCost, (gold, cost) => gold >= cost)
        .ToReadOnlyReactiveProperty();
    // (3단계에서 CanBuyAutoUpgrade · Interval · AddTo 를 이어서 채운다)
}
```

- `CombineLatest`(3차시)가 골드·비용의 **최신값 조합**으로 `bool` 스트림을 만들고, `ToReadOnlyReactiveProperty()`로 "현재값을 보유한 읽기 전용 프로퍼티"로 굳힘.
- 구매 검증은 `TryBuy...` 패턴 — Model이 규칙의 최종 관문이고, UI 활성화는 어디까지나 편의임.

### Presenter — BindInput / BindView에 업그레이드 가닥 추가

같은 두 메서드에 살을 붙인다. 입력은 `BindInput()`에, 표시는 `BindView()`에.

```csharp
// BindInput() 에 추가 — 입력 → Model
m_ClickUpgradeButton.OnClickAsObservable()
    .Subscribe(_ => m_Model.TryBuyClickUpgrade())
    .AddTo(this);

// BindView() 에 추가 — Model → UI: 살 수 있을 때만 버튼 활성화
m_Model.CanBuyClickUpgrade
    .Subscribe(can => m_ClickUpgradeButton.interactable = can)
    .AddTo(this);

// BindView() 에 추가 — 비용 표시
m_Model.ClickUpgradeCost
    .Subscribe(cost => m_ClickUpgradeLabel.text = $"클릭 강화\n비용 {cost:N0}")
    .AddTo(this);
```

> Model의 정리는 이미 `Dispose()`가 `m_Disposables.Dispose()` 한 줄로 처리한다. 새 `ReactiveProperty`마다 따로 `Dispose`를 부를 필요가 없다 — 다음 단계에서 그 이유(모두 `m_Disposables`에 담는다)를 직접 채운다.

**Play → 골드 10을 모으면 버튼이 켜지고, 사면 꺼졌다가 다시 모이면 켜진다.** 활성화 갱신 코드를 어디에도 "호출"하지 않았다는 점을 음미할 것.

## 3단계. 자동 수입 — Interval과 Model의 수명

도우미를 고용하면 1초마다 골드가 자동으로 들어옴. **Model 안에서 시간이 흐르기 시작**하므로, Model의 구독 수명 관리(4차시)가 등장함.

### Model — 생성자를 마저 채운다 (CompositeDisposable로 모으기)

`GoldPerSecond`·`AutoUpgradeCost`·`CanBuyAutoUpgrade` 필드 선언, `m_Disposables`, `TryBuyAutoUpgrade()`, `Dispose()`는 **이미 스켈레톤에 있다**(확인만).

```csharp
// 이미 선언돼 있음 (확인만):
public ReactiveProperty<long> GoldPerSecond { get; } = new(0);
public ReactiveProperty<long> AutoUpgradeCost { get; } = new(25);
public ReadOnlyReactiveProperty<bool> CanBuyAutoUpgrade { get; }
private readonly CompositeDisposable m_Disposables = new();

// 이미 있음 (확인만):
public bool TryBuyAutoUpgrade()
{
    long cost = AutoUpgradeCost.Value;
    if (Gold.Value < cost)
        return false;
    Gold.Value -= cost;
    GoldPerSecond.Value += 1;
    AutoUpgradeCost.Value = (long)Math.Ceiling(cost * AutoUpgradeCostGrowth);
    return true;
}

public void Dispose()
{
    m_Disposables.Dispose();   // Interval 구독까지 전부 한 번에 (이미 있음)
}
```

이제 2단계에서 시작한 생성자를 **완성형**으로 채운다 — `CanBuyAutoUpgrade` 파생, 자동 수입 `Interval`, 그리고 모든 발행기·프로퍼티를 그릇(`m_Disposables`)에 담기.

```csharp
public MyClickerModel()
{
    CanBuyClickUpgrade = Gold
        .CombineLatest(ClickUpgradeCost, (gold, cost) => gold >= cost)
        .ToReadOnlyReactiveProperty()
        .AddTo(m_Disposables);                 // 그릇에 담기 시작

    CanBuyAutoUpgrade = Gold
        .CombineLatest(AutoUpgradeCost, (gold, cost) => gold >= cost)
        .ToReadOnlyReactiveProperty()
        .AddTo(m_Disposables);

    // 자동 수입: 1초마다 GoldPerSecond만큼
    Observable.Interval(TimeSpan.FromSeconds(1))
        .Where(_ => GoldPerSecond.Value > 0)
        .Subscribe(_ => Gold.Value += GoldPerSecond.Value)
        .AddTo(m_Disposables);

    // ReactiveProperty들도 전부 그릇에 — Dispose 한 줄로 정리되게
    Gold.AddTo(m_Disposables);
    GoldPerClick.AddTo(m_Disposables);
    GoldPerSecond.AddTo(m_Disposables);
    ClickUpgradeCost.AddTo(m_Disposables);
    AutoUpgradeCost.AddTo(m_Disposables);
    IsFever.AddTo(m_Disposables);
}
```

> ❗ **여기가 4차시의 실전임.** Model은 MonoBehaviour가 아니라 `AddTo(this)`를 못 씀 → 순수 C# 클래스의 표준인 `CompositeDisposable`로 모든 구독·발행기를 담고, `Dispose()` 한 줄로 정리. Presenter의 `OnDestroy`가 `m_Model.Dispose()`를 불러 주므로(스켈레톤에 이미 있음) 사슬이 완성됨. 이걸 빠뜨리면 **씬을 나가도 Interval이 영원히 도는** 교과서적 누수가 됨 — 6단계에서 Tracker로 직접 확인함.
>
> `IsFever.AddTo(...)`까지 미리 넣어 둔 이유는 5단계에서 `IsFever`를 쓰기 때문 — 모든 `ReactiveProperty`를 한 곳에 모아 누락을 막는다.

### Presenter — BindInput / BindView에 도우미 가닥 추가

```csharp
// BindInput() 에 추가 — 입력 → Model
m_AutoUpgradeButton.OnClickAsObservable()
    .Subscribe(_ => m_Model.TryBuyAutoUpgrade())
    .AddTo(this);

// BindView() 에 추가 — Model → UI
m_Model.CanBuyAutoUpgrade.Subscribe(can => m_AutoUpgradeButton.interactable = can).AddTo(this);
m_Model.AutoUpgradeCost.Subscribe(cost => m_AutoUpgradeLabel.text = $"도우미 고용\n비용 {cost:N0}").AddTo(this);

// 두 스탯을 합쳐 한 줄로 — CombineLatest는 UI 조합에도 쓴다
m_Model.GoldPerClick
    .CombineLatest(m_Model.GoldPerSecond, (perClick, perSec) => $"클릭당 +{perClick:N0} · 초당 +{perSec:N0}")
    .Subscribe(text => m_StatsText.text = text)
    .AddTo(this);
```

**Play → 도우미를 사면 가만히 있어도 골드가 1초마다 오른다.** 골드 텍스트 갱신 코드는 1단계 그대로임 — 출처가 늘어도 바인딩은 그대로.

## 4단계. 콤보 — Merge + Scan 누적기

빠르게 연속 클릭하면 콤보가 쌓이고, 0.8초 쉬면 끊김. 10콤보마다 보너스. **순수하게 연산자 합성(3차시)만으로** 만듦. `AddBonus(amount)`는 Model에 **이미 있으니**(확인만) Presenter의 `BindInput()`만 손본다.

```csharp
// 이미 있음 (확인만): Model의 보너스 입금 창구
public void AddBonus(long amount)
{
    Gold.Value += amount;
}
```

### BindInput()의 클릭 가닥을 콤보까지 확장

1단계에서 넣은 `m_ClickButton.OnClickAsObservable().Subscribe(...).AddTo(this);` 한 줄을, 아래의 **`Share` 기반 배선으로 교체**한다. 클릭 스트림을 골드·콤보·보너스 세 곳에서 쓰기 때문이다.

```csharp
// 클릭 스트림을 세 곳(골드/콤보/보너스)에서 쓰므로 Share로 구독 공유 (3차시 7장)
var clicks = m_ClickButton.OnClickAsObservable().Share();

clicks.Subscribe(_ => m_Model.Click()).AddTo(this);

// 콤보 카운터: 클릭은 +1, 0.8초 잠잠해지면(Debounce) 0 신호 → Scan이 리셋
var combo = clicks
    .Select(_ => 1)
    .Merge(clicks.Debounce(ComboResetDelay).Select(_ => 0))
    .Scan(0, (acc, x) => x == 0 ? 0 : acc + 1)
    .Share();

combo
    .Subscribe(c => m_ComboText.text = c >= 2 ? $"{c} 콤보!" : string.Empty)
    .AddTo(this);

// 10·20·30... 콤보 도달 순간 보너스
combo
    .Where(c => c > 0 && c % ComboBonusUnit == 0)
    .Subscribe(c =>
    {
        long bonus = c * m_Model.GoldPerClick.Value;
        m_Model.AddBonus(bonus);
        Log($"{c}콤보 달성! 보너스 +{bonus:N0}");
    })
    .AddTo(this);
```

> `ComboResetDelay`(0.8초)·`ComboBonusUnit`(10)는 스켈레톤에 상수로 이미 선언돼 있다. 이벤트 표시는 `m_EventLogText.text = ...`를 직접 쓰지 말고 **스켈레톤에 제공된 `Log(...)` 헬퍼**를 쓴다 — 최근 5줄을 모아 한 번에 보여 준다.

마블 다이어그램으로 읽기:

```
clicks:        ──●─●─●────────────●──▶
Select(1):     ──1─1─1────────────1──▶
Debounce(0):   ────────────0─────────▶   (0.8초 잠잠 후)
Merge:         ──1─1─1─────0──────1──▶
Scan:          ──1─2─3─────0──────1──▶   ← 콤보 카운터 완성
```

명령형이라면 "마지막 클릭 시각 저장 + 매 프레임 경과 검사 + 카운터 리셋" 코드가 필요했을 자리임.

**Play → 연타하면 "n 콤보!"가 오르고, 10콤보에 보너스 로그가 찍힌다.**

## 5단계. 피버 — ThrottleFirst 쿨다운과 Timer

피버 버튼을 누르면 5초간 클릭 골드 2배. 쿨다운 15초는 **`ThrottleFirst`가 게이트** 역할(3차시의 "먼저 통과, 구간 무시").

### Model — TryStartFever()를 채운다

`IsFever` 필드와 `FeverDuration`은 **이미 있다**(확인만). `Click()`도 1단계에서 본 대로 이미 피버 배수를 반영하고 있다. 비어 있는 건 **`TryStartFever()` 본문** 하나다.

```csharp
// 이미 선언돼 있음 (확인만):
public ReactiveProperty<bool> IsFever { get; } = new(false);
public TimeSpan FeverDuration { get; } = TimeSpan.FromSeconds(5);

// 비어 있는 TryStartFever() 를 채운다:
public bool TryStartFever()
{
    if (IsFever.Value)
        return false;
    IsFever.Value = true;
    // 5초 뒤 자동 종료 — Timer는 일회성 스트림 (2차시)
    Observable.Timer(FeverDuration)
        .Subscribe(_ => IsFever.Value = false)
        .AddTo(m_Disposables);
    return true;
}
```

> `IsFever`는 3단계 생성자에서 이미 `m_Disposables`에 담았으니, 여기선 `Timer` 구독만 `AddTo(m_Disposables)`로 묶으면 된다.

### Presenter — BindInput / BindView에 피버 가닥 추가

```csharp
// BindInput() 에 추가 — ThrottleFirst가 15초 쿨다운 게이트: 첫 클릭만 통과
m_FeverButton.OnClickAsObservable()
    .ThrottleFirst(FeverCooldown)
    .Subscribe(_ =>
    {
        if (m_Model.TryStartFever())
            Log($"피버 시작! {m_Model.FeverDuration.TotalSeconds:0}초간 클릭 2배");
    })
    .AddTo(this);

// BindView() 에 추가 — 피버 상태 → 문구·버튼 색
m_Model.IsFever
    .Subscribe(isFever =>
    {
        m_FeverStateText.text = isFever ? "피버 중! 클릭 2배" : "피버 (15초 쿨다운)";
        if (m_ClickButtonImage != null)
            m_ClickButtonImage.color = isFever ? new Color(1f, 0.6f, 0.2f) : Color.white;
    })
    .AddTo(this);
```

> `FeverCooldown`(15초)도 스켈레톤 상수로 제공된다. 여기서도 표시는 `Log(...)`를 쓴다.

**Play → 피버를 누르면 버튼이 주황색이 되고 클릭당 골드가 2배, 5초 뒤 원상복구. 15초 안에 다시 눌러도 무시된다.**

> 💡 짚기 — 쿨다운을 위해 시각 저장 변수도, 코루틴도 안 만들었음. "첫 입력만 통과시키고 15초 무시"라는 **정책이 연산자 한 개**임.

## 6단계. 마무리 점검 — 누수 0 확인과 회고

### 막혔을 때 — 완성본과 비교

채운 메서드가 동작하지 않거나 컴파일이 안 되면, 강사 프로젝트의 **완성본 `06_ReactiveClicker` 씬**(또는 `Assets/Scripts/ClickerGame/`의 `ClickerModel.cs`·`ClickerPresenter.cs`)과 같은 메서드를 한 줄씩 대조한다. 클래스 이름만 `ClickerModel`/`ClickerPresenter`로 다를 뿐, 채워야 할 R3 코드는 동일하다.

### ObservableTracker로 검증 (4차시 실전)

1. 아무 스크립트에서 `ObservableTracker.EnableTracking = true;`를 켜거나, 1부 데모 씬을 거쳐 켜진 상태로 시작.
2. `Window > Observable Tracker`를 연다.
3. 클리커 씬에서 실컷 플레이 → 씬을 비우거나 `MyClickerPresenter` 오브젝트를 삭제.
4. Tracker 목록에서 클리커 관련 구독이 **전부 사라지면 합격.** 남아 있다면 `AddTo(this)`/`AddTo(m_Disposables)`가 빠진 곳이 있다는 뜻 — StackTrace로 추적.

일부러 망가뜨려 보기: Model 생성자의 Interval 구독에서 `.AddTo(m_Disposables)`를 지우고 같은 실험 → 씬을 나가도 Interval 구독이 남는 것을 Tracker로 목격. **이 한 번의 목격이 백 마디 경고보다 낫다.**

### 회고 — 어디에 무엇을 썼나

| 코드 위치           | 도구                                | "선언한 의도"                               |
| ------------------- | ----------------------------------- | ------------------------------------------- |
| `Gold` 등 상태      | `ReactiveProperty`                  | 값이 바뀌면 알아서 알려라                   |
| 골드/비용/스탯 표시 | `Subscribe` 바인딩                  | 이 값은 항상 이 텍스트와 같게               |
| 구매 버튼 활성화    | `CombineLatest`+`ToReadOnlyRP`      | 골드 ≥ 비용일 때만 살 수 있다               |
| 자동 수입           | `Interval`+`Where`                  | 1초마다, 도우미가 있으면 지급               |
| 콤보                | `Merge`+`Scan`+`Debounce`           | 클릭은 +1, 0.8초 쉬면 리셋                  |
| 콤보 보너스         | `Where(c % 10 == 0)`                | 10단위 도달 순간만                          |
| 피버 쿨다운/지속    | `ThrottleFirst`+`Timer`             | 첫 입력만 통과, 5초 뒤 종료                 |
| 수명                | `AddTo(this)`+`CompositeDisposable` | 화면이 죽으면 구독도, Model이 죽으면 시계도 |

`Update()` 함수 0개, 상태 플래그 0개, 코루틴 0개로 게임 하나가 완성됨. 동시에 — 이 게임의 **모든 로직이 push가 자연스러운 것들**(입력·시간·값 변화)이었다는 점도 기억할 것. 인벤토리 정렬이나 데미지 공식 계산까지 Observable로 만들 이유는 없음(5차시 안티패턴).

### 확장 아이디어

여력이 되면 같은 구조 위에 더 얹어 보자: 피버 쿨다운 남은 시간 표시(`Observable.Interval` + 남은 초 계산), 크리티컬 확률 업그레이드(클릭 시 확률로 배수), 황금 보너스가 잠깐 출현했다 사라지는 연출(`Timer` + `Where`) 등. 모두 1부에서 배운 연산자 조합만으로 충분하다.
